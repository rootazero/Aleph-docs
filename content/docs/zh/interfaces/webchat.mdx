WebChat 是 Aleph 的内置基于 Web 的聊天接口。它由 Gateway 的 HTTP 服务器直接提供，与 WebSocket 端点并存，无需外部依赖或第三方 API。WebChat 使用与所有其他 Gateway 客户端相同的 JSON-RPC 2.0 协议。

## 架构

```
┌──────────────────────────────────────────────────────────────┐
│                     浏览器 / 嵌入式                           │
│  ┌────────────────────────────────────────────────────────┐  │
│  │                   WebChat UI                            │  │
│  │  ┌──────────┐  ┌───────────┐  ┌──────────────────┐    │  │
│  │  │ 消息历史  │  │  输入区域  │  │  流式传输面板     │    │  │
│  │  └──────────┘  └───────────┘  └──────────────────┘    │  │
│  └──────────┬─────────────────────────────────────────────┘  │
│             │ WebSocket (JSON-RPC 2.0)                        │
└─────────────┼────────────────────────────────────────────────┘
              │
┌─────────────┼────────────────────────────────────────────────┐
│             ▼                                                  │
│  ┌────────────────────────────────────────────────────────┐  │
│  │              Gateway HTTP Server                        │  │
│  │                                                         │  │
│  │  /              → 静态文件 (WebChat UI)                 │  │
│  │  /ws            → WebSocket 端点                        │  │
│  │  /health        → 健康检查                              │  │
│  │  /metrics       → 指标端点                              │  │
│  └────────────────────────────────────────────────────────┘  │
│                        Aleph Server                           │
└──────────────────────────────────────────────────────────────┘
```

WebChat 作为单页应用 (SPA) 从 Gateway 的静态文件目录提供。与 Gateway 的 WebSocket 连接使用与 macOS 应用、Tauri 桌面应用和 CLI 相同的 JSON-RPC 协议提供实时双向通信。

## 面板导航

桌面面板（以及完整的 WebChat SPA）公开了**七个顶层模式**，可从分区切换器进入。它们按以下导航顺序排列：

| 模式 | 路由 | 用途 |
|------|-------|---------|
| Chat | `/chat` | 主对话界面 |
| Dashboard | `/dashboard` | 状态、活动和概览 |
| Memory | `/memory` | 记忆中心 —— 图谱视图和表格视图统一在一个宿主中 |
| Agents | `/agents` | 浏览和配置智能体 |
| Teams | `/teams` | 多智能体团队和群聊 |
| Extensions | `/extensions` | 统一的扩展商店 |
| Settings | `/settings` | 配置 |

这个拓扑带来了几处变化：

- **Extensions**（`/extensions`）是统一的**扩展商店** —— 一个按功能类别浏览技能（Skill）、插件（Plugin）和 MCP 服务器的统一场所。它**取代了旧的 ClawHub 标签页**，后者已被移除。完整概念参见[扩展商店](/docs/zh/concepts/extensions-store)。
- 旧的 **MCP / Plugins / Skills** 设置面板现在位于 **Settings > Advanced** 之下，而非作为顶层标签页。

> 切换模式只改变面板界面；底层的 Gateway 会话和 WebSocket 连接保持不变，因此下文描述的同一 JSON-RPC 协议适用于全部七个模式。

## 能力

| 功能 | 状态 |
|---------|--------|
| 文本消息 | 支持 |
| Markdown 渲染 | 支持（完整 CommonMark） |
| 代码语法高亮 | 支持 |
| Mermaid 图表 | 支持 |
| 实时流式传输 | 支持 |
| 图片显示 | 支持（内联） |
| 文件上传 | 支持 |
| Canvas 交互 | 支持 |
| 输入中指示器 | 视觉（客户端） |
| 多标签 UI | 支持 |

WebChat 在 **WebRich** 交互范式下运行，这使 AI 能够访问包括 Mermaid 图表、canvas 绘图、可折叠部分和流式输出在内的完整能力集。

WebChat 包含在基础 `gateway` 功能标志中。如果你可以启动 Aleph 服务器，WebChat 就可用。

```bash
cargo build --release --features gateway
```


## 配置

### Gateway 配置（隐式）

WebChat 使用 Gateway 的 HTTP 服务器，当 Gateway 运行时始终可用：

```toml
[gateway]
host = "127.0.0.1"
port = 18790
max_connections = 100
```

### 显式 WebChat 配置

对于高级场景，你可以单独配置 WebChat：

```toml
[channels.webchat]
# 启用 WebChat（配置中默认：false，但 HTTP 服务器始终提供静态文件）
enabled = true

# WebChat HTTP 服务器端口（如果与 Gateway 分开）
port = 18790

# 跨域嵌入的 CORS 来源
cors_origins = ["http://localhost:*", "https://your-app.example.com"]
```

### 静态文件目录

Gateway 从可配置目录提供静态文件：

```rust
HttpServerConfig {
    addr: ([127, 0, 0, 1], 18790).into(),
    static_dir: Some(PathBuf::from("/path/to/webchat/dist")),
    fallback_file: Some("index.html".to_string()),
    enable_cors: true,
}
```

`fallback_file` 启用 SPA 路由 —— 任何不匹配静态文件的路径都提供 `index.html`，让前端路由器处理 URL。

## WebSocket 协议

WebChat 使用标准 JSON-RPC 2.0 协议通过 WebSocket 与 Gateway 通信。

### 连接

```javascript
const ws = new WebSocket("ws://127.0.0.1:18790/ws");
```

如果启用了认证，第一条消息必须是 `connect` 请求：

```json
{
  "jsonrpc": "2.0",
  "id": "1",
  "method": "connect",
  "params": {
    "minProtocol": 1,
    "maxProtocol": 1,
    "client": {
      "id": "webchat",
      "version": "1.0.0",
      "platform": "web"
    },
    "role": "operator"
  }
}
```

### 发送消息

```json
{
  "jsonrpc": "2.0",
  "id": "msg-001",
  "method": "agent.run",
  "params": {
    "message": "今天天气怎么样？",
    "session_key": "agent:main:main"
  }
}
```

### 接收流式事件

订阅流式事件以实时接收响应：

```json
{
  "jsonrpc": "2.0",
  "id": "sub-001",
  "method": "events.subscribe",
  "params": { "pattern": "stream.*" }
}
```

事件作为 JSON-RPC 通知到达：

```json
{
  "jsonrpc": "2.0",
  "method": "event",
  "params": {
    "topic": "stream.chunk",
    "data": {
      "run_id": "run-123",
      "content": "根据当前条件..."
    }
  }
}
```

### 事件类型

| 主题 | 描述 |
|-------|-------------|
| `stream.chunk` | 文本内容块（部分响应） |
| `stream.tool_start` | 工具执行已开始 |
| `stream.tool_end` | 工具执行完成 |
| `agent.started` | Agent 运行已开始 |
| `agent.completed` | Agent 运行完成 |
| `agent.error` | Agent 运行遇到错误 |

## 静态文件服务

Gateway 的 HTTP 服务器基于文件扩展名和路径模式处理静态文件请求：

| 模式 | 行为 |
|---------|----------|
| `/` | 提供 `index.html` |
| `/assets/*`, `/static/*` | 提供静态资源 |
| `*.html`, `*.css`, `*.js` | 提供匹配的文件 |
| `*.png`, `*.jpg`, `*.svg` | 提供图片文件 |
| `/favicon.ico` | 提供 favicon |
| 任何其他路径 | 回退到 `index.html`（SPA 路由） |

文件以适当的 MIME 类型（通过 `mime_guess` 自动检测）和 `Cache-Control: public, max-age=3600` 头提供。

## 嵌入网站

WebChat 可以使用 iframe 或加载 WebChat JavaScript 包直接嵌入到外部网站中。

### iframe 嵌入

```html
<iframe
  src="http://localhost:18790"
  width="400"
  height="600"
  style="border: none; border-radius: 12px;"
></iframe>
```

### CORS 配置

要允许从你的网站进行跨域访问，请配置 CORS 来源：

```toml
[channels.webchat]
cors_origins = [
  "https://your-website.com",
  "https://app.your-website.com",
  "http://localhost:3000"  # 用于开发
]
```

支持通配符模式（例如 `http://localhost:*` 匹配任何端口）。

## 认证

当 Gateway 上启用 `require_auth` 时，WebChat 用户必须认证：

1. **基于令牌** — 在 `connect` 请求中传递 bearer 令牌
2. **设备配对** — 使用配对流程将浏览器注册为受信任设备
3. **无认证**（默认）— 当 `require_auth` 为 `false` 时，接受所有 WebSocket 连接

```toml
[gateway]
require_auth = false  # 设置为 true 用于生产部署
```

<Callout type="warn">
如果你将 Gateway 暴露在网络中（将 `host` 从 `127.0.0.1` 更改为 `0.0.0.0`），强烈建议启用认证。
</Callout>

## 会话管理

WebChat 会话默认使用主会话密钥：

```
agent:main:main
```

WebChat UI 可以指定自定义会话密钥以支持多个对话：

```json
{
  "method": "agent.run",
  "params": {
    "message": "你好",
    "session_key": "agent:main:webchat:session-1"
  }
}
```

会话历史、压缩和持久化与所有其他接口完全相同——由 SQLite 会话存储支持。

## InteractionManifest

WebChat 使用 **WebRich** 范式，提供最完整的能力集：

```rust
InteractionManifest {
    paradigm: InteractionParadigm::WebRich,
    capabilities: {
        RichText,        // 完整 Markdown 支持
        CodeHighlight,   // 代码块中的语法高亮
        MultiGroupUI,    // 标签页、手风琴、可折叠部分
        Streaming,       // 实时令牌流式传输
        MermaidCharts,   // Mermaid 图表渲染
        ImageInline,     // 内联图片显示
        Canvas,          // 用于绘图/编辑的交互式 canvas
    },
    constraints: InteractionConstraints {
        max_output_chars: None,  // 无限制
        supports_streaming: true,
        prefer_compact: false,
    },
}
```

这意味着 Agent 可以通过 WebChat 响应时使用其完整的输出格式范围，包括 Mermaid 图表、交互式 canvas 和结构化的多部分布局。

## 健康检查

HTTP 服务器提供健康端点：

```bash
curl http://127.0.0.1:18790/health
```

这可用于监控和负载均衡器健康检查。

## 故障排除

| 问题 | 解决方案 |
|---------|----------|
| WebChat 页面空白 | 验证 `static_dir` 路径包含构建的 WebChat UI 文件 |
| WebSocket 连接被拒绝 | 确保 Gateway 正在运行且端口与你的 URL 匹配 |
| 浏览器控制台中的 CORS 错误 | 将来源添加到 WebChat 配置中的 `cors_origins` |
| "Authentication required" | 禁用 `require_auth` 或在 `connect` 请求中提供有效令牌 |
| 流式传输不工作 | 在发送 `agent.run` 之前订阅 `stream.*` 事件 |
| 静态资源返回 404 | 检查文件是否存在于静态目录中且具有正确的扩展名 |
| SPA 路由损坏 | 确保 HTTP 服务器配置中的 `fallback_file` 设置为 `"index.html"` |

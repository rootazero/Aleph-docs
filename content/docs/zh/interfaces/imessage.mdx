iMessage 接口通过读取 Messages.app SQLite 数据库中的消息并通过 AppleScript 发送回复来提供原生 macOS 集成。这是一个平台特定的接口，仅在 macOS 上运行。

## 架构

与基于云的接口（Telegram、Discord）不同，iMessage 集成完全是本地的：

```
┌─────────────────────────────────────────────────────────┐
│                     macOS 系统                           │
│                                                          │
│  ┌──────────────┐     ┌──────────────────────────────┐  │
│  │ Messages.app │     │        Aleph Server           │  │
│  │              │     │                               │  │
│  │  chat.db ────┼─────┼──> SQLite 轮询 (只读)          │  │
│  │  (SQLite)    │     │                               │  │
│  │              │ <───┼─── AppleScript (发送消息)      │  │
│  └──────────────┘     └──────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

**接收消息：** Aleph 以只读模式轮询 `~/Library/Messages/chat.db` SQLite 数据库，检测自上次轮询以来的新行。

**发送消息：** Aleph 使用 `osascript` 执行控制 Messages.app 的 AppleScript 命令，通过 iMessage 服务发送文本和文件附件。

## 能力

| 功能 | 状态 |
|---------|--------|
| 文本消息 | 支持 |
| 富文本 (Markdown) | 不支持（仅纯文本） |
| 照片 | 支持（发送和接收） |
| 文档 | 支持（发送和接收） |
| 音频/视频 | 支持（发送和接收） |
| 轻点反馈反应 | 有限（通过 AppleScript 发送不完全支持） |
| 回复线程 | 不支持 |
| 消息编辑 | 不支持 |
| 消息删除 | 不支持 |
| 输入中指示器 | 不支持 |
| 已读回执 | 不支持 |
| 最大消息长度 | ~20,000 字符 |
| 最大附件大小 | 100 MB |

## 前置条件

<Steps>
<Step>

### macOS 要求

iMessage 接口仅在 macOS 上工作。它使用 `#[cfg(target_os = "macos")]` 进行条件编译，在 Linux 或 Windows 构建中不可用。

</Step>
<Step>

### 完全磁盘访问

Aleph 需要**完全磁盘访问**权限来读取 `~/Library/Messages/chat.db`：

1. 打开 **系统设置 > 隐私与安全性 > 完全磁盘访问**
2. 点击 **+** 按钮
3. 添加 Aleph 服务器二进制文件（或如果通过终端运行，则添加 Terminal.app）
4. 重启 Aleph 服务器

没有此权限，数据库打开将失败并显示 "permission denied" 错误。

</Step>
<Step>

### 自动化权限

Aleph 需要**自动化**权限通过 AppleScript 控制 Messages.app：

1. Aleph 第一次发送消息时，macOS 会显示权限对话框
2. 点击 **允许** 授予自动化访问权限
3. 你可以在 **系统设置 > 隐私与安全性 > 自动化** 中管理此权限

</Step>
</Steps>

## 配置

### 最小配置

```toml
[[channels]]
id = "imessage"
channel_type = "imessage"
enabled = true

[channels.config]
enabled = true
```

### 完整配置参考

```toml
[[channels]]
id = "imessage"
channel_type = "imessage"
enabled = true

[channels.config]
# 启用 iMessage 频道（默认：false）
enabled = true

# Messages 数据库路径（默认：~/Library/Messages/chat.db）
db_path = "~/Library/Messages/chat.db"

# 轮询间隔（毫秒）（默认：1000）
poll_interval_ms = 1000

# 入站消息的 DM 策略
# 选项："pairing"、"allowlist"、"open"、"disabled"
dm_policy = "pairing"

# 群组消息策略
# 选项："open"、"allowlist"、"disabled"
group_policy = "open"

# DM 的电话号码/电子邮件白名单
allow_from = ["+15551234567", "user@example.com"]

# 群组聊天的白名单（按聊天标识符）
group_allow_from = []

# 在群组聊天中需要 @mention（默认：true）
require_mention = true

# 用于群组中提及检测的机器人名称
bot_name = "Aleph"

# 在入站消息中包含附件（默认：true）
include_attachments = true

# 最大附件大小（字节）（0 = 无限制）
max_attachment_size = 0

# 入站消息去抖（毫秒）（默认：500）
# 防止处理快速爆发的消息
inbound_debounce_ms = 500
```

## DM 策略

`dm_policy` 设置控制 Aleph 如何处理来自未知发送者的直接消息：

| 策略 | 行为 |
|--------|----------|
| **`pairing`** (默认) | 未知发送者在 Aleph 响应之前被提示输入配对码 |
| **`allowlist`** | 仅处理 `allow_from` 中的发送者；忽略所有其他发送者 |
| **`open`** | 无限制地接受所有发送者 |
| **`disabled`** | 完全禁用 DM |

<Callout type="info">
**pairing** 策略与 Aleph 的邀请系统集成。所有者可以生成配对码，当未知发送者通过 iMessage 发送该码时，他们的电话号码或电子邮件会自动添加到白名单。
</Callout>

### 群组策略

| 策略 | 行为 |
|--------|----------|
| **`open`** (默认) | 接受所有群聊（带有可选的提及要求） |
| **`allowlist`** | 仅处理 `group_allow_from` 中的群组 |
| **`disabled`** | 完全禁用群组消息 |

## 消息轮询如何工作

轮询循环在后台 Tokio 任务中运行：

1. **打开数据库** — `chat.db` 以 `SQLITE_OPEN_READ_ONLY` 只读模式打开
2. **记录最后 ROWID** — 启动时，当前最大消息 ROWID 被记录为基线
3. **轮询循环** — 每 `poll_interval_ms` 毫秒：
   - 查询 `ROWID > last_seen` 且 `is_from_me = 0` 的消息
   - 对于每条新消息，解析发送者句柄和聊天信息
   - 如果设置了 `cache_has_attachments`，提取附件
   - 转换为 `InboundMessage` 并通过 mpsc 频道发送
4. **更新标记** — 每批处理后更新最后看到的 ROWID

### Apple 时间戳

Messages.app 使用 "Apple Cocoa Core Data 时间戳" — 自 2001 年 1 月 1 日 UTC 以来的纳秒数。Aleph 将其转换为标准 Unix 时间戳：

```
unix_timestamp = apple_timestamp / 1,000,000,000 + 978,307,200
```

## 发送消息

消息通过控制 Messages.app 的 AppleScript 发送：

### 文本消息

```applescript
tell application "Messages"
    set targetService to 1st account whose service type = iMessage
    set targetBuddy to participant "+15551234567" of targetService
    send "Hello from Aleph!" to targetBuddy
end tell
```

### 文件附件

```applescript
tell application "Messages"
    set targetService to 1st account whose service type = iMessage
    set targetBuddy to participant "+15551234567" of targetService
    set theFile to POSIX file "/path/to/file.png"
    send theFile to targetBuddy
end tell
```

### 群组聊天消息

群组聊天通过聊天 ID 而不是参与者来寻址：

```applescript
tell application "Messages"
    set targetChat to chat id "chat123456"
    send "Hello group!" to targetChat
end tell
```

## 媒体处理

### 接收附件

附件通过消息表中的 `cache_has_attachments` 标志检测，并通过 `message_attachment_join` 表解析：

| 字段 | 来源 |
|-------|--------|
| `id` | 附件 GUID |
| `mime_type` | 来自 `attachment.mime_type` 列 |
| `filename` | 来自 `attachment.filename` 或 `transfer_name` |
| `size` | 来自 `attachment.total_bytes` |

### 发送附件

出站文件附件必须提供为本地文件路径。文件通过 AppleScript `send POSIX file` 命令发送。

## 会话路由

iMessage 对话根据聊天上下文映射到会话密钥：

| 上下文 | 会话密钥 |
|---------|------------|
| 与 +15551234567 的 DM | `agent:main:dm:+15551234567` 或 `agent:main:imessage:dm:+15551234567` |
| 群组聊天（聊天 ID） | `agent:main:imessage:group:{chat_identifier}` |

`chat_identifier` 通常是 DM 的电话号码或群组聊天的内部 ID。

## 数据库架构

Aleph 从 `chat.db` 中的三个主要表读取：

### `message` 表

| 列 | 类型 | 用途 |
|--------|------|---------|
| `ROWID` | INTEGER | 自增消息 ID |
| `guid` | TEXT | 唯一消息标识符 |
| `text` | TEXT | 消息正文 |
| `handle_id` | INTEGER | `handle` 表的外键 |
| `date` | INTEGER | Apple 时间戳（纳秒） |
| `is_from_me` | INTEGER | 如果由 Mac 所有者发送则为 1 |
| `cache_has_attachments` | INTEGER | 如果消息有附件则为 1 |

### `handle` 表

| 列 | 类型 | 用途 |
|--------|------|---------|
| `ROWID` | INTEGER | 句柄 ID |
| `id` | TEXT | 电话号码或电子邮件 |
| `service` | TEXT | "iMessage" 或 "SMS" |

### `chat` 表

| 列 | 类型 | 用途 |
|--------|------|---------|
| `ROWID` | INTEGER | 聊天 ID |
| `guid` | TEXT | 聊天 GUID |
| `chat_identifier` | TEXT | 电话号码或群组 ID |
| `display_name` | TEXT | 群组名称（如果已设置） |
| `group_id` | TEXT | 群组聊天非空 |

## 限制

- **仅限 macOS** — 无法在 Linux 或 Windows 上运行
- **需要完全磁盘访问** — 数据库受 macOS TCC（透明度、同意和控制）保护
- **非实时交付** — 基于轮询，可配置间隔（最小实际值：~500ms）
- **无富文本** — iMessage 内部支持富文本，但 AppleScript 仅发送纯文本
- **无输入中指示器** — 需要更深入的系统集成
- **轻点反馈反应** — 跨 macOS 版本通过 AppleScript 发送轻点反馈不可靠支持；接收轻点反馈尚未实现
- **无消息编辑/删除** — AppleScript 不公开这些操作

## 故障排除

| 问题 | 解决方案 |
|---------|----------|
| "Failed to open Messages database" | 在系统设置中授予 Aleph 二进制文件完全磁盘访问权限 |
| "AppleScript execution failed" | 授予自动化权限；确保 Messages.app 已安装并登录 |
| 机器人看不到新消息 | 验证 `db_path` 指向正确的数据库；检查 `is_from_me = 0` 过滤器是否工作 |
| Messages.app 意外打开 | 这是正常的 — AppleScript 在发送时可能会激活应用 |
| 响应时间慢 | 减少 `poll_interval_ms`（例如到 500）；注意这会增加 SQLite 读取频率 |
| "Invalid target" 错误 | 确保收件人是有效的电话号码（带国家代码）或电子邮件地址 |
| 群组消息被忽略 | 检查 `group_policy` 和 `group_allow_from`；验证 `require_mention` 设置 |

Aleph 使用标准的 Rust 测试生态系统 —— `cargo test` 作为运行器，并附加了用于 BDD 风格集成测试和基准测试的框架。本页介绍如何运行测试、它们的组织方式以及如何编写新测试。

## 测试框架栈

| 工具 | 用途 |
|------|------|
| `cargo test` | 单元测试和集成测试的内置测试运行器 |
| `criterion` | 统计基准测试（支持异步 Tokio） |
| `cucumber` | BDD 风格的功能测试（Gherkin 语法） |
| `tempfile` | 用于测试隔离的临时目录 |
| `serial_test` | 序列化共享全局状态的测试 |
| `filetime` | 用于 watcher 测试的文件时间戳操作 |

这些在 `core/Cargo.toml` 的 dev-dependencies 中声明：

```toml
[dev-dependencies]
criterion = { version = "0.5", features = ["html_reports", "async_tokio"] }
cucumber = { version = "0.21", features = ["macros"] }
tempfile = "3.8"
filetime = "0.2"
serial_test = "3.0"
aleph-sdk = { path = "../apps/shared" }
aleph-protocol = { path = "../shared/protocol" }
```

## 测试类别

### 单元测试

单元测试位于它们测试的源文件**内部**，在 `#[cfg(test)]` 模块中：

```rust
// src/memory/context.rs

pub fn normalize_fact(raw: &str) -> String {
    raw.trim().to_lowercase()
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_normalize_fact_trims_whitespace() {
        assert_eq!(normalize_fact("  hello  "), "hello");
    }

    #[test]
    fn test_normalize_fact_lowercases() {
        assert_eq!(normalize_fact("Hello World"), "hello world");
    }
}
```

异步单元测试使用 `tokio::test` 宏：

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[tokio::test]
    async fn test_session_creation() {
        let manager = SessionManager::new_test().await;
        let session = manager.create("test-user").await.unwrap();
        assert!(!session.id.is_empty());
    }
}
```

### 集成测试

集成测试位于 `core/tests/` 中，测试跨模块的交互。它们作为单独的 crate 编译，因此只能使用 `alephcore` 的公共 API。

```
core/tests/
├── cucumber.rs          # BDD 测试运行器（Gherkin 功能）
├── features/            # Cucumber 测试的 .feature 文件
└── ...
```

Cucumber 测试运行器配置为自定义测试工具：

```toml
[[test]]
name = "cucumber"
harness = false
```

### 文档测试

`///` 注释中的文档示例会被编译并作为测试运行：

```rust
/// 从字符串解析任务状态。
///
/// # 示例
///
/// ```
/// use alephcore::poe::types::TaskStatus;
///
/// let status: TaskStatus = "pending".parse().unwrap();
/// assert_eq!(status, TaskStatus::Pending);
/// ```
impl FromStr for TaskStatus {
    // ...
}
```

### 基准测试

Criterion 基准测试位于 `core/benches/` 中，并在 `Cargo.toml` 中声明：

```toml
[[bench]]
name = "memory_benchmarks_simple"
harness = false

[[bench]]
name = "approval_performance"
harness = false
```

## 运行测试

### 所有测试

```bash
# 运行工作区中的所有测试
cargo test --workspace

# 仅运行 core crate 中的测试
cargo test -p alephcore

# 运行 CLI crate 中的测试
cargo test -p aleph-cli
```

### 特定测试

```bash
# 按名称运行单个测试
cargo test test_normalize_fact_trims_whitespace

# 运行模块中的所有测试
cargo test memory::context::tests

# 运行匹配模式的测试
cargo test session

# 运行特定的集成测试文件
cargo test --test cucumber
```

### 测试选项

```bash
# 显示通过测试的 stdout（通常被抑制）
cargo test -- --nocapture

# 顺序运行测试（对调试有用）
cargo test -- --test-threads=1

# 列出所有测试名称而不运行它们
cargo test -- --list

# 运行被忽略的测试（通常是慢的或需要外部服务的）
cargo test -- --ignored
```

### 基准测试

```bash
# 运行所有基准测试
cargo bench

# 运行特定的基准测试
cargo bench --bench memory_benchmarks_simple

# 使用 HTML 报告进行基准测试
cargo bench -- --output-format=bencher
```

Criterion 将 HTML 报告写入 `target/criterion/`。

### 特性门控测试

某些测试需要启用特定特性：

```bash
# 运行需要 test-helpers 特性的测试
cargo test --features test-helpers

# 运行 gateway 特性的测试
cargo test --features gateway

# 使用所有特性运行完整测试套件
cargo test --all-features
```

## 测试夹具和辅助函数

### 临时目录

为创建文件或数据库的测试使用 `tempfile`：

```rust
use tempfile::TempDir;

#[tokio::test]
async fn test_database_persistence() {
    let tmp = TempDir::new().unwrap();
    let db_path = tmp.path().join("test.db");

    let store = SqliteStore::open(&db_path).await.unwrap();
    store.insert_fact("test fact").await.unwrap();

    // tmp 在末尾被丢弃，自动清理
}
```

### 序列化测试

共享全局状态（环境变量、单例注册表）的测试使用 `serial_test` crate：

```rust
use serial_test::serial;

#[test]
#[serial]
fn test_config_from_env() {
    std::env::set_var("ANTHROPIC_API_KEY", "test-key");
    let config = Config::from_env().unwrap();
    assert_eq!(config.api_key, "test-key");
    std::env::remove_var("ANTHROPIC_API_KEY");
}
```

### 测试配置

需要有效的 `Config` 对象但不要真实 API 密钥的测试：

```rust
fn test_config() -> Config {
    Config {
        api_key: "test-key-not-real".into(),
        base_url: "http://localhost:9999".into(),
        model: "test-model".into(),
        ..Config::default()
    }
}
```

## Mock 模式

Aleph 主要使用**基于 trait 的 Mock** —— 为外部边界定义 trait 并提供测试实现。

### 基于 Trait 的测试替身

```rust
// 生产 trait
#[async_trait]
pub trait MemoryStore: Send + Sync {
    async fn store_fact(&self, fact: MemoryFact) -> Result<()>;
    async fn search(&self, query: &str, limit: usize) -> Result<Vec<MemoryFact>>;
}

// 测试实现
#[cfg(test)]
pub struct MockMemoryStore {
    facts: std::sync::Mutex<Vec<MemoryFact>>,
}

#[cfg(test)]
#[async_trait]
impl MemoryStore for MockMemoryStore {
    async fn store_fact(&self, fact: MemoryFact) -> Result<()> {
        self.facts.lock().unwrap().push(fact);
        Ok(())
    }

    async fn search(&self, query: &str, limit: usize) -> Result<Vec<MemoryFact>> {
        let facts = self.facts.lock().unwrap();
        Ok(facts.iter()
            .filter(|f| f.content.contains(query))
            .take(limit)
            .cloned()
            .collect())
    }
}
```

### 集成测试的进程内服务器

对于网关集成测试，在测试进程中启动真实服务器：

```rust
#[tokio::test]
async fn test_gateway_roundtrip() {
    // 在随机端口上启动服务器
    let server = TestServer::start().await;
    let addr = server.addr();

    // 连接客户端
    let client = AlephClient::connect(&format!("ws://{addr}")).await.unwrap();

    // 发送请求
    let response = client.send_message("hello").await.unwrap();
    assert!(response.is_ok());
}
```

## CI 流水线

### 推荐的 CI 步骤

Aleph 的典型 CI 工作流：

```yaml
# .github/workflows/ci.yml
name: CI

on: [push, pull_request]

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable

      # 格式检查
      - run: cargo fmt --all -- --check

      # 代码检查
      - run: cargo clippy --workspace --all-targets -- -D warnings

      # 类型检查（比完整构建更快）
      - run: cargo check --workspace

      # 测试
      - run: cargo test --workspace

      # 文档测试
      - run: cargo doc --workspace --no-deps
```

### 关键 CI 检查

| 检查 | 命令 | 用途 |
|------|------|------|
| 格式 | `cargo fmt --all -- --check` | 一致的代码格式化 |
| 代码检查 | `cargo clippy --workspace -- -D warnings` | 捕获常见错误 |
| 类型检查 | `cargo check --workspace` | 快速编译验证 |
| 单元测试 | `cargo test --workspace` | 正确性验证 |
| 文档构建 | `cargo doc --workspace --no-deps` | 确保文档注释可编译 |

## 测试覆盖率

### 生成覆盖率报告

使用 `cargo-llvm-cov` 生成覆盖率报告：

```bash
# 安装
cargo install cargo-llvm-cov

# 生成 HTML 报告
cargo llvm-cov --workspace --html

# 打开报告
open target/llvm-cov/html/index.html

# 生成 LCOV 格式（用于 CI 集成）
cargo llvm-cov --workspace --lcov --output-path coverage.lcov
```

### 覆盖率指南

Aleph 不强制执行严格的覆盖率百分比，但遵循以下原则：

- **核心逻辑**（agent 循环、内存、工具）应该有彻底的单元测试
- **解析和序列化**（FromStr, Serialize/Deserialize）应该测试往返
- **错误路径**应该与成功路径一样彻底测试
- **集成边界**（数据库、网络）应该使用真实或进程内后端进行集成测试
- **UI 代码**（Control Plane）是手动测试的 —— WASM 测试工具有限

## 编写良好的测试

### 命名约定

测试名称描述场景和预期结果：

```rust
#[test]
fn parse_task_status_returns_pending_for_lowercase_input() { ... }

#[test]
fn session_manager_rejects_expired_tokens() { ... }

#[test]
fn memory_search_returns_empty_vec_when_no_facts_match() { ... }
```

### 测试结构（Arrange-Act-Assert）

```rust
#[tokio::test]
async fn memory_store_returns_facts_sorted_by_relevance() {
    // Arrange
    let store = setup_test_store().await;
    store.insert(fact("Rust is a systems language")).await.unwrap();
    store.insert(fact("Python is interpreted")).await.unwrap();

    // Act
    let results = store.search("Rust programming", 10).await.unwrap();

    // Assert
    assert!(!results.is_empty());
    assert!(results[0].content.contains("Rust"));
}
```

### 避免测试反模式

- 不要对 AI 生成的输出断言精确的字符串相等性
- 不要依赖单元测试中的网络可用性
- 不要在未使用 `serial_test` 的情况下跨测试共享可变状态
- 不要使用 `sleep` 进行同步 —— 使用通道或条件变量

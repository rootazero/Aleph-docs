本页记录了 Aleph Rust 代码库中使用的核心设计模式。这些不是抽象的建议 —— 它们是正在积极使用的模式，配有从生产代码中提取的具体示例。遵循它们可以保持代码库的一致性，并使新贡献者更容易上手。

有关文件和模块结构的相关指导，请参阅本页底部的 [模块组织](#模块组织) 部分。

## Context 模式

### 问题

随着 API 的演进，函数签名会累积参数。一个带有七个参数的函数难以阅读、难以扩展且容易误用：

```rust
// 之前：7 个位置参数
pub async fn run(
    &self,
    request: String,
    context: RequestContext,
    tools: Vec<UnifiedTool>,
    identity: IdentityContext,
    callback: impl LoopCallback,
    abort_signal: Option<watch::Receiver<bool>>,
    initial_history: Option<String>,
) -> LoopResult
```

### 解决方案

将相关参数分组到 **context 结构体**中。必需参数放在构造函数中；可选参数使用 builder 风格的 `with_*` 方法：

```rust
#[derive(Clone)]
pub struct RunContext {
    pub request: String,
    pub context: RequestContext,
    pub tools: Vec<UnifiedTool>,
    pub identity: IdentityContext,
    pub abort_signal: Option<watch::Receiver<bool>>,
    pub initial_history: Option<String>,
}

impl RunContext {
    pub fn new(
        request: impl Into<String>,
        context: RequestContext,
        tools: Vec<UnifiedTool>,
        identity: IdentityContext,
    ) -> Self {
        Self {
            request: request.into(),
            context,
            tools,
            identity,
            abort_signal: None,
            initial_history: None,
        }
    }

    pub fn with_abort_signal(mut self, signal: watch::Receiver<bool>) -> Self {
        self.abort_signal = Some(signal);
        self
    }

    pub fn with_initial_history(mut self, history: impl Into<String>) -> Self {
        self.initial_history = Some(history.into());
        self
    }
}
```

函数签名变得简洁且稳定：

```rust
// 之后：2 个参数
pub async fn run(
    &self,
    run_context: RunContext,
    callback: impl LoopCallback,
) -> LoopResult
```

### 用法

```rust
let run_context = RunContext::new(request, RequestContext::empty(), tools, identity)
    .with_abort_signal(abort_rx)
    .with_initial_history(history);

let result = agent_loop.run(run_context, callback).await;
```

### 何时应用

- 函数有 **5 个或更多**参数
- 多个参数是可选的
- 参数形成一个逻辑组
- API 可能会随着新参数而演进
- 函数从多个位置被调用

### 代码库中的位置

- `agent_loop::RunContext` —— agent 循环执行参数

## Newtype 模式

### 问题

原始类型不携带语义含义。一个接受两个 `String` 参数（用于不同类型的 ID）的函数，即使在参数交换时也能编译：

```rust
fn assign(experiment_id: String, variant_id: String) { ... }

// 能编译，但是错误的 —— 参数被交换了
assign(variant_id, experiment_id);
```

### 解决方案

将原始类型包装在 **newtype 结构体**中，在类型级别赋予它们不同的标识：

```rust
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub struct ExperimentId(String);

impl ExperimentId {
    pub fn new(id: impl Into<String>) -> Self {
        Self(id.into())
    }

    pub fn as_str(&self) -> &str {
        &self.0
    }
}

impl Deref for ExperimentId {
    type Target = str;
    fn deref(&self) -> &Self::Target {
        &self.0
    }
}

impl Display for ExperimentId {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}", self.0)
    }
}

impl From<String> for ExperimentId {
    fn from(s: String) -> Self {
        Self(s)
    }
}
```

现在编译器会捕获这个错误：

```rust
fn assign(experiment_id: ExperimentId, variant_id: VariantId) { ... }

let exp = ExperimentId::new("exp-001");
let var = VariantId::new("control");

assign(exp, var);   // 正确
assign(var, exp);   // 编译错误！
```

### 集合 Newtype

Newtype 也适用于带有领域特定操作的集合：

```rust
#[derive(Debug, Clone)]
pub struct Ruleset(Vec<PermissionRule>);

impl Ruleset {
    pub fn new() -> Self {
        Self(Vec::new())
    }

    pub fn add(&mut self, rule: PermissionRule) {
        self.0.push(rule);
    }

    pub fn rules(&self) -> &[PermissionRule] {
        &self.0
    }
}

impl FromIterator<PermissionRule> for Ruleset {
    fn from_iter<T: IntoIterator<Item = PermissionRule>>(iter: T) -> Self {
        Self(iter.into_iter().collect())
    }
}
```

### 标准 Trait 清单

每个 newtype 应该实现一组一致的 trait：

**必需：**
- `Debug` —— 调试输出
- `Clone` —— 值复制
- `PartialEq`, `Eq` —— 相等比较
- `Hash` —— 用于 `HashMap`/`HashSet`

**推荐：**
- `Display` —— 面向用户的字符串输出
- `From<T>` —— 符合人体工程学的构造
- `Deref` —— 透明访问内部类型
- `Serialize`, `Deserialize` —— JSON/TOML 支持

**可选：**
- `FromStr` —— 从字符串解析
- `FromIterator` —— 用于集合 newtype
- `Default` —— 如果存在合理的默认值

### Newtype 目录

| 类型 | 内部类型 | 用途 | 模块 |
|------|----------|------|------|
| `ExperimentId` | `String` | A/B 测试实验 ID | `ab_testing/types` |
| `VariantId` | `String` | A/B 测试变体 ID | `ab_testing/types` |
| `ContextId` | `String` | 浏览器上下文 ID | `browser/context_registry` |
| `TaskId` | `String` | 浏览器任务 ID | `browser/context_registry` |
| `SubscriptionId` | `String` | 事件总线订阅 ID | `event/global_bus` |
| `Ruleset` | `Vec<PermissionRule>` | 权限规则集合 | `permission/rule` |
| `Answer` | `Vec<String>` | 用户问题选择 | `event/question` |

## Builder 模式

### 问题

复杂对象有很多字段，有些是必需的，有些是可选的。一个有 10 个参数的构造函数不可用，而手动填写每个字段既冗长又容易出错。

### 解决方案

为必需参数提供**构造函数**，为可选参数提供**流畅的 `with_*` 方法**：

```rust
impl RunContext {
    // 构造函数中的必需参数
    pub fn new(
        request: impl Into<String>,
        context: RequestContext,
        tools: Vec<UnifiedTool>,
        identity: IdentityContext,
    ) -> Self { ... }

    // 通过 builder 方法设置可选参数
    pub fn with_abort_signal(mut self, signal: watch::Receiver<bool>) -> Self {
        self.abort_signal = Some(signal);
        self
    }

    pub fn with_initial_history(mut self, history: impl Into<String>) -> Self {
        self.initial_history = Some(history.into());
        self
    }
}
```

用法读起来像一个句子：

```rust
let context = RunContext::new(request, ctx, tools, identity)
    .with_abort_signal(abort_rx)
    .with_initial_history(history);
```

### 好处

- **流畅的 API** —— 方法链读起来很自然
- **自文档化** —— 每个 `with_*` 方法都命名了它设置的参数
- **编译时安全** —— 必需参数由构造函数强制执行
- **向后兼容** —— 添加新的 `with_*` 方法不会破坏现有调用者

注意，在 Aleph 中，Builder 模式通常与 Context 模式结合使用，而不是与单独的 `Builder` 结构体一起独立使用。

## FromStr Trait 模式

### 问题

许多类型需要从字符串解析 —— 配置值、枚举变体、用户输入。没有标准接口，每种类型都会发明自己的 `parse` 或 `from_string` 方法。

### 解决方案

实现 `FromStr` 以与 Rust 的标准 `str::parse()` 集成：

```rust
impl FromStr for TaskStatus {
    type Err = String;

    fn from_str(s: &str) -> Result<Self, Self::Err> {
        match s.to_lowercase().as_str() {
            "pending" => Ok(Self::Pending),
            "running" => Ok(Self::Running),
            "completed" => Ok(Self::Completed),
            "failed" => Ok(Self::Failed),
            _ => Err(format!("Invalid TaskStatus: {}", s)),
        }
    }
}
```

这实现了标准的 `.parse()` API 和通用解析函数：

```rust
// 直接解析
let status: TaskStatus = "pending".parse()?;

// 在配置加载中
let status = config.get("status")?.parse::<TaskStatus>()?;

// 通用辅助函数
fn parse_config<T: FromStr>(value: &str) -> Result<T, T::Err> {
    value.parse()
}
```

### Aleph 中实现 FromStr 的类型

以下类型为实现一致解析而实现 `FromStr`：

- **任务/会话类型：** `TaskStatus`, `SessionStatus`, `TraceRole`
- **内存类型：** `FactType`, `FactSpecificity`, `TemporalScope`
- **扩展类型：** `HookKind`, `HookPriority`, `PromptScope`
- **设备类型：** `DeviceType`, `DeviceRole`
- **运行时类型：** `RuntimeKind`, `EvolutionStatus`, `EventType`
- **风险类型：** `RiskLevel`, `Lane`

### 约定

- 使用 `to_lowercase()` 进行不区分大小写的解析
- 返回 `String` 作为错误类型（简单，无额外依赖）
- 匹配所有已知变体，对未知值返回 `Err`

## 模式组合

Aleph 中的模式很少单独使用。最常见的组合：

### Context + Builder

`RunContext` 使用 Context 模式来分组参数，使用 Builder 模式来处理可选字段：

```rust
let run_context = RunContext::new(required_params)
    .with_optional_param1(value1)
    .with_optional_param2(value2);
```

### Newtype + FromStr

许多 newtype 实现 `FromStr`，以便可以从配置文件解析：

```rust
let status: TaskStatus = "pending".parse()?;
let device: DeviceType = config.get("device")?.parse()?;
```

### Newtype + Deref

包装 `String` 的 Newtype 实现 `Deref<Target = str>` 以实现透明访问：

```rust
let id = ExperimentId::new("exp-001");
if id.starts_with("exp-") { ... }  // 通过 Deref 到 str 工作
```

## 错误处理约定

Aleph 使用两层错误处理方法：

### 边界错误：`thiserror`

对于跨越模块边界且需要调用者匹配的错误，使用 `thiserror` 来派生结构化错误类型：

```rust
#[derive(Debug, thiserror::Error)]
pub enum GatewayError {
    #[error("authentication failed: {0}")]
    AuthFailed(String),
    #[error("session not found: {0}")]
    SessionNotFound(String),
    #[error("rate limit exceeded for client {client_id}")]
    RateLimited { client_id: String },
    #[error("internal error: {0}")]
    Internal(#[from] anyhow::Error),
}
```

### 内部错误：`anyhow`

对于模块内部调用者不需要区分变体的错误，使用 `anyhow::Result`：

```rust
use anyhow::{Context, Result};

fn load_plugin(path: &Path) -> Result<Plugin> {
    let bytes = std::fs::read(path)
        .context("failed to read plugin file")?;
    let manifest = parse_manifest(&bytes)
        .context("invalid plugin manifest")?;
    Ok(Plugin::from_manifest(manifest))
}
```

### 经验法则

- 如果错误跨越 **crate 边界**或是 **公共 API** 的一部分，使用 `thiserror`
- 如果错误在 **单个模块** 或 **二进制入口点** 内处理，使用 `anyhow`
- 在 `anyhow` 错误上使用 `.context()` 添加人类可读的描述

## 模块组织

这些原则规定了文件和模块的结构。另请参阅 [代码组织指南](https://github.com/rootazero/Aleph/blob/main/docs/reference/CODE_ORGANIZATION.md) 获取完整参考。

### 核心原则

**单一职责。** 每个文件只拥有一个概念。如果你需要用 "and" 来描述一个文件的功能，它需要被拆分。

**关注点分离。** 沿着自然的断层线拆分：

| 关注点 | 文件名 |
|--------|--------|
| 类型定义（struct/enum） | `types.rs` 或 `model.rs` |
| 业务逻辑 | 主模块文件 |
| 外部集成（数据库、网络） | `store.rs`, `executor.rs` |
| 测试替身 | `mock.rs`，位于 `#[cfg(test)]` 下 |
| 错误类型 | `error.rs` |

**可见性最小化。** 对跨模块访问使用 `pub(crate)`。将 `pub` 保留给 crate 的公共 API 表面。内部细节应该是 `pub(super)` 或私有的。

### 标准文件名

| 文件名 | 内容 |
|--------|------|
| `mod.rs` | 模块入口点，重新导出（保持精简） |
| `types.rs` | 枚举和值对象 |
| `model.rs` | 聚合根和实体 |
| `pool.rs` | 连接/资源池 |
| `factory.rs` | 构造函数，builder |
| `executor.rs` | 调用外部系统的逻辑 |
| `registry.rs` | 查找和查询逻辑 |
| `callback.rs` | 事件处理程序和钩子 |
| `mock.rs` | 测试替身（位于 `#[cfg(test)]` 下） |
| `error.rs` | 领域特定错误枚举 |

### 何时拆分文件

**必须拆分**当：
- 行数达到 500+ 且文件涵盖多个概念
- 存在针对不同类型的多个 `impl Trait for T` 块
- 生产代码与测试替身混合
- 单个结构体有 20+ 个公共方法跨越不相关的关注点

**考虑拆分**当：
- 单个 `impl` 块超过 300 行
- 一个函数超过 100 行
- `use` 导入跨越 3 个以上不相关的模块

### 模块模式

**单结构体模块** —— 用于以一个结构体为中心的模块：
```
my_module/
├── mod.rs      # MyStruct 定义 + 核心 impl
├── types.rs    # 支持枚举和值对象
└── error.rs    # MyModuleError 枚举
```

**领域模型模块** —— 用于具有多种类型的丰富领域模型：
```
memory/
├── mod.rs      # 模块入口，重新导出
├── types.rs    # FactType, MemoryLayer, MemoryCategory 等
├── model.rs    # MemoryFact（聚合根）, CompressionSession
├── anchor.rs   # ContextAnchor, MemoryEntry（查询结构）
└── store/      # 持久化实现
    └── lance/
        └── facts.rs
```

**Manager Facade** —— 用于将工作委派给子组件的大型 manager：
```
extension/
├── mod.rs          # ExtensionManager（精简 facade，委派）
├── executor.rs     # PluginExecutor —— 工具/钩子执行
├── registry.rs     # SkillRegistry —— 技能/命令查找
├── controller.rs   # ServiceController —— 启动/停止/状态
├── loader.rs       # 插件加载和发现
└── types.rs        # ExtensionConfig, LoadSummary
```

**Startup Builder** —— 用于复杂的初始化序列：
```
bin/aleph_server/commands/
├── start.rs        # 入口点：解析参数，调用 builder
└── builder/
    ├── mod.rs      # ServerBuilder 结构体
    ├── providers.rs
    ├── tools.rs
    ├── gateway.rs
    ├── channels.rs
    └── config.rs
```

### 需要避免的反模式

- **上帝对象：** 一个结构体有 40+ 个方法跨越不相关的关注点。拆分为一个委派给专注子组件的 facade。
- **扁平脚本：** 一个 700 行的初始化函数。提取一个具有单独 `initialize_*` 方法的 builder。
- **类型倾倒场：** 一个文件中有 14 个类型和 31 个 impl 块。分离到 `types.rs`、`model.rs` 和领域特定文件中。

## 迁移指南

### 添加 Context 模式

1. 识别一个有 5+ 参数的函数（几个可选）
2. 创建一个 context 结构体，包含必需字段 + 可选字段的 `Option`
3. 实现 `new()`，只有必需参数
4. 为每个可选参数添加 `with_*` builder 方法
5. 更新函数签名以接受 context
6. 更新所有调用点
7. 在模块的公共 API 中导出 context 类型

### 添加 Newtype

1. 识别一个携带语义含义的原始类型
2. 创建一个包装该原始的元组结构体
3. 派生 `Debug`, `Clone`, `PartialEq`, `Eq`, `Hash`
4. 实现 `new()`, `as_str()`（或等效访问器）
5. 根据需要实现 `Deref`, `Display`, `From<T>`
6. 更新所有使用点以使用 newtype
7. 将类型添加到 newtype 目录

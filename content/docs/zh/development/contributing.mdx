Aleph 欢迎社区贡献。无论你是修复拼写错误、添加功能还是改进文档，本指南都会解释从 fork 到合并的完整流程。

## 入门

1. **Fork** 仓库 [github.com/rootazero/Aleph](https://github.com/rootazero/Aleph)
2. **克隆** 你的 fork 到本地：
   ```bash
   git clone https://github.com/<your-username>/Aleph.git
   cd Aleph
   ```
3. **设置上游** 以与主仓库保持同步：
   ```bash
   git remote add upstream https://github.com/rootazero/Aleph.git
   ```
4. **安装前置条件** —— 参阅 [构建](./building) 指南了解工具链设置
5. **运行测试** 以确认你的环境正常工作：
   ```bash
   cargo test --workspace
   ```

## 开发工作流

### 分支命名

从 `main` 创建一个具有描述性名称的分支：

| 前缀 | 用途 | 示例 |
|------|------|------|
| `feat/` | 新功能 | `feat/telegram-voice-messages` |
| `fix/` | Bug 修复 | `fix/memory-search-timeout` |
| `refactor/` | 代码重构 | `refactor/extension-manager-split` |
| `docs/` | 文档更改 | `docs/tool-system-guide` |
| `test/` | 测试添加或修复 | `test/gateway-integration` |
| `chore/` | 构建、CI、依赖更新 | `chore/update-tokio-1.36` |

```bash
git checkout main
git pull upstream main
git checkout -b feat/my-feature
```

### 进行修改

1. 进行专注的增量更改 —— 每个提交一个逻辑更改
2. 在推送前运行检查套件：
   ```bash
   # 格式化
   cargo fmt --all

   # 代码检查
   cargo clippy --workspace --all-targets -- -D warnings

   # 测试
   cargo test --workspace
   ```
3. 推送到你的 fork：
   ```bash
   git push origin feat/my-feature
   ```

### 拉取请求流程

1. 在 upstream 仓库上针对 `main` 打开 PR
2. 使用以下内容填写 PR 模板：
   - **What** 更改做了什么
   - **Why** 需要它
   - **How** 测试它
3. 确保 CI 通过（格式、代码检查、测试）
4. 请求维护者审查
5. 用新提交处理审查反馈（审查期间不要 force-push）
6. 一旦获得批准，维护者会 squash-merge 该 PR

## 代码风格

### 格式化

Aleph 使用默认配置的 `rustfmt`。在每次提交前运行它：

```bash
cargo fmt --all
```

CI 会拒绝 `cargo fmt --all -- --check` 失败的 PR。

### 代码检查

Clippy 是标准的 Rust linter。所有警告在 CI 中都被视为错误：

```bash
cargo clippy --workspace --all-targets -- -D warnings
```

常见 Clippy 修复：

- 在函数参数中优先使用 `&str` 而不是 `&String`
- 使用 `if let` 代替只有单个分支和通配符的 `match`
- 当值不被消耗时，用借用替换 `.clone()`
- 使用 `?` 运算符代替 `Result` 上的显式 `match`

### 命名约定

| 项目 | 约定 | 示例 |
|------|------|------|
| Crate 名称 | `kebab-case` | `aleph-protocol` |
| 模块名称 | `snake_case` | `agent_loop` |
| 类型（struct, enum） | `PascalCase` | `RunContext`, `TaskStatus` |
| 函数、方法 | `snake_case` | `execute_tool`, `parse_config` |
| 常量 | `SCREAMING_SNAKE_CASE` | `DEFAULT_PORT`, `MAX_RETRIES` |
| 特性标志 | `kebab-case` | `control-plane`, `plugin-wasm` |
| 类型参数 | 单个大写字母或短 `PascalCase` | `T`, `Cb` |

### 可见性

遵循最小可见性原则：

```rust
// 对内部跨模块访问优先使用 pub(crate)
pub(crate) fn internal_helper() { ... }

// 对父模块访问使用 pub(super)
pub(super) fn module_helper() { ... }

// 将 pub 保留给 crate 的公共 API 表面
pub fn public_api_method() { ... }
```

### 错误处理

Aleph 使用两层错误策略：

- **`thiserror`** 用于跨模块边界的领域特定错误类型：
  ```rust
  #[derive(Debug, thiserror::Error)]
  pub enum MemoryError {
      #[error("fact not found: {0}")]
      NotFound(String),
      #[error("vector search failed: {0}")]
      SearchFailed(#[from] lancedb::Error),
  }
  ```
- **`anyhow`** 用于调用者不需要区分错误变体的内部函数：
  ```rust
  use anyhow::Result;

  fn load_config() -> Result<Config> {
      let text = std::fs::read_to_string("~/.aleph/config.toml")?;
      let config: Config = toml::from_str(&text)?;
      Ok(config)
  }
  ```

### 模块组织

参阅 [设计模式](./design-patterns) 了解详细模式。关键规则：

- 当文件包含多个概念时，保持文件在 500 行以下
- 将类型定义（`types.rs`）与业务逻辑分离
- 使用 `mod.rs` 作为精简的入口点并重新导出，而不是倾倒场
- 在 `#[cfg(test)]` 下门控测试替身（`MockFoo`）

## 提交信息约定

### 格式

```
<scope>: <description>

[optional body]

[optional footer]
```

### 范围

使用正在更改的模块或区域：

| 范围 | 区域 |
|------|------|
| `core` | 核心库更改 |
| `gateway` | 网关/WebSocket 服务器 |
| `agent` | Agent 循环和 AI 逻辑 |
| `memory` | 内存系统 |
| `tools` | 工具系统 |
| `extension` | 扩展/插件系统 |
| `cli` | CLI 应用程序 |
| `desktop` | 桌面（Tauri）应用程序 |
| `protocol` | 共享协议类型 |
| `docs` | 文档 |
| `ci` | CI/CD 流水线 |
| `deps` | 依赖更新 |

### 示例

```
gateway: add rate limiting to WebSocket connections

Implement per-client rate limiting using a token bucket algorithm.
Connections exceeding 100 requests/minute receive a JSON-RPC error
response with code -32005.

Closes #42
```

```
memory: fix fact deduplication during compression

Facts with identical content but different timestamps were not
being deduplicated, causing the memory database to grow unbounded.
```

```
deps: update tokio to 1.36
```

### 规则

- 使用**祈使语气**（"add", "fix", "update" —— 不是 "added", "fixes", "updates"）
- 保持第一行在 **72 个字符**以内
- 主题行不要以句号结尾
- 用空行分隔主题和正文
- 在页脚中使用 `Closes #N` 或 `Refs #N` 引用 issue 编号

## Issue 标签

| 标签 | 含义 |
|------|------|
| `bug` | 某些功能无法正常工作 |
| `feature` | 新功能请求 |
| `enhancement` | 现有功能的改进 |
| `documentation` | 文档添加或更正 |
| `good first issue` | 适合新贡献者 |
| `help wanted` | 维护者希望社区提供意见 |
| `breaking` | 涉及破坏性 API 更改 |
| `P0` / `P1` / `P2` / `P3` | 优先级（从严重到低） |

## 文档贡献

始终欢迎文档改进。文档站点是一个使用 Next.js 和 Fumadocs 构建的独立项目 ([AlephDocs](https://github.com/rootazero/AlephDocs))。

### 文档位置

| 内容 | 位置 |
|------|------|
| 参考文档（事实来源） | `Aleph/docs/reference/*.md` |
| 网站文档（MDX 页面） | `AlephDocs/content/docs/en/**/*.mdx` |
| 内联 API 文档 | Rust 源文件中的 `///` 注释 |

### 写作风格

- 使用**现在时**（"The server starts" 不是 "The server will start"）
- 使用**主动语态**（"The agent executes tools" 不是 "Tools are executed by the agent"）
- 优先使用**短段落**（最多 3-4 句话）
- 为任何非显而易见的 API 包含**代码示例**
- 用语言标注代码块（`rust`, `bash`, `toml`）

## 社区指南

- **保持尊重。** 技术分歧是可以的；人身攻击则不行。
- **保持耐心。** 维护者在业余时间审查 PR。
- **保持具体。** Bug 报告应包含重现步骤、预期行为和实际行为。
- **先搜索。** 在打开重复项之前检查现有的 issue 和 PR。
- **从小处开始。** 如果你是项目新手，选择一个 `good first issue` 来熟悉代码库和工作流。

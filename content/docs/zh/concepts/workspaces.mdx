Aleph 中的工作区是包含代理操作所需一切的磁盘状态目录：配置、会话历史、内存数据库、插件、技能和日志。工作区模型确保不同代理之间的干净分离，并为所有 Aleph 组件提供可预测的目录结构来读写状态。

## 目录结构

默认情况下，Aleph 将其工作区存储在 `~/.config/aleph/`（`~/.aleph/` 作为常见的符号链接或替代）。完整结构：

```
~/.config/aleph/
├── config.toml              # 主配置文件
├── providers.toml           # AI 提供商凭据和设置
├── sessions.db              # 会话历史的 SQLite 数据库
├── memory/
│   ├── facts.lance/         # 用于事实的 LanceDB 向量存储
│   └── graph.db             # 知识图谱 (节点 + 边)
├── skills/                  # 用户和演化技能
│   ├── code-review/
│   │   └── SKILL.md
│   └── git-commit/
│       └── SKILL.md
├── plugins/                 # 已安装的扩展
│   ├── my-plugin/
│   │   ├── aleph_plugin.toml
│   │   └── dist/
│   └── another-plugin/
├── protocols/               # 自定义协议定义 (YAML)
│   └── my-custom-api.yaml
├── soul.md                  # 全局身份/个性定义
├── keys/                    # API 密钥和秘密库
├── logs/                    # 网关和运行时日志
│   ├── gateway.log
│   └── agent.log
└── evolution.db             # 技能演化跟踪数据库
```

## 配置文件

### config.toml

主配置文件控制所有运行时行为。它按部分组织：

```toml
[agent]
max_iterations = 25
token_budget = 128000
thinking_level = "medium"
enable_compression = true
compression_threshold = 100000

[gateway]
host = "127.0.0.1"
port = 18790
require_auth = false

[interfaces.telegram]
token = "BOT_TOKEN"
allow_from = ["+1234567890"]

[interfaces.discord]
token = "BOT_TOKEN"
guilds = ["guild-id-1"]

[skills]
enabled = true
skills_dir = "skills"
auto_match_enabled = false

[extensions]
enabled = true
hot_reload = true

[memory]
auto_extract = true
max_facts = 10000
```

### providers.toml

提供商凭据单独存储以确保安全。此文件永远不会提交到版本控制：

```toml
[providers.claude]
protocol = "anthropic"
model = "claude-opus-4-20250514"
api_key = "sk-ant-..."
enabled = true

[providers.openai]
protocol = "openai"
model = "gpt-4o"
api_key = "sk-..."
enabled = true

[providers.ollama]
protocol = "ollama"
model = "llama3.2"
base_url = "http://localhost:11434"
enabled = true
```

## 会话存储

所有对话会话都持久化在 `sessions.db` SQLite 数据库中：

```sql
CREATE TABLE sessions (
    session_key TEXT PRIMARY KEY,
    messages TEXT,           -- Message 对象的 JSON 数组
    created_at INTEGER,
    updated_at INTEGER,
    message_count INTEGER,
    token_count INTEGER
);

CREATE TABLE session_metadata (
    session_key TEXT PRIMARY KEY,
    agent_id TEXT,
    channel TEXT,
    last_compaction INTEGER
);
```

会话密钥使用分层格式，对代理、通道和范围进行编码：

| 密钥格式 | 示例 | 说明 |
|------------|---------|-------------|
| `agent:main:main` | 共享会话 | 跨通道默认会话 |
| `agent:main:telegram:dm:user123` | 每用户 DM | 隔离的 Telegram 对话 |
| `agent:main:discord:group:guild-id` | 群组聊天 | Discord 公会对话 |
| `agent:main:cron:daily-summary` | Cron 任务 | 计划任务会话 |

## 内存存储

内存子系统在工作区内使用两个存储后端：

### LanceDB 向量存储

事实与向量嵌入一起存储在 `memory/facts.lance/` 目录中以进行语义搜索。这使代理能够使用相似性搜索从过去的对话中回忆相关信息。

### 知识图谱

`memory/graph.db` SQLite 数据库将实体之间的结构化关系存储为节点和边的图。这通过启用基于遍历的查询（如"关于项目 X 及其依赖项我知道什么？"）来补充向量存储。

## 插件目录

扩展存储在 `~/.aleph/plugins/` 中，每个插件位于自己的子目录中：

```
plugins/
├── my-plugin/
│   ├── aleph_plugin.toml    # 插件清单
│   ├── package.json         # (Node.js 插件)
│   ├── dist/
│   │   └── index.js         # 编译后的入口点
│   └── src/
│       └── index.ts         # 源代码
└── wasm-tool/
    ├── aleph_plugin.toml
    └── plugin.wasm          # 编译后的 WASM 模块
```

扩展系统按顺序扫描三个搜索路径：

1. `~/.aleph/plugins/` -- 用户安装的插件（最高优先级）
2. `/usr/local/share/aleph/plugins/` -- 系统范围的插件
3. `./plugins/` -- 项目本地插件（相对于工作目录）

当启用热重载时，文件监视器监视这些目录，并在文件更改时自动加载、重新加载或卸载插件。

## 身份和 Soul 文件

Aleph 使用分层身份系统，可以在多个级别指定 soul 定义：

```
优先级 (从高到低):
┌─────────────┐
│  会话       │  ← 通过 RPC 运行时覆盖
├─────────────┤
│  项目       │  ← 项目目录中的 .soul/identity.md
├─────────────┤
│  全局       │  ← ~/.aleph/soul.md
├─────────────┤
│  默认       │  ← 内置空清单
└─────────────┘
```

工作区中的全局 `soul.md` 文件定义代理的默认个性：

```markdown
---
relationship: mentor
expertise:
  - Rust
  - System design
---

# Identity

I am Aleph, your AI programming partner.

## Directives

- Be helpful and encouraging
- Explain concepts clearly
- Suggest best practices

## Anti-Patterns

- Never be condescending
- Never make up information
```

## 日志文件

运行时日志存储在 `logs/` 目录中：

- `gateway.log` -- WebSocket 服务器事件、RPC 调用、连接生命周期。
- `agent.log` -- 代理循环事件、工具执行、LLM 调用。

Aleph 使用带有结构化日志的 `tracing` crate。可以通过 `RUST_LOG` 环境变量配置日志级别：

```bash
RUST_LOG=alephcore=debug cargo run -p alephcore --features gateway --bin aleph-gateway -- start
```

## 多代理工作区分离

当运行多个代理时（例如，主代理和专门的子代理），每个代理可以在共享相同工作区根的同时维护自己的状态：

```
~/.config/aleph/
├── config.toml                    # 共享配置
├── sessions.db                    # 包含所有代理的会话
│   ├── agent:main:main            # 主代理会话
│   ├── agent:main:telegram:dm:*   # 主代理 Telegram 会话
│   └── subagent:agent:main:translator  # 子代理会话
└── evolution.db                   # 共享技能演化跟踪
```

子代理会话使用命名空间会话密钥 (`subagent:agent:main:translator`) 以在共享数据库中保持隔离。`SessionCoordinator` 管理子代理会话生命周期：

```rust
pub struct SessionCoordinator {
    db: Arc<VectorDatabase>,
    config: CoordinatorConfig,
}

impl SessionCoordinator {
    pub async fn create_session(&self, agent_type: &str, parent_id: &str)
        -> Result<SessionHandle, AlephError>;
    pub async fn acquire_session(&self, agent_type: &str, parent_id: &str)
        -> Result<SessionHandle, AlephError>;
    pub async fn release_session(&self, session_id: &str)
        -> Result<(), AlephError>;
}
```

空闲的子代理会话可以被 `SwapManager` **交换到磁盘**以释放内存：

```rust
pub struct SwapManager {
    db: Arc<VectorDatabase>,
    swap_dir: PathBuf,
    config: SwapConfig,
}

impl SwapManager {
    pub async fn swap_out(&self, session_id: &str, context: &SwappedContext)
        -> Result<SwapResult, AlephError>;
    pub async fn swap_in(&self, session_id: &str)
        -> Result<SwappedContext, AlephError>;
}
```

## 热重载

配置更改通过文件监视自动检测：

```
~/.aleph/config.toml 已修改
    │
    ▼
去抖动 (500ms)
    │
    ▼
解析新配置 → 针对模式验证
    │
    ▼
应用更改：
  • 重启受影响的接口
  • 更新路由规则
  • 发出 config.changed 事件
```

大多数配置更改不需要重启。提供商凭据、接口令牌、技能目录和扩展设置都支持热重载。

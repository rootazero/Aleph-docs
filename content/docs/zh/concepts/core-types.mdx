`core` 模块提供在 Aleph 所有层之间共享的纯数据类型。它没有业务逻辑，没有平台特定代码，只有一个外部依赖：`serde`。

## 设计理念

`core` 模块遵循**高内聚零耦合**的原则。此模块中的每个类型都是具有 `Debug`、`Clone`、`Serialize` 和 `Deserialize` 派生实现的纯数据容器。除了简单的字段访问之外没有方法 —— 业务逻辑存在于消费这些类型的模块中。

这种设计确保：
- 类型可以在服务器、桌面桥接和客户端接口之间共享，而不会引入繁重的依赖
- 序列化合约通过 `serde` 保持稳定和版本化
- 模块编译快速，永远不会阻塞其他模块

## MediaAttachment

多模态内容支持，用于来自剪贴板或文件上传的图像、文档和视频。

```rust
pub struct MediaAttachment {
    pub media_type: String,       // "image", "document", "video", "file"
    pub mime_type: String,        // "image/png", "application/pdf"
    pub data: String,             // 内容 (base64 或 utf8)
    pub encoding: String,         // "base64" | "utf8"
    pub filename: Option<String>, // 原始文件名
    pub size_bytes: u64,          // 原始大小
}
```

`encoding` 字段决定如何解释 `data`：
- `"base64"` — 二进制内容编码为 Base64（图像、PDF）
- `"utf8"` — 纯文本内容（markdown、提取的文本）

**用法：** 由 `CapturedContext` 用于将剪贴板内容从 Desktop Bridge 传递到 Thinker，以及由生成或接收文件的工具输出使用。

## CapturedContext

来自用户活动应用程序的上下文，从原生桌面层桥接。

```rust
pub struct CapturedContext {
    pub window_title: Option<String>,
    pub attachments: Option<Vec<MediaAttachment>>,
    pub session_id: Option<String>,
}
```

**字段：**
- `window_title` — 活动窗口的标题（例如，"Notes.app", "VS Code"）
- `attachments` — 交互时刻捕获的剪贴板内容（图像、文件）
- `session_id` — 多轮对话的会话标识符

**用法：** 当用户通过 Desktop Bridge 交互时注入到提示词上下文中，使 AI 了解用户当前正在处理的内容。

## CompressionStats

关于内存压缩状态的统计信息，供设置 UI 使用以显示内存健康状态。

```rust
pub struct CompressionStats {
    pub total_raw_memories: u64,
    pub total_facts: u64,
    pub valid_facts: u64,
    pub facts_by_type: HashMap<String, u64>,
}
```

**字段：**
- `total_raw_memories` — 第 1 层：压缩前存储的原始用户消息
- `total_facts` — 第 2 层：从记忆中提取的压缩事实
- `valid_facts` — 仍然被视为活动的未失效事实
- `facts_by_type` — 按类型细分（偏好、计划、学习等）

**用法：** 由 Gateway 的配置处理程序查询，以在 Panel UI 中填充内存仪表板。

## MemoryEntry

由 Memory 系统的 API 返回的记忆记录，适合序列化到客户端。

```rust
pub struct MemoryEntry {
    pub id: String,
    pub window_title: String,
    pub user_input: String,
    pub ai_output: String,
    pub timestamp: i64,
    pub similarity_score: Option<f32>,
}
```

**用法：** 由内存搜索查询返回。当通过向量搜索检索条目时，`similarity_score` 字段被填充，指示与查询的相关性。

## 这些类型的使用位置

| 类型 | 使用者 | 目的 |
|---|---|---|
| `MediaAttachment` | Desktop Bridge, Thinker, Tools | 多模态内容传递 |
| `CapturedContext` | Desktop Bridge, Orchestrator | 上下文注入 |
| `CompressionStats` | Memory, Gateway, Panel UI | 内存健康显示 |
| `MemoryEntry` | Memory, Gateway | API 响应序列化 |

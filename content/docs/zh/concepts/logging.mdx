`logging` 模块提供 Aleph 的日志基础设施。它有意保持轻量 —— 大部分繁重工作委托给专用的 `aleph-logging` crate，而核心模块专注于 PII（个人可识别信息）清理和上下文感知格式化。

## 设计理念

Aleph 中的日志遵循两个原则：

1. **默认 PII 安全** — 日志绝不能包含 API 密钥、令牌、个人消息或识别信息，除非显式配置
2. **结构化输出** — 所有日志都是结构化的（JSON）以供机器解析，而不仅仅是供人阅读

核心 `logging` 模块是一个薄包装器，使用 Aleph 特定的设置配置 `aleph-logging` crate。

## PII 清理

### 清理内容

| 模式 | 示例输入 | 清理后的输出 |
|---|---|---|
| 电子邮件地址 | `user@example.com` | `[REDACTED-EMAIL]` |
| API 密钥 | `sk-abc123...` | `[REDACTED-API-KEY]` |
| 电话号码 | `+1-555-123-4567` | `[REDACTED-PHONE]` |
| IP 地址 | `192.168.1.1` | `[REDACTED-IP]` |
| 会话 ID | `sess-abc123` | `[REDACTED-SESSION]` |

### 清理行为

PII 清理在格式化层而非事件层操作：

```rust
// 此日志调用未更改
tracing::info!("User {} logged in", email);

// PiiScrubbingFormat 拦截输出：
// BEFORE: "User user@example.com logged in"
// AFTER:  "User [REDACTED-EMAIL] logged in"
```

这种设计意味着：
- 开发人员不需要记住清理每个日志调用
- 清理可以为调试禁用（显式选择加入）
- 性能影响最小（仅在格式化输出上进行正则匹配）

### 配置

```toml
[logging]
level = "info"                    # trace, debug, info, warn, error
format = "json"                   # json 或 pretty
pii_scrubbing = true              # 启用 PII 修订
scrub_patterns = ["custom-regex"] # 附加模式
output = "~/.aleph/logs/"         # 日志目录
max_files = 7                     # 轮换: 保留 7 天
```

## 结构化日志

所有日志都作为 JSON 输出，具有标准字段：

```json
{
  "timestamp": "2026-04-25T14:32:01Z",
  "level": "INFO",
  "target": "aleph::gateway",
  "message": "Session created",
  "fields": {
    "session_id": "[REDACTED-SESSION]",
    "channel": "telegram"
  },
  "span": {
    "name": "handle_message",
    "trace_id": "abc123"
  }
}
```

**优点：**
- 易于日志聚合系统（Loki、Elasticsearch、Datadog）解析
- 通过 `trace_id` 跨度进行关联
- 基于字段的过滤，无需正则

## 上下文感知日志

日志系统自动捕获请求上下文：

```rust
use tracing::{info, instrument};

#[instrument(skip(self), fields(session_id = %session.id))]
async fn handle_message(&self,
    session: &Session,
    message: Message,
) -> Result<(), Error> {
    info!("Processing message"); // 自动包含 session_id

    // 嵌套跨度创建分层跟踪树
    process_message(message).await?;

    Ok(())
}
```

`#[instrument]` 宏自动：
- 为函数创建跟踪跨度
- 将函数参数捕获为结构化字段
- 测量执行时间
- 将子跨度链接到父跨度

## 日志级别

| 级别 | 用于 | 示例 |
|---|---|---|
| `TRACE` | 详细执行流程 | "进入函数 X", "循环迭代 5" |
| `DEBUG` | 开发诊断 | "缓存命中", "SQL 查询: ..." |
| `INFO` | 正常操作 | "会话已创建", "工具已执行" |
| `WARN` | 可恢复的问题 | "配置重载失败，使用缓存" |
| `ERROR` | 需要关注的故障 | "数据库连接丢失" |

**生产默认值：** `INFO` — 由于数量原因，生产部署不应发出 `DEBUG` 或 `TRACE` 日志。

## 实现说明

`src/logging/` 中的 `PiiScrubbingLayer` 结构体是公共 API 表面和文档占位符。实际清理由 `PiiScrubbingFormat` 执行：

```rust
pub struct PiiScrubbingLayer;

// Layer::on_event 是无操作 — 清理发生在 Format 层
impl<S> Layer<S> for PiiScrubbingLayer {
    fn on_event(&self, _event: &Event<'_>, _ctx: Context<'_>, S) {
        // 无操作: 实际清理在 PiiScrubbingFormat 中
    }
}
```

这种拆分设计允许：
- Layer 独立注册/注销
- Format 可以为测试交换（无操作清理器）
- 未来扩展（例如，异步清理、基于 ML 的检测）

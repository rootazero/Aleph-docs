`scheduler` 和 `tasks` 模块处理 Aleph 中的任务执行调度。调度器管理具有资源约束的子代理执行，而 tasks 模块提供基于时间 (cron) 和健康检查 (heartbeat) 的作业调度。

## 设计理念

调度系统遵循三个原则：

1. **资源隔离** — 每个执行"车道"具有独立的配额和并发限制
2. **反饥饿** — 长时间等待的作业获得优先级提升以防止无限阻塞
3. **递归安全** — 跟踪子代理生成并限制以防止无限递归

## 基于车道的调度器

`LaneScheduler` 为子代理执行提供结构化并发：

```
┌─────────────────────────────────────────────┐
│           LaneScheduler                      │
├─────────────────────────────────────────────┤
│                                             │
│  车道 A (前台)  max=3, quota=100            │
│    ├─ 运行 1 (运行中)                       │
│    ├─ 运行 2 (排队中)                       │
│    └─ 运行 3 (排队中)                       │
│                                             │
│  车道 B (后台)  max=1, quota=50             │
│    └─ 运行 4 (运行中)                       │
│                                             │
│  全局限制: 5 个并发运行                     │
└─────────────────────────────────────────────┘
```

### Lane

具有自己配额和并发限制的隔离执行车道：

```rust
pub struct Lane {
    name: String,
    max_concurrent: usize,
    quota: LaneQuota,
}
```

### LaneScheduler

```rust
pub struct LaneScheduler {
    lanes: Vec<Lane>,
    global_max: usize,
    recursion_tracker: RecursionTracker,
}

impl LaneScheduler {
    pub async fn schedule(
        &self,
        run: QueuedRun,
    ) -> Result<ScheduleGuard> { /* ... */ }

    pub fn stats(&self,
    ) -> SchedulerStats { /* ... */ }
}
```

**ScheduleGuard** 使用 RAII — 当丢弃时，运行的许可自动释放。调度器使用 `std::mem::forget(permit)` + `ScheduleGuard` 进行手动控制，这是正确的但需要小心：`forget` 和 guard 构造之间的提前返回会泄漏许可。

### 反饥饿

```rust
pub struct WaitTimeTracker;

impl WaitTimeTracker {
    pub fn sweep_anti_starvation(
        &self,
        lanes: &mut [LaneState],
    ) { /* ... */ }
}
```

等待时间超过阈值的作业获得优先级提升。注意：提升按车道应用，因此提升车道中的新作业也受益。

### 递归跟踪

```rust
pub struct RecursionTracker {
    max_depth: usize,
}
```

跟踪子代理生成深度。如果子代理尝试生成另一个超出 `max_depth` 的子代理，请求会被阻止。

### LaneState

```rust
pub struct LaneState {
    queued: VecDeque<QueuedRun>,
    running: HashMap<RunId, RunningRun>,
}
```

**整数安全：** 等待时间计算使用 `.max(0)` 防止时钟偏差下溢，`.min(10)` 在 `as i8` 之前防止溢出。

## Cron 任务

`tasks::cron` 模块调度周期性作业：

```rust
pub struct CronService {
    db: Connection,
    clock: Arc<dyn Clock>,
}

impl CronService {
    pub async fn create_job(
        &self,
        config: JobConfig,
    ) -> Result<JobId> { /* ... */ }

    pub async fn list_jobs(&self,
    ) -> Result<Vec<Job>> { /* ... */ }

    pub async fn delete_job(
        &self,
        id: JobId,
    ) -> Result<()> { /* ... */ }
}
```

**特性：**
- Cron 表达式解析 (标准 Unix cron 语法)
- 作业执行历史
- 超时处理
- 三阶段并发: schedule → execute → cleanup (最小化锁持有时间)

**时钟抽象：** 使用带有 `FakeClock` 的 `Clock` trait 以提高可测试性。

### JobRun

```rust
pub struct JobRun {
    started_at: i64,
    completed_at: Option<i64>,
}

impl JobRun {
    pub fn success(self) -> Self {
        let now = self.clock.now();
        let duration_ms = now
            .saturating_sub(self.started_at)
            .saturating_mul(1000);
        // ...
    }
}
```

**整数安全：** 使用 `saturating_sub` 和 `saturating_mul` 防止时钟偏差下溢。

## 心跳任务

`tasks::heartbeat` 模块监控外部服务：

```rust
pub struct HeartbeatService {
    intervals: Vec<HeartbeatInterval>,
}

impl HeartbeatService {
    pub async fn check_all(
        &self,
    ) -> Vec<HeartbeatResult> { /* ... */ }
}
```

**特性：**
- URL 健康检查
- 响应时间跟踪
- 去重 (避免冗余检查)
- 可配置间隔和超时

## 任务基础设施

`tasks::shared` 模块提供横切基础设施：

- `Clock` trait — 抽象时间以提高可测试性
- `Schedule` — 计划解析和下次运行计算
- `Delivery` — 任务结果交付机制
- `Store` — 任务状态的持久化层

---

## 安全属性

- **无下溢** — 所有时间计算使用 `saturating_sub`/`saturating_mul`
- **信号量安全** — `was_running` 检查防止许可膨胀
- **无锁问题** — 使用 `unwrap_or_else(|e| e.into_inner())`
- **无 SQL 注入** — 所有查询使用参数化 `params![]`
- **UTF-8 安全** — `truncate_string()` 使用 `char_indices()`
- **无 `static mut`** — 使用 `crate::sync_primitives::Arc`

本指南涵盖了在生产环境中运行 Aleph 的最常见方式——从简单的前台进程到完全托管的系统服务，再到带有 TLS 的反向代理。

## 本地开发

运行 Aleph 最简单的方式是直接从命令行：

```bash
# 前台运行（Ctrl+C 停止）
cargo run --bin aleph-server

# 或使用 just
just dev
```

网关默认监听 `ws://127.0.0.1:18790`。

## macOS — launchd 服务

在 macOS 上，创建一个 launchd 用户代理来自动启动。

### 创建 plist

创建 `~/Library/LaunchAgents/com.aleph.daemon.plist`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.aleph.daemon</string>
    <key>RunAtLoad</key>
    <true/>
    <key>KeepAlive</key>
    <true/>
    <key>ProgramArguments</key>
    <array>
        <string>/usr/local/bin/aleph-server</string>
    </array>
    <key>EnvironmentVariables</key>
    <dict>
        <key>RUST_LOG</key>
        <string>info</string>
    </dict>
    <key>StandardOutPath</key>
    <string>/Users/YOUR_USERNAME/.aleph/logs/aleph.log</string>
    <key>StandardErrorPath</key>
    <string>/Users/YOUR_USERNAME/.aleph/logs/aleph-error.log</string>
</dict>
</plist>
```

### 管理服务

```bash
# 加载服务
launchctl load ~/Library/LaunchAgents/com.aleph.daemon.plist

# 启动
launchctl start com.aleph.daemon

# 检查状态
launchctl list | grep com.aleph

# 停止
launchctl stop com.aleph.daemon

# 卸载
launchctl unload ~/Library/LaunchAgents/com.aleph.daemon.plist
```

## Linux — systemd 服务

在 Linux 上，为 Aleph 创建一个 systemd 用户服务。

### 创建 Unit 文件

```bash
mkdir -p ~/.config/systemd/user
```

```ini title="~/.config/systemd/user/aleph.service"
[Unit]
Description=Aleph AI Assistant Gateway
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
ExecStart=%h/.cargo/bin/aleph-server
Restart=on-failure
RestartSec=5
Environment=ALEPH_HOME=%h/.aleph
Environment=RUST_LOG=info

# 资源限制
MemoryMax=1G
MemoryHigh=512M
CPUQuota=80%

# 安全加固
NoNewPrivileges=true
ProtectHome=read-only
ReadWritePaths=%h/.aleph

StandardOutput=journal
StandardError=journal

[Install]
WantedBy=default.target
```

### 启用并启动

```bash
# 重新加载 unit 文件
systemctl --user daemon-reload

# 开机自启（需要 lingering）
systemctl --user enable aleph

# 立即启动
systemctl --user start aleph

# 检查状态
systemctl --user status aleph
```

### 启用用户 lingering

默认情况下，systemd 用户服务在你注销时停止。启用 lingering 以便即使没有活跃会话，Aleph 也能运行：

```bash
sudo loginctl enable-linger $USER
```

### 查看日志

```bash
journalctl --user -u aleph -f
```

## Docker 部署

### Dockerfile

创建一个多阶段 Dockerfile 以构建最小化的生产镜像：

```dockerfile title="Dockerfile"
# 构建阶段
FROM rust:1.92-bookworm AS builder

WORKDIR /build
COPY . .

# 安装 wasm-bindgen 和 npm 依赖以构建 WASM
RUN cargo install wasm-bindgen-cli
RUN apt-get update && apt-get install -y npm

# 构建服务器（只需构建即可处理 WASM + Swift + Rust）
RUN cargo build -p alephcore --bin aleph-server --release

# 运行阶段
FROM debian:bookworm-slim

RUN apt-get update && apt-get install -y \\
    ca-certificates \\
    libsqlite3-0 \\
    && rm -rf /var/lib/apt/lists/*

COPY --from=builder /build/target/release/aleph-server /usr/local/bin/aleph-server

# 创建 aleph 用户
RUN useradd -m -s /bin/bash aleph
USER aleph

RUN mkdir -p /home/aleph/.aleph/logs

EXPOSE 18790

ENTRYPOINT ["aleph-server"]
```

### Docker Compose

用于带有持久存储的完整设置：

```yaml title="docker-compose.yml"
services:
  aleph:
    build: .
    container_name: aleph
    restart: unless-stopped
    ports:
      - "127.0.0.1:18790:18790"
    volumes:
      - aleph-data:/home/aleph/.aleph
    environment:
      - ANTHROPIC_API_KEY=${ANTHROPIC_API_KEY}
      - RUST_LOG=info

volumes:
  aleph-data:
```

启动堆栈：

```bash
# 设置你的 API 密钥
echo "ANTHROPIC_API_KEY=sk-ant-..." > .env

# 构建并启动
docker compose up -d

# 检查日志
docker compose logs -f aleph
```

## 环境变量

生产环境中识别的环境变量：

| 变量 | 描述 | 默认值 |
|----------|-------------|---------|
| `ANTHROPIC_API_KEY` | Anthropic API 密钥 | — |
| `OPENAI_API_KEY` | OpenAI API 密钥 | — |
| `GOOGLE_AI_API_KEY` | Google AI API 密钥 | — |
| `OPENROUTER_API_KEY` | OpenRouter API 密钥 | — |
| `ALEPH_HOME` | 覆盖配置目录 | `~/.aleph` |
| `ALEPH_MASTER_KEY` | 加密密钥用于密钥保险库 | — |
| `RUST_LOG` | 日志级别（`trace`, `debug`, `info`, `warn`, `error`） | `info` |

API 密钥也可以通过配置文件设置。环境变量优先级更高。

## 反向代理

如果你将 Aleph 暴露在网络中，请将其放在反向代理后面以进行 TLS 终止和访问控制。网关使用 WebSocket，因此请确保你的代理支持 WebSocket 升级。

### Nginx

```nginx title="/etc/nginx/sites-available/aleph"
upstream aleph_gateway {
    server 127.0.0.1:18790;
}

server {
    listen 443 ssl http2;
    server_name aleph.example.com;

    ssl_certificate     /etc/letsencrypt/live/aleph.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/aleph.example.com/privkey.pem;

    location / {
        proxy_pass http://aleph_gateway;
        proxy_http_version 1.1;

        # WebSocket 支持
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 长连接 WebSocket 的超时时间
        proxy_read_timeout 86400s;
        proxy_send_timeout 86400s;
    }
}
```

启用并重新加载：

```bash
sudo ln -s /etc/nginx/sites-available/aleph /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
```

### Caddy

Caddy 通过 Let's Encrypt 自动处理 TLS：

```txt title="Caddyfile"
aleph.example.com {
    reverse_proxy 127.0.0.1:18790
}
```

Caddy 默认支持 WebSocket 升级——无需额外配置。

```bash
sudo caddy reload --config /etc/caddy/Caddyfile
```

## HTTPS 和 TLS

Aleph 也支持不通过反向代理的原生 TLS。在 `~/.aleph/config.toml` 中配置：

```toml
[gateway]
port = 18790
bind = "0.0.0.0"

[gateway.tls]
cert = "/etc/letsencrypt/live/aleph.example.com/fullchain.pem"
key = "/etc/letsencrypt/live/aleph.example.com/privkey.pem"
```

配置 TLS 后，网关会监听 `wss://` 而不是 `ws://`。

对于大多数部署，使用反向代理进行 TLS 更简单——它处理证书续期，并允许你在同一主机上提供其他服务。

## 监控和健康检查

### 健康端点

网关在 HTTP 根路径暴露健康检查：

```bash
curl http://localhost:18790/health
# 预期：HTTP 200 并带有状态信息
```

在 Docker Compose、systemd 或监控工具中使用此端点。

### 日志轮转

在 systemd 中，journald 自动处理轮转。对于其他部署，配置 logrotate：

```txt title="/etc/logrotate.d/aleph"
/home/aleph/.aleph/logs/*.log {
    daily
    rotate 14
    compress
    delaycompress
    missingok
    notifempty
}
```

## 备份策略

需要备份的关键数据位于 `~/.aleph/`：

| 路径 | 内容 | 优先级 |
|------|----------|----------|
| `config.toml` | 所有配置 | 高 |
| `sessions/` | 会话数据库 | 中 |
| `plugins/` | 已安装插件 | 低（可以重新安装） |
| `logs/` | 应用日志 | 低 |

一个简单的 cron 任务：

```bash
# 每天备份配置和会话
0 3 * * * tar -czf ~/backups/aleph-$(date +\%Y\%m\%d).tar.gz \\
    ~/.aleph/config.toml \\
    ~/.aleph/sessions/
```

对于 Docker 卷，使用 `docker cp` 或将卷挂载到备份容器。

## 安全检查清单

在将 Aleph 暴露在网络之前：

- [ ] **绑定地址** — 除非需要外部访问，否则将 `gateway.bind` 设置为 `"127.0.0.1"`
- [ ] **认证** — 启用 `gateway.require_auth = true`
- [ ] **TLS** — 始终在生产环境中使用 HTTPS/WSS
- [ ] **防火墙** — 只开放 443 端口（通过反向代理），永远不要直接暴露 18790
- [ ] **主密钥** — 设置 `ALEPH_MASTER_KEY` 以加密密钥保险库
- [ ] **白名单** — 配置 `channel.telegram.allow_from` 以限制访问

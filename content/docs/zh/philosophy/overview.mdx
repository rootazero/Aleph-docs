Aleph 不仅仅是另一个 AI 助手。它是对个人 AI 应该是什么的深思熟虑、第一性原理思考的产物：自托管、私密、自适应和多态。本页解释了驱动项目中每个设计决策的哲学基础。

Aleph 中的每个技术选择都追溯到关于智能、自主性和人类与 AI 系统之间关系的一小组核心信念。

## 为什么 Aleph 存在

当前 AI 助手领域由云托管、供应商锁定的服务主导。它们很强大，但伴随着基本的权衡：

- **隐私**：您的对话、记忆和行为模式存在于别人的服务器上。
- **控制**：您无法修改系统的行为、扩展其能力或决定它如何处理您的数据。
- **依赖**：如果服务关闭、更改定价或更改其政策，您会失去一切。
- **统一性**：每个用户获得具有相同个性、相同约束和相同限制的相同助手。

Aleph 存在是因为这些权衡不是不可避免的。个人 AI 助手可以真正个人化 —— 在您自己的硬件上运行，从您的特定上下文学习，并随着时间的推移与您的需求一起进化。

> *"Aleph 是空间中包含所有其他点的点之一。"*
> -- 豪尔赫·路易斯·博尔赫斯，《El Aleph》

就像博尔赫斯的 Aleph —— 一个包含整个宇宙的晶体点 —— Aleph 项目旨在成为一个包含您所需所有能力的单一系统，统一在一个连贯的架构中。

## 第一性原理

在编写单行代码之前，Aleph 锚定在一个基本问题上：**个人 AI 助手的成功是什么样子的？**

这是驱动 Aleph 代理行为的相同"第一性原理思考"。Aleph 不是复制现有助手的做法并希望最好，而是定义显式的成功标准：

1. **它必须是自托管的。** 您的数据、您的硬件、您的规则。核心功能不依赖云。
2. **它必须是自适应的。** 系统应该从经验中学习并真正随着时间的推移而改进 —— 不仅仅是积累聊天历史。
3. **它必须是多态的。** 一个智能，多种表现形式。相同的核心应该在 CLI、桌面、移动设备、消息平台和语音接口上工作。
4. **它必须是可扩展的。** 用户和开发人员应该能够在不修改核心系统的情况下添加新功能。
5. **它默认必须是安全的。** 每个工具执行、每个外部调用、每个插件操作都应该通过显式审批工作流。

这些不是理想目标 —— 它们是塑造每个架构决策的硬约束。

## 什么使 Aleph 不同

### Rust 原生核心

Aleph 是从头开始用 Rust 构建的。这不是一个包装在 Web 框架中的 Python 脚本 —— 它是一个为可靠性、性能和安全性设计的系统级应用程序。

Rust 提供：

- **没有垃圾回收的内存安全** —— 对于管理状态、连接和并发工具执行的长时间运行服务器至关重要。
- **无畏并发** —— Gateway 处理多个同时接口连接、代理循环和工具执行而没有数据竞争。
- **零成本抽象** —— 基于 trait 的架构（实体、聚合根、值对象）不增加运行时开销。
- **单二进制部署** —— 整个服务器编译为一个二进制文件。没有依赖地狱，没有运行时版本冲突。

### 多态架构

大多数 AI 助手是单体的：一个接口，一个交互模型。Aleph 采用根本不同的方法，以其服务器为中心的架构：

```
                    ┌──────────────────────────────┐
                    │        ALEPH SERVER           │
                    │   (大脑 + 手 + 记忆)          │
                    └──────────────┬───────────────┘
                                   │
                    ┌──────────────┴───────────────┐
                    │     GATEWAY (WebSocket)       │
                    └──┬────┬────┬────┬────┬───────┘
                       │    │    │    │    │
                    CLI  Desktop  macOS  Telegram  Discord
```

服务器是大脑和手 —— 它做所有思考、记忆和行动。接口是处理输入和显示的轻量级 I/O 层。这意味着：

- 所有接口共享相同的记忆、相同的上下文、相同的个性。
- 您可以在 Telegram 上开始对话并在桌面应用上继续。
- 添加新接口只需要实现一个薄的 WebSocket 客户端 —— 不需要 AI 逻辑。

### 涌现驱动设计

Aleph 围绕[五层涌现](./five-layers)模型组织，该模型描述智能如何从原始知识构建到自主行为。Aleph 不是试图硬编码每个能力，而是创造条件让智能行为从更简单的组件中涌现。

这反映在具体机制中：

- **经验固化**：成功的任务完成被记录，重复模式自动提升为可重用技能。
- **技能进化**：最初作为一次性解决方案的东西可以成为原子技能，然后成为可组合模块，然后成为多态代理的一部分。
- **自适应行为**：系统真正基于以前有效的方法改变其方法 —— 它不仅仅检索聊天历史。

### 自托管隐私

Aleph 完全在您自己的硬件上运行。服务器、记忆数据库、向量存储、配置 —— 一切都存在于您的机器上。唯一的外部调用是到 LLM 提供商（您自己配置的）和本质上需要网络访问的工具。

这不是哲学立场 —— 它是架构保证。没有遥测、没有云同步、没有外部分析。您的数据留在您放置它的地方。

## 设计目标

### 自主性

代理应该能够在没有持续人工干预的情况下完成复杂的多步骤任务。[POE 架构](./agent-thinking#poe-architecture)（原则-操作-评估）使目标导向行为成为可能，代理可以在其中计划、执行、评估和迭代。

然而，自主性是有界限的。安全系统确保潜在危险的操作（shell 命令、文件修改、网络请求）需要显式审批。自主性并不意味着不受控制。

### 安全性

安全性不是事后考虑 —— 它 baked into 执行管道。每个工具调用都通过检查可配置允许列表和审批规则的守卫系统。默认姿势是拒绝：如果工具执行未被显式允许，它需要人工审批。

有关安全模型的详细信息，请参阅[安全](/docs/zh/security)。

### 可扩展性

Aleph 支持多种扩展机制：

- **工具**：用于常见操作的内置工具（19+），加上自定义工具注册。
- **MCP 协议**：用于标准化工具集成的模型上下文协议支持。
- **插件**：用于沙箱扩展的 WASM 和 Node.js 运行时支持。
- **技能**：将工具和知识组合到可重用工作流中的更高级别能力。

有关扩展 Aleph 的详细信息，请参阅[工具](/docs/zh/tools)。

### 多通道存在

Gateway 协议（通过 WebSocket 的 30+ JSON-RPC 方法）为所有接口提供统一的控制平面。这意味着 Aleph 可以存在于您需要的任何地方：

- 用于开发工作流的命令行
- 用于专注工作会话的桌面应用
- 具有系统集成的原生 macOS 应用
- 用于消息原生交互的 Telegram 和 Discord
- 语音和 Canvas 接口（计划中）

有关 Gateway 的详细信息，请参阅[网关](/docs/zh/gateway)。

## 名称

**Aleph** 来自三个来源：

- **数学**：阿列夫数（ℵ₀, ℵ₁, ℵ₂...）代表 progressively larger infinities。Aleph 的智能通过离散级别进化，每个级别超越前一个。
- **文学**：在博尔赫斯的故事《El Aleph》中，Aleph 是空间中包含所有其他点的点 —— 一个在一个统一架构中包含所有能力的系统的隐喻。
- **希伯来语**：作为希伯来字母表的第一个字母（א），Aleph 代表起源、统一和使所有语言成为可能的静默呼吸。

这个名字捕捉了项目的雄心：一个包含所有点的单一点，一个从起源开始并攀登向 ever-greater intelligence 的系统。

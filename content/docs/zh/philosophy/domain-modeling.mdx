Aleph 的代码库使用领域驱动设计（DDD）原则组织，通过 Rust 的 trait 系统实现。Aleph 不依赖于重量级的 DDD 框架，而是使用轻量级的 trait 契约在编译时强制执行领域规则。本页描述了建模方法、核心领域原语以及限界上下文如何将系统划分为内聚模块。

Aleph 中的 DDD 不是仪式性的 —— 它服务于实际目的。通过使领域概念在类型系统中显式化，代码库清晰地传达其意图，并通过编译时保证防止整类错误。

## 为什么 Aleph 使用 DDD

个人 AI 助手是一个复杂的领域。它涉及对话、记忆、任务编排、工具执行、安全策略和多通道通信。没有显式建模，这些关注点会模糊在一起，创建一个所有事物都依赖于所有事物的代码库。

DDD 提供了组织支柱：

- **限界上下文**防止跨领域耦合。记忆系统不需要了解调度器的任务图，调度器也不需要了解网关的会话管理。
- **聚合根**强制执行一致性边界。对相关对象的修改通过一个入口点进行，防止不一致状态。
- **值对象**消除身份混淆。当且仅当两个对象的属性匹配时它们才相等，使它们成为值对象可以防止意外假设身份的错误。
- **实体**使身份显式化。当一个对象需要在操作之间持久化并被其他对象引用时，将其标记为实体使此契约在代码中可见。

## 核心原语

Aleph 在其领域模块中定义了三个基础 DDD trait。这些按设计是最小化的 —— 它们强制执行契约而不增加运行时开销。

### 实体

**实体**是一个具有唯一身份的对象，该身份在状态更改期间持久存在。两个实体相等当且仅当它们共享相同的 ID，无论其其他属性如何。

```rust
pub trait Entity {
    type Id: Eq + Clone + std::fmt::Display;
    fn id(&self) -> &Self::Id;
}
```

**特征**：
- 具有通过 `id()` 访问的唯一标识符。
- 身份是稳定的 —— 同一实体在所有修改中保留其 ID。
- 状态可以更改，但身份不能。
- 相等性完全由 ID 比较决定。

**何时使用实体**：当对象需要在操作之间被跟踪、被其他对象引用或具有生命周期（创建、更新、删除）时。

**示例**：

```rust
pub struct Task {
    id: String,
    name: String,
    status: TaskStatus,
    dependencies: Vec<String>,
}

impl Entity for Task {
    type Id = String;
    fn id(&self) -> &Self::Id { &self.id }
}
```

`Task` 是一个实体，因为它具有生命周期 —— 它被创建，其状态更改，依赖项被更新，最终它完成或失败。具有相同数据但不同 ID 的两个任务是不同的任务。具有相同 ID 但不同状态的两个任务是同一任务的不同状态。

### 聚合根

**聚合根**是聚合的入口点 —— 为了保持一致性目的而被视为一个单元的相关对象集群。对聚合内对象的所有修改必须通过聚合根进行。

```rust
pub trait AggregateRoot: Entity {}
```

**特征**：
- 继承所有实体属性（具有身份，在更改中稳定）。
- 作为事务边界 —— 外部代码仅通过根与聚合交互。
- 负责维护聚合内所有对象的一致性。
- 对聚合内对象的外部引用通过根进行。

**何时使用聚合根**：当一个对象管理一组相关对象并需要跨它们强制执行一致性规则时。

**示例**：

```rust
pub struct TaskGraph {
    id: String,
    tasks: Vec<Task>,
    edges: Vec<(String, String)>,
}

impl Entity for TaskGraph {
    type Id = String;
    fn id(&self) -> &Self::Id { &self.id }
}

impl AggregateRoot for TaskGraph {}
```

`TaskGraph` 是一个聚合根，因为它管理 `Task` 实体集合及其依赖边。添加任务、移除任务或修改依赖项必须通过 `TaskGraph` 进行，以确保图保持有效（无循环、无悬空引用、正确排序）。

外部代码永远不应直接修改 `Task` —— 它应该调用维护不变量的 `TaskGraph` 方法：

```rust
impl TaskGraph {
    pub fn add_task(&mut self, task: Task) -> Result<(), GraphError> {
        // 验证无重复 ID
        // 验证依赖项存在
        // 维护拓扑顺序
        self.tasks.push(task);
        Ok(())
    }

    pub fn update_status(&mut self, task_id: &str, status: TaskStatus) -> Result<(), GraphError> {
        // 验证任务存在
        // 验证状态转换是否合法
        // 将状态更改传播到依赖项
        // ...
        Ok(())
    }
}
```

### 值对象

**值对象**是一个完全由其属性定义的不可变对象。它没有身份 —— 两个值对象相等当且仅当它们的所有属性都相等。

```rust
pub trait ValueObject: Eq + Clone {}
```

**特征**：
- 不可变 —— 一旦创建，值对象永远不会更改。
- 无身份 —— 相等性由比较所有属性决定。
- 可自由复制和替换。
- 无副作用 —— 对值对象的操作返回新值对象。

**何时使用值对象**：当对象表示度量、分类、状态或任何身份无关紧要的概念 —— 只有值重要时。

**示例**：

```rust
#[derive(Eq, PartialEq, Clone)]
pub struct TaskStatus {
    pub state: TaskState,
    pub progress: f32,
}

impl ValueObject for TaskStatus {}
```

`TaskStatus` 是一个值对象，因为我们只关心状态*是什么*，而不关心它是哪个特定实例。"运行中 50%" 的状态与任何其他"运行中 50%" 的状态相同，无论它何时或在何处创建。

## 限界上下文

Aleph 的领域被划分为限界上下文 —— 具有自己的通用语言、自己的模型和自己的规则的领域内聚区域。不同上下文中的对象可能共享名称但具有不同的含义。

### 调度器上下文

**职责**：DAG 调度、工具编排、任务状态管理。

调度器上下文处理代理的执行端 —— 将请求分解为任务图、管理依赖项和跟踪进度。

| 类型 | DDD 角色 | 位置 |
|------|----------|----------|
| `TaskGraph` | 聚合根 | `dispatcher/agent_types/graph.rs` |
| `Task` | 实体 | `dispatcher/agent_types/task.rs` |
| `TaskStatus` | 值对象 | `dispatcher/agent_types/status.rs` |

`TaskGraph` 聚合根确保：
- 任务按依赖顺序执行。
- 不存在循环依赖。
- 状态传播是一致的（父任务在子任务仍在"运行"时不能是"完成"）。

### 记忆上下文

**职责**：事实存储、RAG 检索、知识压缩。

记忆上下文管理代理的持久知识 —— 存储事实、检索相关上下文和压缩旧记忆以保持效率。

| 类型 | DDD 角色 | 位置 |
|------|----------|----------|
| `MemoryFact` | 聚合根 | `memory/context.rs` |
| `ContextAnchor` | 值对象 | `memory/context.rs` |
| `FactType` | 值对象 | `memory/context.rs` |

`MemoryFact` 聚合根确保：
- 事实正确锚定到其上下文（对话、主题、领域）。
- 事实类型一致且有效。
- 压缩操作保留基本信息。

### 意图上下文

**职责**：意图检测、L1-L3 分层过滤。

意图上下文处理代理的感知端 —— 分析用户输入并将其分类为可操作的意图。

| 类型 | DDD 角色 | 位置 |
|------|----------|----------|
| `AggregatedIntent` | 值对象 | `intent/types.rs` |
| `IntentSignal` | 值对象 | `intent/types.rs` |

注意，意图上下文仅使用值对象 —— 意图被分类，而非跟踪。意图没有生命周期；它是代理在某一时刻理解的快照。

### POE 上下文

**职责**：成功合约、验证规则、评估结果。

POE 上下文处理代理的评估端 —— 定义成功是什么样子的并验证是否实现。

| 类型 | DDD 角色 | 位置 |
|------|----------|----------|
| `SuccessManifest` | 聚合根 | `poe/types.rs` |
| `ValidationRule` | 值对象 | `poe/types.rs` |
| `Verdict` | 值对象 | `poe/types.rs` |

`SuccessManifest` 聚合根确保：
- 验证规则一致且完整。
- 裁决通过评估所有规则相对于实际输出来产生。
- 清单在评估开始后不能被修改（评估期间不可变）。

## 上下文关系

限界上下文通过定义良好的接口交互，而非直接依赖：

```
                  ┌────────────────┐
                  │  意图上下文     │
                  │  （感知）       │
                  └───────┬────────┘
                          │ 分类意图
                          v
┌────────────────┐   ┌────────────────┐   ┌────────────────┐
│  POE 上下文    │<──│  调度器        │──>│  记忆上下文    │
│  （评估）      │   │  上下文        │   │  （知识）      │
│                │   │  （执行）      │   │                │
└────────────────┘   └────────────────┘   └────────────────┘
        │                                         ^
        │          经验固化                        │
        └─────────────────────────────────────────┘
```

- **意图 --> 调度器**：分类意图被传递给调度器以进行任务图构建。
- **调度器 --> 记忆**：调度器在执行期间查询记忆以获取相关上下文。
- **调度器 --> POE**：完成的任务根据成功清单进行评估。
- **POE --> 记忆**：成功的评估触发经验固化到记忆中。

每个上下文拥有自己的模型，不从其他上下文导入类型。通信通过边界处定义的共享接口（trait、消息类型）进行。

## 实现新的领域类型

当向 Aleph 的领域模型添加新概念时，遵循此决策过程：

### 步骤 1：确定角色

问这些问题：

1. **这个对象需要唯一身份吗？** 它需要被跟踪、引用或具有生命周期吗？
   - 是 --> 它是**实体**。
2. **它管理一组具有一致性要求的相关对象吗？**
   - 是 --> 它是**聚合根**（也是实体）。
3. **它只是一个身份不重要的属性容器吗？**
   - 是 --> 它是**值对象**。

### 步骤 2：实现 Trait

```rust
use crate::domain::{Entity, AggregateRoot, ValueObject};

// 实体
impl Entity for MyEntity {
    type Id = String;
    fn id(&self) -> &Self::Id { &self.id }
}

// 聚合根（也先实现实体）
impl AggregateRoot for MyAggregate {}

// 值对象（派生 Eq 和 Clone）
#[derive(Eq, PartialEq, Clone)]
pub struct MyValue { /* 字段 */ }
impl ValueObject for MyValue {}
```

### 步骤 3：放置在正确的上下文中

确定新类型属于哪个限界上下文。如果它不适合任何现有上下文，考虑是否需要新上下文。

### 步骤 4：添加测试

Aleph 对领域类型使用双重测试策略：

**Gherkin 测试**（在 `core/tests/features/domain/` 中）用于确定性行为：

```gherkin
Feature: TaskGraph 一致性
  Scenario: 不能添加具有重复 ID 的任务
    Given 一个带有任务 "task-001" 的任务图
    When 我添加另一个带有 ID "task-001" 的任务
    Then 操作应该失败并显示 "duplicate ID"
```

**YAML 规范测试**（在 `core/tests/specs/` 中）用于 AI 相关的语义行为：

```yaml
scenarios:
  - name: "SuccessManifest 正确验证"
    given:
      - success_manifest: { id: "sm-001" }
    then:
      - assertion_type: "deterministic"
        check: "has_identity"
        expected: true
```

这种双重方法确保正确性（Gherkin）和语义有效性（由 LLM 评判评估的 YAML 规范）。

## 设计原则

### 保持聚合小

大聚合创建性能瓶颈和争用。每个聚合应仅包含必须始终彼此保持一致的对象。如果两个对象可以暂时不一致，它们属于不同的聚合。

### 优先使用值对象

值对象比实体更简单、更安全、更高效。如果一个对象不需要身份或生命周期，使其成为值对象。这消除了与身份混淆和引用管理相关的整类错误。

### 在边界强制执行不变量

聚合根负责在其边界内强制执行所有业务规则。外部代码永远不应能够将聚合置于无效状态。这意味着：

- 构造函数验证初始状态。
- 变异方法验证状态转换。
- 类型系统在可能的情况下防止非法操作。

### 使用 Rust 的类型系统

Rust 的所有权模型、trait 系统和类型级编程与 DDD 自然对齐：

- **所有权**强制执行聚合边界 —— 如果聚合根拥有其子项，外部代码不能持有对它们的可变引用。
- **Traits**表达没有运行时成本的领域契约。
- **Enums**使用穷尽匹配对判别联合（如 `FactType` 或 `TaskState`）进行建模。
- **生命周期**防止跨上下文边界的悬空引用。

title: 架构红线
description: 统治 Aleph 中所有开发决策的七个不可违反的设计约束 —— 没有代码可以违反的最高优先级规则。
---

Aleph 的架构由七条**红线**管理 —— 优先于所有其他设计考虑的不可协商约束。这些不是指导方针或最佳实践。它们是硬边界。违反任何红线的代码不能合并，无论它启用的功能多么有用。

红线存在是因为某些架构错误是灾难性和不可逆的。一旦核心模块依赖于平台 API，或接口层积累了业务逻辑，损害就会随着每次后续提交而复合。红线在这些问题开始之前防止这些失败模式。

每个贡献者、每个拉取请求审查者和每个自动化检查都必须将这七条规则视为公理。

## R1. 脑-肢分离

**核心从不接触平台 API。它只定义能力契约。**

Aleph 的 Rust 核心（`src/`）是一个纯推理引擎。它决定做什么、管理状态和路由任务。它必须永远不知道 —— 或关心 —— 它运行在什么操作系统上。

### 规则

核心为平台特定操作定义抽象 trait（能力契约）。物理实现位于桌面桥接（Tauri-Rust 层）中，它通过 IPC 与核心通信。

### 原理

如果核心导入 `AppKit` 来捕获屏幕截图，它就成为 macOS 应用程序。如果它调用 `windows-rs` 来读取窗口标题，它就成为 Windows 应用程序。大脑中的平台耦合破坏了"一个核心，多个外壳"的承诺，并使每个新平台成为分支而非集成。

### 禁止

- 在 `src/` 中导入 `AppKit`、`Vision`、`CoreGraphics`、`windows-rs` 或任何平台特定的系统 API
- 直接从核心调用 OS 级 API 进行屏幕截图、窗口管理、可访问性或输入模拟
- 在核心中引入平台特定行为的条件编译（`#[cfg(target_os)]`）

### 允许

- 在核心中定义诸如 `ScreenCapture`、`WindowManager`、`InputSimulator` 之类的 trait
- 桌面桥接通过平台原生 API 实现这些 trait
- 核心通过 trait 接口在 IPC（UDS JSON-RPC）上调用能力

## R2. UI 单一事实来源

**所有业务 UI 存在于 Leptos/WASM 中。原生外壳只是窗口容器，仅此而已。**

Aleph 有一个 UI 实现，用 Leptos 编写并编译为 WebAssembly。这个单一实现在任何地方运行 —— Web、macOS、Windows、Linux。原生应用外壳（Tauri）提供窗口框架、系统托盘集成和原生动画。它们从不实现业务逻辑或复杂 UI。

### 规则

复杂的设置页面、表单、列表、仪表板和任何具有业务逻辑的 UI 必须在 Leptos（WASM）中实现。Tauri 外壳仅提供：窗口容器、原生菜单栏项、原生动画和系统级集成（通知、文件对话框）。

### 原理

跨平台重复 UI 逻辑是维护灾难。每个错误必须修复 N 次，每个功能实现 N 次，行为一致性变得不可能保证。单一的 Leptos/WASM 代码库消除了这整类问题。

### 禁止

- 在 Tauri/Swift/Kotlin 中实现设置页面、任务列表、对话视图或任何承载业务逻辑的 UI
- 在外壳层构建复杂表单或交互组件
- 在 Leptos 之外渲染数据驱动视图（记忆浏览器、工具配置）

### 允许

- Tauri 提供托管 Leptos/WASM 应用的 `WebView` 容器
- 原生菜单栏项和系统托盘交互（这些是平台集成，而非业务 UI）
- 原生窗口边框、动画和 OS 级便利设施（拖动区域、活力效果）
- 通过 Tauri API 委托的简单原生对话框（文件选择器、警报）

## R3. 核心极简主义

**核心调度。它不为非核心功能做重活。**

Aleph 的核心必须保持精简。如果一个能力不是推理、状态管理或路由的基础，它就不属于核心。非核心功能作为技能（Python/Bash 脚本）或 MCP 服务器实现 —— 核心编排但不包含的外部进程。

### 规则

不要为可以作为技能或 MCP 集成交付的功能向 `core/` 添加重量级第三方库。核心是调度器和协调器。它调用专门的工作者而非吸收它们的复杂性。

### 原理

添加到核心的每个依赖项都会增加编译时间、二进制大小、攻击面和认知负载。为一个功能添加的 200MB 图像处理库会惩罚每个构建、每个部署和每个开发人员 —— 即使那些从不使用该功能的人。技能和 MCP 服务器是隔离的、独立可部署的，并且可以用任何语言编写。

### 禁止

- 为一次性非核心功能向 `core/Cargo.toml` 添加大型 crate（例如，PDF 渲染库、视频转码器）
- 直接在 Rust 核心中实现领域特定专业知识（PowerPoint 生成、代码审查启发式）
- 在没有明确理由的情况下增长核心的依赖树

### 允许

- 核心定义工具接口并调度调用到外部技能/MCP 服务器
- 服务于核心基本职责的轻量级、广泛使用的 crate（序列化、异步运行时、数据库驱动程序）
- 代码组织的奥卡姆剃刀原则（拆分大文件、移除死代码） —— 这补充核心极简主义，而非与之冲突

## R4. 仅 I/O 接口

**接口层是纯 I/O。它们将输入转换为 JSON-RPC 并渲染响应。仅此而已。**

每个接口 —— CLI、Telegram 机器人、Discord 机器人、桌面应用 —— 都是一个薄转换层。它接收用户输入，将其转换为对核心的 JSON-RPC 调用，接收响应，并为用户渲染它。任何接口层都不得执行数据持久化、记忆检索、任务规划或任何形式的业务逻辑。

### 规则

接口代码（在 `apps/` 和网关接口中）仅处理两个职责：(1) 将用户输入转换为核心的 JSON-RPC 消息，(2) 以适当格式将核心响应渲染回给用户。

### 原理

如果 Telegram 机器人实现自己的记忆搜索，它将与 CLI 的记忆搜索不同。如果桌面应用管理自己的任务队列，它就成为一个并行的大脑。智能必须集中在核心中，以便每个接口提供相同的体验，仅在展示上有所不同。

### 禁止

- 从接口代码持久化数据（写入数据库、文件或缓存）
- 在接口层中实现记忆检索、RAG 搜索或知识查询
- 在核心之外构建任务规划器、调度器或多步骤执行逻辑
- 从接口代码进行 LLM API 调用

### 允许

- 将用户消息转换为 JSON-RPC `request` 对象
- 在接口的原生格式中渲染流式响应（markdown、代码块、图像）
- 维护没有业务意义的短暂 UI 状态（滚动位置、输入缓冲区、动画状态）
- 缓存渲染内容以用于显示性能（而非业务数据）

## R5. 菜单栏优先

**在 macOS 上，Aleph 生活在菜单栏中。没有 Dock 图标。Halo 覆盖层是主要交互表面。**

Aleph 在 macOS 上的默认存在是不可见的 —— 一个菜单栏图标和一个浮动的 Halo 覆盖层。它不占用 Dock 空间，不要求永久窗口，也不与用户的活动应用程序争夺屏幕空间。

### 规则

macOS 应用作为没有 Dock 图标的菜单栏常驻程序启动。Halo 浮动覆盖层作为主要快速交互表面。完整窗口（设置、扩展对话、调试面板）按需打开但不是默认状态。

### 原理

个人 AI 助手应该是环境性的 —— 始终可用但从不侵入。Dock 图标暗示用户必须主动管理的"重量级"应用。菜单栏常驻表明 Aleph 是一个系统级服务，始终准备就绪但从不碍事。Halo 覆盖层提供即时访问而无需窗口管理的开销。

### 禁止

- 在 macOS 上默认显示 Dock 图标
- 需要可见窗口进行基本交互（快速问题、状态检查、通知）
- 为了隐形而牺牲可用性 —— 真正需要完整窗口的复杂任务必须有一个

### 允许

- 带有下拉菜单的菜单栏图标，用于状态、快速操作和偏好设置
- 作为主要对话界面的 Halo 浮动覆盖层
- 用于设置、长对话、调试面板和其他复杂场景的完整应用窗口
- 用户偏好可选地显示 Dock 图标，如果他们喜欢传统窗口管理

## R6. AI 来找你

**减少上下文切换。AI 在用户当前环境中帮助，而非在它自己的窗口中。**

Aleph 不要求用户来找它。它去找用户 —— 作为浮动覆盖层出现在他们当前的工作区中，内联显示建议，在正确的时刻发送通知。用户永远不应该需要离开他们当前的任务来与他们的 AI 助手交互。

### 规则

设计所有交互以最小化上下文切换。优先选择覆盖层、内联建议和 ambient notifications 而非模态对话框和专用窗口。AI 应该感觉像用户工作之上的一个层，而非一个单独的目的地。

### 原理

每次上下文切换都有认知成本。从代码编辑器切换到聊天窗口，输入问题，阅读响应，然后切换回来会干扰心流状态。Aleph 应该如此无缝地集成到用户的环境中，以至于与它交互感觉像大声思考，而非操作一个单独的应用程序。

### 禁止

- 在没有显式用户操作的情况下从用户的活动应用程序窃取焦点
- 显示阻止用户工作流的模态对话框
- 要求用户导航到特定窗口或屏幕以访问核心功能

### 允许

- Halo 覆盖层出现在光标处或一致的屏幕位置
- 用户可以操作或忽略的非侵入性通知
- 在上下文中出现的内联建议（例如，代码完成、写作辅助）
- 用户显式启动的任务的完整窗口（设置、长形式交互、调试）
- 尊重边界：不要为了回避显示窗口而保留必要的 UI —— 可用性总是获胜

## R7. 一个核心，多个外壳

**Rust 核心是唯一的大脑。Leptos/WASM 是唯一的 UI 逻辑。原生外壳只是容器。**

这条红线是架构论文陈述。它将 R1（脑-肢分离）和 R2（UI 单一事实来源）综合成产品级承诺：Aleph 恰好有一个大脑（Rust 核心），恰好有一个 UI 实现（Leptos/WASM），以及尽可能多的原生外壳 —— 每个都是一个薄容器，托管 WASM UI 并桥接到平台能力。

### 规则

所有智能、规划、记忆和决策都存在于 Rust 核心中。所有业务 UI 都存在于 Leptos/WASM 中。原生外壳（桌面上的 Tauri，未来的潜在移动外壳）提供窗口托管和平台 API 桥接。没有例外。

### 原理

这个约束是使 Aleph 真正跨平台而没有困扰大多数多平台应用程序的碎片化的原因。添加新平台意味着编写新外壳（窗口容器 + 平台桥接），而非重新实现智能或 UI。每个新平台的边际成本接近最小可能。

### 禁止

- 在 Rust 核心之外实现推理、规划或记忆逻辑
- 在 Leptos/WASM 之外构建业务 UI
- 创建做出自主决策的平台特定"智能"外壳
- 分支代码库以支持新平台

### 允许

- Rust 核心作为所有智能的单一事实来源
- Leptos/WASM 作为所有业务 UI 的单一事实来源
- 托管 WebView（用于 WASM UI）并通过 IPC 暴露平台 API 的薄原生外壳
- 通过编写新外壳而不 touching 核心或 UI 代码来添加新平台

## 北极星

七条红线定义了 Aleph 绝不能做什么。北极星定义了 Aleph 要去哪里。这三条原则指导所有前瞻性决策。

### 架构已定 —— 填充，不要重建

1-2-3-4 架构模型（1 核心、2 面孔、3 肢体、4 神经）是稳定且经过验证的。上面的红线保护它。未来的工作是在这个结构中填充能力，而非重新设计结构本身。

这意味着：当面对新需求时，第一个问题总是"这适合现有架构的哪里？" —— 而非"我们需要新架构吗？"第二个问题的答案是否定的。

### 桌面桥接协议已完成

桌面桥接协议 —— 核心和原生外壳之间的 UDS JSON-RPC 2.0 接口 —— 在 Tauri 版本中完全定义和实现。它涵盖屏幕捕获、窗口管理、输入模拟、OCR 和画布渲染。

未来的工作侧重于跨平台覆盖（确保 Windows 和 Linux 桥接与 macOS 能力匹配）和性能优化，而非协议重新设计。

### 技能驱动未来

架构是骨架。技能是 flesh。一旦 1-2-3-4 结构稳固且七条红线被强制执行，使 Aleph 更有用的主要向量就是添加技能。

技能是可独立部署的领域专业知识单元 —— Python 脚本、Bash 工作流、MCP 服务器 —— 核心通过其工具系统编排。它们决定 Aleph 可以为用户节省多少实际工作。投资于设计良好的技能的每一小时都在每个接口和每个平台上获得回报，因为技能通过核心运行，而核心平等地服务于所有外壳。

Aleph 的未来不是更多架构。它是更多技能。

Aleph 的代理不是一个简单的请求-响应聊天机器人。它是一个认知架构，旨在有目的地追求目标、推理复杂任务并从经验中学习。本页描述了驱动每个代理交互的思维模型 —— 从观察-思考-行动-反馈循环到平衡速度与深度的双过程认知。

该思维模型借鉴认知科学，特别是 Daniel Kahneman 的双过程理论，并通过 Rust 中的具体架构模式实现。

## OTAF 循环

每个代理交互都遵循**观察-思考-行动-反馈**循环。这是驱动 Aleph 行为的基本循环：

```
     ┌──────────┐
     │  观察    │  感知环境：用户输入、
     │          │  上下文、记忆、系统状态
     └────┬─────┘
          │
          v
     ┌──────────┐
     │  思考    │  推理该做什么：分类意图、
     │          │  检索经验、规划行动
     └────┬─────┘
          │
          v
     ┌──────────┐
     │  行动    │  执行计划：调用工具、生成
     │          │  响应、修改状态
     └────┬─────┘
          │
          v
     ┌──────────┐
     │  反馈    │  评估结果：成功了吗？学到
     │          │  了什么？应该重试还是升级？
     └────┬─────┘
          │
          └──────────> （返回观察进行下一次迭代）
```

### 观察

观察阶段在任何推理开始之前收集所有可用上下文：

- **用户输入**：来自任何已连接接口的当前消息或命令。
- **会话上下文**：对话历史、活动会话状态和通道元数据。
- **记忆检索**：通过语义搜索和上下文锚定从记忆系统检索相关事实。
- **系统状态**：可用工具、活动配置、安全策略和资源约束。

关键洞察是观察不是被动的。代理在思考之前主动查询其记忆和环境以构建丰富的上下文。这防止了代理仅基于即时消息而不考虑历史或上下文进行响应的常见失败。

### 思考

思考阶段是双过程认知运作的地方（下面详细描述）。代理：

1. **分类意图** —— 用户试图完成什么？这是一个问题、一个命令、一个创造性任务还是一个多步骤工作流？
2. **检索相关经验** —— 代理以前处理过类似的任务吗？什么有效？什么失败了？
3. **生成计划** —— 什么行动序列将实现目标？需要什么工具？潜在的失败模式是什么？
4. **定义成功** —— "完成"是什么样子的？这是 POE 架构中的成功清单。

### 行动

行动阶段通过调度器和工具系统执行计划：

- **工具调用**：调用内置工具、MCP 服务器或插件功能。
- **响应生成**：为用户生成文本、代码或结构化输出。
- **状态修改**：更新记忆、会话状态或系统配置。
- **子任务编排**：通过 TaskGraph 将复杂计划分解为 DAG 子任务。

所有操作都通过安全守卫系统。潜在危险的操作在执行前需要显式审批。

### 反馈

反馈阶段评估发生了什么并确定下一步：

- **成功评估**：行动是否实现了定义的成功标准？
- **经验记录**：成功的完成被存储在经验数据库中以供将来检索。
- **错误分析**：如果某事失败了，出了什么问题？可以恢复吗？

反馈阶段直接反馈到观察中，创建一个连续循环，可以在不丢失上下文的情况下处理多步骤任务。

## 双过程认知

受 Daniel Kahneman 的《思考，快与慢》启发，Aleph 实现了双过程认知架构，包含两个互补的系统：

### 系统 1：快速且直观

系统 1 是快速的、基于模式匹配的层。它基于积累的经验提供快速的"直觉"：

- **启发式规则**：匹配常见场景的预定义模式（例如，"如果用户要求编辑文件，先检查它是否存在"）。
- **经验检索**：对过去成功任务完成的向量相似性搜索，返回相关解决方案模式。
- **模式匹配**：识别当前任务在结构上与先前解决的某个任务相似。
- **快速分类**：立即将请求分类为简单（直接处理）或复杂（启动系统 2）。

系统 1 是使 Aleph 响应迅速的原因。对于常规任务，代理可以通过识别模式并应用已知解决方案模板几乎立即生成计划。

### 系统 2：缓慢且审慎

系统 2 是深度推理层。它处理直觉不足的情况：

- **LLM 推理**：使用语言模型的完整推理能力来分析复杂问题、生成新颖解决方案和评估权衡。
- **语义验证**：检查提议的计划是否实际上满足用户的意图，而不仅仅是表面级别的请求。
- **成功清单生成**：创建明确的、可测试的标准来说明"完成"是什么样子的。
- **多步骤规划**：将复杂目标分解为有序的行动序列，带有依赖关系和回退策略。

系统 2 是使 Aleph 具备能力的原因。对于新颖或复杂的任务，代理进行更深入的推理以产生超出模式匹配的解决方案。

### 它们如何协作

两个系统协同工作，而非竞争：

```
用户请求
     │
     v
┌─────────────────────────────────┐
│         系统 1（快速）          │
│  针对经验数据库进行模式匹配     │
│                                 │
│  找到匹配？                     │
│  ├── 是（高置信度）             │──> 应用已知解决方案
│  ├── 部分（一些相关性）         │──> 用作系统 2 的起点
│  └── 否（新颖情况）             │──> 完全交给系统 2
└─────────────────────────────────┘
                │
                v （部分或无匹配）
┌─────────────────────────────────┐
│         系统 2（缓慢）          │
│  通过 LLM 进行深度推理          │
│  ├── 分析问题                   │
│  ├── 生成候选计划               │
│  ├── 评估权衡                   │
│  └── 选择最佳方法               │
└─────────────────────────────────┘
                │
                v
         执行计划
```

这反映了专家人类如何解决问题。经验丰富的开发人员不会从头分析每一行代码 —— 他们立即识别模式（系统 1），然后在模式不适用的地方应用仔细的推理（系统 2）。随着时间的推移，随着更多经验被固化，系统 1 处理越来越多的任务，使代理更快、更高效。

## 思考级别

并非每个请求都需要相同深度的思考。Aleph 实现了分层思考模型，将认知努力与任务复杂度相匹配：

### 级别 0：反射性

直接模式匹配，具有高置信度。不需要 LLM 调用。

- 示例："现在几点了？" —— 立即调用时钟工具。
- 示例：问候消息 —— 用上下文问候响应。

### 级别 1：联想性

系统 1 在经验数据库中找到强匹配。最小化 LLM 推理以适应模板。

- 示例："总结这个文件" —— 将摘要技能模板应用于特定文件。
- 示例："运行测试" —— 执行当前项目的已知测试命令。

### 级别 2：分析性

系统 2 参与规划和推理。LLM 生成多步骤计划。

- 示例："将这个模块重构为使用仓库模式" —— 需要理解当前代码、设计重构架构和规划更改序列。
- 示例："调试为什么 API 返回 500 错误" —— 需要调查、假设生成和迭代测试。

### 级别 3：创造性

完整的系统 2 参与，具有多次推理传递。新颖的问题解决。

- 示例："为我们的实时数据管道设计缓存策略" —— 需要领域知识、权衡分析和原创架构思考。
- 示例："为认证系统编写全面的测试套件" —— 需要理解安全边缘情况、生成多样化的测试场景和确保覆盖。

思考级别在观察阶段根据意图分类、任务复杂度评估和经验数据库匹配动态确定。

## 第一性原理锚定

在任何任务执行开始之前，Aleph 应用**第一性原理思考** —— 在开始执行之前定义成功。

传统的 AI 代理直接进入行动：

```
用户请求  -->  执行  -->  希望是正确的
```

Aleph 采用不同的方法：

```
用户请求  -->  定义成功  -->  执行  -->  根据合约验证
```

代理生成**成功清单** —— 一个明确定义以下内容的合约：

- **完成标准**："完成"是什么样子的？应该存在什么工件？
- **硬约束**：必须满足什么条件？必须保持什么不变量？
- **软指标**：应该优化什么质量？什么权衡是可以接受的？

这防止了代理"完成"任务但错过实际意图的常见失败模式。当您知道成功是什么样子的时，每个行动都变得有目的性。

## POE 架构

**原则-操作-评估**（POE）架构将思考模型编排为具有明确关注点分离的结构化执行循环：

```
┌────────────────────────────────────────────────────────┐
│                      POE 循环                          │
├────────────────────────────────────────────────────────┤
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │  P - 原则                                        │  │
│  │  锚定在第一性原理上。                            │  │
│  │  从用户意图生成成功清单。                        │  │
│  └──────────────────────────────────────────────────┘  │
│                        |                               │
│                        v                               │
│  ┌──────────────────────────────────────────────────┐  │
│  │  O - 操作                                        │  │
│  │  使用启发式指导执行。                            │  │
│  │  检索类似经验。应用技能。                        │  │
│  │  对已知模式使用系统 1，                          │  │
│  │  对新颖挑战使用系统 2。                          │  │
│  └──────────────────────────────────────────────────┘  │
│                        |                               │
│                        v                               │
│  ┌──────────────────────────────────────────────────┐  │
│  │  E - 评估                                        │  │
│  │  根据成功清单验证输出。                          │  │
│  │  独立的批评者 —— 不信任执行者                     │  │
│  │  的自我评估。                                    │  │
│  └──────────────────────────────────────────────────┘  │
│                        |                               │
│                        v                               │
│  ┌──────────────────────────────────────────────────┐  │
│  │  决策分支                                        │  │
│  │  通过     --> 固化经验 --> 完成                  │  │
│  │  卡住     --> 切换策略                           │  │
│  │  预算     --> 升级给人类                         │  │
│  │  重试     --> 注入反馈 --> 返回 O               │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 原则阶段

原则阶段将整个执行锚定在第一性原理上：

1. 从用户的消息和上下文中解析用户的意图。
2. 生成定义完成标准、硬约束和软指标的成功清单。
3. 此清单成为评估所有后续阶段的合约。

原则阶段确保代理在开始移动之前始终知道它要去哪里。

### 操作阶段

操作阶段使用双过程认知架构执行计划：

1. 从向量数据库检索类似的过去经验（系统 1）。
2. 如果存在高置信度匹配，应用已知解决方案模式。
3. 如果不存在，让 LLM 参与深度推理和计划生成（系统 2）。
4. 通过调度器的任务图执行计划，调用工具并生成响应。

### 评估阶段

评估阶段根据成功清单验证结果：

1. 独立的批评者评估输出 —— 它不信任执行者的自我评估。
2. 检查成功清单中的每个标准：硬约束必须全部通过；软指标被评分。
3. 评估产生裁决：通过、重试、卡住或预算超出。

执行者和批评者之间的分离至关重要。自我评估是不可靠的 —— 产生有缺陷输出的相同过程通常会判断它是正确的。独立的评估者捕获执行者遗漏的错误。

### 决策分支

基于评估裁决：

- **通过**：任务完成。成功的执行路径被固化到经验数据库中以供将来重用。完成。
- **重试**：输出接近但不完全正确。评估反馈被注入回操作阶段以进行另一次尝试。
- **卡住**：当前策略不起作用。代理切换到替代方法 —— 不同的工具、不同的推理路径、不同的分解。
- **预算超出**：熵预算（重试尝试次数的上限）已耗尽。代理升级给人类用户，而非在无限循环中旋转。

### 熵预算

熵预算是一个关键的安全机制。它防止代理在无法取得进展的任务上浪费资源：

- 每个 POE 循环迭代消耗熵。
- 如果熵预算在实现成功清单之前耗尽，代理停止并升级。
- 这防止了无限重试循环，并确保代理知道何时请求帮助。

预算是根据任务复杂度校准的 —— 简单任务获得小预算（快速失败），复杂任务获得较大预算（在困难中坚持）。

## 自我学习

思维模型不是静态的。每个成功的任务完成都通过经验固化管道反馈回系统：

1. **记录**：完整的执行跟踪 —— 输入、计划、行动、结果 —— 被存储为经验条目。
2. **检测**：当积累 3 个或更多类似经验时，系统检测到一个重复模式并创建一个候选技能。
3. **提升**：当候选技能已被成功重用 5 次或更多次且可靠性高时，它被提升为永久技能。

```
临时成功        -->  经验条目（向量数据库）
重复模式        -->  候选技能
已证明的可靠性  -->  永久技能
```

永久技能成为系统 1 模式库的一部分，允许代理使用级别 1（联想性）思考而非级别 2（分析性）思考来处理以前复杂的任务。这就是代理如何随着时间的推移变得更快、更有能力。

## 整合所有内容

完整的思维模型工作方式如下：

1. **OTAF 循环**提供时刻到时刻的执行循环。
2. **双过程认知**平衡速度（系统 1）与深度（系统 2）。
3. **思考级别**将认知努力与任务复杂度相匹配。
4. **第一性原理锚定**确保代理始终知道其目标。
5. **POE 架构**将执行结构化为原则、操作和评估阶段，并带有问责制。
6. **自我学习**将成功执行转化为可重用的技能。

结果是一个不仅仅响应提示，而是**有目的地追求目标、从经验中学习并知道何时请求帮助**的代理。

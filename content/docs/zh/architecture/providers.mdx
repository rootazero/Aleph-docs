Aleph 的提供商架构将**供应商**与**协议**解耦。 Aleph 不是为每个提供商进行一次性集成，而是按协议族对提供商进行分组。单个 OpenAI 协议适配器处理 OpenAI、DeepSeek、Moonshot 和任何其他 OpenAI 兼容 API。添加新供应商通常只需要一个预设条目 —— 无需新代码。

## 提供商架构

系统组织为三个协议层：

```
┌─────────────────────────────────────────────────────────────┐
│                      提供商栈                                │
├─────────────────────────────────────────────────────────────┤
│  第 1 层 — 内置协议（编译的 Rust）                           │
│    ├─ openai         → HttpProvider + OpenAiProtocol        │
│    ├─ anthropic      → HttpProvider + AnthropicProtocol     │
│    ├─ gemini         → HttpProvider + GeminiProtocol        │
│    ├─ openai-responses → HttpProvider + OpenAiResponsesProtocol │
│    ├─ codex / chatgpt → HttpProvider + Codex 变体           │
│    └─ ollama         → OllamaProvider (原生)                │
├─────────────────────────────────────────────────────────────┤
│  第 2 层 — 可配置协议（YAML，可热重载）                       │
│    ├─ 最小模式：扩展基础协议 + 差异                           │
│    └─ 自定义模式：完整模板渲染                                │
├─────────────────────────────────────────────────────────────┤
│  第 3 层 — 扩展协议（未来）                                   │
│    └─ 插件提供的适配器（WASM / Node.js）                      │
└─────────────────────────────────────────────────────────────┘
                              │
                    ProtocolRegistry
                              │
                   create_provider(name, config)
```

### 提供商解析流程

1. **预设查找** —— `create_provider("deepseek", config)` 检查 `presets::get_preset("deepseek")`
2. **应用预设默认值** —— 如果缺失，自动填充 `base_url`、`protocol`、`color`
3. **协议解析** —— `config.protocol()` 确定适配器（默认为 `"openai"`）
4. **特殊情况** —— `ollama` 和 `mock` 使用原生实现
5. **适配器查找** —— `ProtocolRegistry::global().get(protocol_name)` 返回适配器
6. **提供商实例化** —— `HttpProvider::new(name, config, adapter)` 包装适配器

## AiProvider Trait

所有 AI 后端都实现 `AiProvider`，提供统一的异步接口：

```rust
pub trait AiProvider: Send + Sync {
    /// 核心方法 —— 处理请求并返回结构化响应
    fn process<'a>(
        &'a self,
        payload: adapter::RequestPayload<'a>,
    ) -> Pin<Box<dyn Future<Output = Result<ProviderResponse>> + Send + 'a>>;

    /// 提供商名称（如 "deepseek"、"claude"）
    fn name(&self) -> &str;

    /// 用于 UI 的提供商品牌颜色（十六进制字符串）
    fn color(&self) -> &str;

    /// 此提供商是否支持原生 tool_use
    fn supports_native_tools(&self) -> bool { false }

    /// 此提供商是否支持扩展思考
    fn supports_thinking(&self) -> bool { false }

    /// 用于模型行为解析的协议标识符
    fn protocol(&self) -> &str { "unknown" }

    /// 模型行为覆盖（如用于 OpenRouter 路由到 Claude 的 "anthropic"）
    fn model_behavior_override(&self) -> Option<&str> { None }
}
```

提供程序是线程安全的（`Send + Sync`）并通过 `Arc<dyn AiProvider>` 共享。单个 `process()` 方法接受包含结构化 `UnifiedMessage` 历史的 `RequestPayload`，协议适配器将它们转换为原生 API 格式。

## 协议适配器

`ProtocolRegistry` 将协议名称映射到 `ProtocolAdapter` 实现。内置协议在初始化时通过 `Lazy` 静态初始化注册。

```rust
pub struct ProtocolRegistry {
    dynamic: RwLock<HashMap<String, Arc<dyn ProtocolAdapter>>>,  // YAML 加载的
    builtin: RwLock<HashMap<String, ProtocolFactory>>,           // 编译的 Rust
}
```

查找顺序：
1. **动态协议优先** —— 从 `~/.aleph/protocols/*.yaml` 加载
2. **内置协议回退** —— 工厂函数按需实例化适配器

启动时注册的内置协议：

| 协议 | 适配器 | 用例 |
|------|--------|------|
| `openai` | `OpenAiProtocol` | OpenAI 和 OpenAI 兼容 API |
| `anthropic` | `AnthropicProtocol` | Claude API（原生 Messages API） |
| `gemini` | `GeminiProtocol` | Google Gemini API |
| `codex` / `chatgpt` | `OpenAiResponsesProtocol`（Codex 变体） | 通过 OAuth 的 ChatGPT 订阅 |
| `openai-responses` | `OpenAiResponsesProtocol` | OpenAI /v1/responses API、OpenRouter |

每个 `ProtocolAdapter` 实现两个方法：
- `build_request(payload, config)` —— 构造 HTTP 请求构建器（流式优先：始终设置 `stream: true`）
- `stream_deltas(response)` —— 将 SSE/流式响应解析为细粒度的 `ProviderDelta` 事件

## 提供商配置

每个提供商都通过 `config.toml` 中的 `ProviderConfig` 进行配置：

```toml
[providers.claude]
protocol = "anthropic"
models = ["claude-sonnet-4-5-20250514"]
api_key = "sk-ant-..."
max_tokens = 8192
temperature = 0.7
enabled = true
```

### ProviderConfig 字段

```rust
pub struct ProviderConfig {
    /// 协议："openai"、"anthropic"、"gemini"、"ollama" 等
    pub protocol: Option<String>,

    /// API 密钥（仅运行时，从不持久化到 config.toml）
    #[serde(skip_serializing)]
    pub api_key: Option<String>,

    /// 模型列表。第一个模型是默认模型。
    /// 接受 `model = "xxx"`（向后兼容）和 `models = ["xxx", ...]`。
    pub models: Vec<String>,

    /// 自定义 API 端点（可选）
    pub base_url: Option<String>,

    /// 用于 UI 的品牌颜色（默认："#808080"）
    pub color: String,

    /// 请求超时（秒）（默认：300）
    pub timeout_seconds: u64,

    /// 提供商是否启用（默认：false）
    pub enabled: bool,

    // --- 生成参数 ---
    pub max_tokens: Option<u32>,
    pub temperature: Option<f32>,
    pub top_p: Option<f32>,
    pub top_k: Option<u32>,

    // --- OpenAI 特定 ---
    pub frequency_penalty: Option<f32>,
    pub presence_penalty: Option<f32>,

    // --- Claude / Gemini / Ollama ---
    pub stop_sequences: Option<String>,

    // --- Gemini 特定 ---
    pub thinking_level: Option<String>,      // "LOW" 或 "HIGH"
    pub media_resolution: Option<String>,    // "LOW"、"MEDIUM"、"HIGH"

    // --- Ollama 特定 ---
    pub repeat_penalty: Option<f32>,

    // --- 系统提示词处理 ---
    /// "prepend"（默认）：前置到用户消息
    /// "standard"：使用单独的系统消息
    pub system_prompt_mode: Option<String>,

    // --- 模型行为覆盖 ---
    /// 覆盖基于协议的自动映射。
    /// 示例：用于 OpenRouter 提供商路由到 Claude 的 "anthropic"。
    pub model_behavior: Option<String>,

    /// 提供商是否通过了测试连接
    pub verified: bool,
}
```

### 基于预设的配置

对于已知提供商，预设自动填充默认值：

```toml
[providers.deepseek]
# 无需指定协议或 base_url —— 预设处理它
models = ["deepseek-chat"]
api_key = "sk-..."
enabled = true
```

`create_provider` 工厂在路由到协议适配器之前应用预设默认值：

```rust
// 预设：base_url、protocol 和 color 自动配置
let provider = create_provider("deepseek", config)?;

// 自定义端点：手动指定 base_url
config.base_url = Some("https://my-proxy.example.com/v1".to_string());
let provider = create_provider("my-provider", config)?;
```

## 故障转移和健康

`FailoverProvider` 包装多个提供商并自动路由绕过故障。它实现相同的 `AiProvider` trait，因此调用者看到单个统一的提供商。

```
┌─────────────────────────────────────────┐
│         FailoverProvider                │
├─────────────────────────────────────────┤
│  ┌─────────┐ ┌─────────┐ ┌─────────┐  │
│  │ Claude  │ │ OpenAI  │ │ Gemini  │  │
│  │ (pri=1) │ │ (pri=2) │ │ (pri=3) │  │
│  └────┬────┘ └────┬────┘ └────┬────┘  │
│       │           │          │         │
│       └───────────┼──────────┘         │
│                   ▼                     │
│     Circuit Breaker + Health Monitor    │
│         (每个提供商状态)                │
└─────────────────────────────────────────┘
```

### FailoverConfig

```rust
pub struct FailoverConfig {
    pub providers: Vec<ProviderEntry>,
    pub max_retries: u32,                      // 默认：2
    pub health_check_interval_secs: u64,       // 默认：60
    pub unhealthy_cooldown_secs: u64,          // 默认：300（5 分钟）
    pub health_monitoring_enabled: bool,       // 默认：true
}

pub struct ProviderEntry {
    pub name: String,
    pub priority: u32,    // 越小 = 优先级越高
    pub config: ProviderConfig,
}
```

### 断路器状态机

每个提供商都有一个带有三态断路器的 `HealthState`：

```
                    ┌─────────────┐
         success    │             │    failure count < 3
         ┌──────────│   Closed    │◀─────────────────┐
         │          │  (healthy)  │                  │
         │          └──────┬──────┘                  │
         │                 │ failure count >= 3     │
         │                 ▼                         │
         │          ┌─────────────┐                  │
         │          │             │  cooldown expired│
         │          │    Open     │──────────────────┤
         │          │  (blocked)  │                   │
         │          └──────┬──────┘                   │
         │                 │ probe request allowed    │
         │                 ▼                          │
         │          ┌─────────────┐   probe failed    │
         └──────────│  HalfOpen   │───────────────────┘
                    │  (testing)  │   (cooldown × 2)
                    └─────────────┘
```

**状态转换：**

| 从 | 到 | 触发条件 |
|------|----|---------|
| `Closed` | `Open` | 3 次连续失败（阈值） |
| `Open` | `HalfOpen` | 冷却期已过（默认 5 分钟） |
| `HalfOpen` | `Closed` | 探测请求成功 |
| `HalfOpen` | `Open` | 探测请求失败 —— 冷却时间加倍（最大 10 分钟） |

**错误分类：**
- **不可重试**（立即故障转移）：`AuthenticationError`、`InvalidConfig`
- **可重试**（带退避重试）：速率限制、超时、网络错误
- **速率限制**（特殊）：在将提供商标记为不健康后触发立即故障转移

### 健康追踪

`health.rs` 模块提供一个单独的 `ProviderHealth` 枚举，由认证配置文件系统使用：

```rust
pub enum ProviderHealth {
    Healthy,
    Degraded { since: Instant, cooldown_until: Instant, consecutive_failures: u32 },
    Unavailable { since: Instant, reason: String },
}
```

冷却使用指数退避：基数 30 秒，每次失败加倍，上限为 5 分钟。带有 `retry_after` 标头的速率限制响应对第一次失败使用 `max(retry_after, base_cooldown)`。

### 指标

每个提供商通过原子计数器追踪实时指标：

```rust
pub struct ProviderMetrics {
    pub total_requests: AtomicU64,
    pub success_count: AtomicU64,
    pub failure_count: AtomicU64,
    pub total_latency_ms: AtomicU64,
}
```

通过 `get_metrics()` 查询每个提供商的成功率和平均延迟。

### 故障转移配置示例

```toml
[failover]
max_retries = 2
unhealthy_cooldown_secs = 300

[[failover.providers]]
name = "claude"
priority = 1

[[failover.providers]]
name = "openai"
priority = 2

[[failover.providers]]

```

## 可热重载协议

除了内置协议，你还可以在 YAML 中定义自定义协议并将它们放在 `~/.aleph/protocols/` 中。系统监视此目录并自动重新加载定义。

### 工作原理

1. **文件监视器**检测 Create / Modify / Delete 事件（500 毫秒去抖动）
2. **YAML 解析**为 `ProtocolDefinition`
3. 创建 **`ConfigurableProtocol` 适配器** —— 支持两种模式：
   - **最小模式**（`extends: openai`）：重用基础协议，覆盖 auth/header 差异
   - **自定义模式**（`custom:` 块）：使用 JSONPath 响应映射的完整模板渲染
4. 通过 `ProtocolRegistry::register()` 原子地**更新注册表**
5. **新请求**立即使用更新的协议

### 最小模式示例

```yaml
# ~/.aleph/protocols/my-proxy.yaml
name: my-proxy
extends: openai
base_url: https://proxy.example.com/v1
differences:
  auth:
    header: X-API-Key
    prefix: "Bearer "
```

### 自定义模式示例

```yaml
# ~/.aleph/protocols/exotic-ai.yaml
name: exotic-ai
base_url: https://api.exotic.ai
custom:
  auth:
    type: header
    header: Authorization
    prefix: "Bearer "
  endpoints:
    chat: /v2/completions
    stream: /v2/completions/stream
  request_template: |
    {"model": "{{config.model}}", "messages": [{"role": "user", "content": "{{input}}"}]}
  response_mapping:
    content: "$.choices[0].message.content"
    error: "$.error.message"
```

**重载延迟：** 从文件更改到活动约 600 毫秒（500 毫秒去抖动 + 解析/注册开销）。

## 创建自定义提供商

要添加使用现有协议的新提供商：

### 步骤 1：添加预设

编辑 `src/providers/presets.rs` 并将条目添加到 `PRESETS` HashMap：

```rust
m.insert(
    "my-provider",
    ProviderPreset {
        base_url: "https://api.my-provider.com/v1",
        protocol: "openai",
        color: "#ff6600",
        default_model: "my-model-v1",
    },
);
```

### 步骤 2：实现适配器（仅适用于新协议）

如果供应商使用尚未支持的协议，请实现 `ProtocolAdapter`：

```rust
#[async_trait]
impl ProtocolAdapter for MyProtocol {
    fn build_request(&self, payload: &RequestPayload, config: &ProviderConfig) 
        -> Result<reqwest::RequestBuilder> { ... }
    
    async fn stream_deltas(&self, response: reqwest::Response)
        -> Result<BoxStream<'static, Result<ProviderDelta>>> { ... }
    
    fn name(&self) -> &'static str { "my-protocol" }
}
```

在 `ProtocolRegistry::register_builtin()` 中注册它。

### 步骤 3：注册和使用

对于仅预设的添加（现有协议），无需注册步骤。工厂自动解析：

```rust
let provider = create_provider("my-provider", config)?;
```

对于新协议，在 `ProtocolRegistry` 中注册：

```rust
registry.register("my-protocol", Arc::new(MyProtocol::new(client)))?;
```

1-2-3-4 模型是 Aleph 工程架构的核心组织原则。它回答一个问题：**自托管 AI 助手如何弥合纯推理与物理/数字行动之间的鸿沟？**

答案是在四个编号层之间进行严格的职责分离：

- **1** 核心（大脑）—— 推理、状态、路由
- **2** 面孔（接口）—— 用户如何与 Aleph 交互
- **3** 肢体（执行系统）—— Aleph 如何对世界采取行动
- **4** 神经（通信协议）—— 每一层如何与核心通信

## 1 核心 —— 大脑

Rust 核心是 Aleph 的灵魂。它只负责三件事，别无其他：

| 职责 | 说明 |
|------|------|
| **推理** | 决定下一步做什么。核心编排[工具链](./harness)（观察-思考-行动-反馈），调用 LLM 提供商，并运行[提示系统](./prompt-system)以进行目标导向的执行。 |
| **状态管理** | 维护对话上下文、任务图、[会话密钥](./routing)和[记忆系统](./memory)。 |
| **路由** | 根据任务需求将工作分派到正确的执行系统 —— 原生工具、MCP 服务器或技能插件。 |

### 核心不做的事

核心从不渲染 UI，从不调用平台特定 API（AppKit、Vision、CoreGraphics），也从不会为 niche 功能捆绑重型第三方库。这些是架构红线（R1、R3），保持核心轻量且可移植。

```rust
// 核心将能力契约定义为 trait。
// 物理实现存在于桌面桥接中。
//
// 示例：核心可以请求截图，但实际的
// CGWindowListCreateImage 调用发生在 Tauri 中。
pub trait DesktopCapability: Send + Sync {
    async fn screenshot(&self, region: Option<ScreenRegion>) -> Result<Vec<u8>>;
    async fn click(&self, x: f64, y: f64, button: MouseButton) -> Result<()>;
    async fn type_text(&self, text: &str) -> Result<()>;
}
```

## 2 面孔 —— 接口

Aleph 向世界呈现两张面孔。两者都是纯 I/O 层 —— 它们将用户输入转换为 JSON-RPC 调用，并将响应渲染回给用户。两张面孔都不包含业务逻辑（红线 R4）。

### 面孔 1：统一面板（Leptos/WASM）

单个 Leptos 代码库编译为 WASM，作为所有平台的 UI：

| 平台 | 宿主 | 说明 |
|------|------|------|
| Web | 浏览器 | 由 Aleph 服务器提供 |
| macOS | Tauri 壳 | 菜单栏优先，Halo 浮动窗口 |
| Windows | Tauri 壳 | 系统集成托盘 |
| Linux | Tauri 壳 | 标准桌面窗口 |

原生壳（Tauri）仅提供窗口容器、系统托盘和原生动画。所有复杂 UI —— 设置页面、对话视图、调试面板 —— 都在 WASM 层（红线 R2）。

### 面孔 2：社交机器人网关

[网关](./gateway)将 Aleph 连接到消息平台，作为持久的后台智能：

- **Telegram** —— 通过 teloxide 的机器人接口
- **Discord** —— 通过 serenity 的机器人接口
- **iMessage** —— 通过 AppleScript/Shortcuts 桥接

网关将平台特定的消息格式转换为 Aleph 的统一 JSON-RPC 协议。单个 Aleph 核心可以同时服务多个社交频道，每个频道都有独立的[会话路由](./routing)。

## 3 肢体 —— 执行系统

肢体是 Aleph 对世界采取行动的方式。有三个类别，按与系统的亲近程度排序：

### 肢体 1：原生能力（肌肉）

通过[桌面桥接](./desktop-bridge)和 shell 执行进行直接系统控制：

| 能力 | 实现 | 协议 |
|------|------|------|
| 截图 / OCR | 桌面桥接（Tauri） | UDS JSON-RPC |
| 鼠标 / 键盘 | 桌面桥接（Tauri） | UDS JSON-RPC |
| 窗口管理 | 桌面桥接（Tauri） | UDS JSON-RPC |
| 画布覆盖 | 桌面桥接（Tauri） | UDS JSON-RPC |
| Shell 命令 | 内置 Exec 系统 | 直接 |

桌面桥接作为单独的 Tauri 进程运行。核心通过 Unix 域套接字使用 JSON-RPC 2.0 与其通信，保持大脑-肢体分离原则。

### 肢体 2：MCP（外部工具）

模型上下文协议让 Aleph 利用更广泛的工具生态系统：

```
核心 ──JSON-RPC──▶ MCP 服务器 (Playwright)
核心 ──JSON-RPC──▶ MCP 服务器 (Google Maps)
核心 ──JSON-RPC──▶ MCP 服务器 (GitHub)
```

MCP 服务器作为单独进程运行。Aleph 的 `mcp/` 模块处理服务器生命周期、能力发现和 JSON-RPC 传输。

### 肢体 3：技能和插件（专业知识）

技能代表领域知识和专门能力：

| 技能类型 | 运行时 | 示例 |
|----------|--------|------|
| WASM 插件 | Extism 沙箱 | 数据转换、自定义逻辑 |
| Node.js 插件 | IPC 子进程 | 复杂集成 |
| Python 技能 | 托管子进程 | 数据分析、ML 管道 |
| Bash 技能 | Shell 执行 | 系统管理 |

技能是 Aleph 未来的增长向量。架构是骨架；技能是肌肉和血液。

## 4 神经 —— 通信协议

核心与其面孔或肢体之间的每次交互都通过四个神经通道之一：

| 神经 | 通道 | 协议 | 用途 |
|------|------|------|------|
| **N1** | 核心 —— UI | WebSocket + JSON-RPC | 驱动统一面板。双向：UI 发送用户输入，核心流式传输响应和事件。 |
| **N2** | 核心 —— 桌面桥接 | UDS + JSON-RPC 2.0 | 驱动桌面控制。每个连接一个请求-响应，通过 `~/.aleph/run/desktop-bridge.sock`。 |
| **N3** | 核心 —— 网关 | gRPC / NATS | 驱动社交机器人。高吞吐，支持多频道分发的扇出。 |
| **N4** | 核心 —— MCP | JSON-RPC over stdio/SSE | 驱动外部工具。标准 MCP 传输，每个工具域一个服务器。 |

### 协议选择原理

```
┌──────────────────────────────────────────────────────────────────────┐
│                        协议映射                                       │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  N1 (WebSocket)     最适合：实时双向流式传输                         │
│  ────────────────   原因：UI 需要实时 token 流式传输、                │
│                     事件驱动更新、低延迟                              │
│                                                                      │
│  N2 (UDS/IPC)       最适合：本地高频控制                             │
│  ────────────────   原因：桌面桥接是共置的，                         │
│                     UDS 避免 TCP 开销，亚毫秒级延迟                   │
│                                                                      │
│  N3 (gRPC/NATS)     最适合：结构化服务间通信                         │
│  ────────────────   原因：网关可能运行在不同主机上，                  │
│                     需要模式演进、扇出                                 │
│                                                                      │
│  N4 (JSON-RPC)      最适合：标准化工具集成                           │
│  ────────────────   原因：MCP 生态系统使用 JSON-RPC over             │
│                     stdio 或 SSE，Aleph 遵循标准                      │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

## 架构红线

1-2-3-4 模型由七条架构红线强制执行。违反者不会被合并：

| 红线 | 规则 | 后果 |
|------|------|------|
| **R1** | 大脑-肢体分离 | `core/src` 从不导入平台 API。核心定义 trait；桌面桥接实现它们。 |
| **R2** | UI 单一事实来源 | Tauri 壳没有业务逻辑 UI。所有复杂视图都在 Leptos/WASM 中。 |
| **R3** | 核心极简主义 | 不为 niche 功能引入重型第三方 crate。优先使用技能或 MCP。 |
| **R4** | 仅 I/O 接口 | 面孔从不持久化数据、检索记忆或计划任务。 |
| **R5** | 菜单栏优先 | macOS 默认菜单栏模式，带 Halo 覆盖层。仅在需要时使用完整窗口。 |
| **R6** | AI 走向你 | 减少上下文切换。Halo、通知、内联建议。 |
| **R7** | 一个核心，多个壳 | Rust 核心是单一大脑。UI 通过 WASM 统一。原生壳仅做包装。 |

## 数据流：端到端请求

以下是用户消息如何通过 1-2-3-4 模型流动：

```
1. 用户通过 Telegram 发送消息
   │
   ▼
2. 网关（面孔 2）接收消息，解析路由
   │  → SessionKey: agent:main:telegram:dm:user123
   │
   ▼
3. 核心（大脑）接收 JSON-RPC 请求
   │  → 代理循环：观察 → 思考 → 行动
   │  → 思考器调用 LLM 提供商
   │  → LLM 决定："截图，然后 OCR"
   │
   ▼
4. 核心分派到桌面桥接（肢体 1，神经 N2）
   │  → UDS JSON-RPC: desktop.screenshot → base64 图像
   │  → UDS JSON-RPC: desktop.ocr → 提取的文本
   │
   ▼
5. 核心继续使用 OCR 结果进行推理
   │  → LLM 生成最终响应
   │
   ▼
6. 核心通过网关发送响应（神经 N3）
   │
   ▼
7. 网关将响应投递到 Telegram
   │
   ▼
8. 用户在聊天中看到回复
```

## 设计理念

1-2-3-4 模型反映了一个核心信念：**AI 助手应该是带有可替换肢体的大脑，而非单体。**通过将推理与执行分离，Aleph 可以：

- 添加新频道（面孔）而无需触碰核心
- 添加新能力（肢体）而无需触碰 UI
- 独立升级通信协议（神经）
- 在 headless 服务器上运行核心，而面孔和肢体运行在其他地方

这不是为了微服务而微服务。这是一种务实的分离，让小团队也能维护复杂系统。架构现在已经稳定 —— 未来的工作是填充能力，而非重构骨架。

`teams` 模块为多代理协作提供团队管理，支持具有生命周期管理、计划审批和任务跟踪的结构化团队协作。

## 概览

团队支持：
- **团队创建** —— 定义由指定**队长**与成员组成的代理团队
- **战略协调**（"团队战略协调"，Team Strata）—— 队长先制定计划，该计划被焊入每位成员的提示词
- **协调任务**（Coord Task）—— 为每位成员分配任务，遵循 `Pending → InProgress → WaitingReview → Completed` 生命周期
- **验证循环** —— 队长在成员交付物被视为完成前进行接受或拒绝
- **生命周期管理** —— 跟踪团队从组建到完成的状态
- **SQLite 持久化** —— 团队数据的持久存储

## 架构

```
TeamStore (SQLite)
  ├── Team —— 团队定义和元数据
  ├── TeamMember —— 成员资格和角色
  ├── Plans —— 团队计划和审批状态
  ├── Sessions —— 活动团队会话
  └── Artifacts —— 可交付成果和输出
```

## 战略协调（"团队战略协调"，Team Strata）

团队拥有一名**队长**和若干成员。在团队的**首条消息**时，队长先制定计划 —— 在任何成员开始工作之前。该计划环节**每个团队仅触发一次**（按团队键控的 fire-once 策略），生成的计划会被**焊入每位成员的提示词**，使所有成员共享同一战略上下文。

```
向团队发送首条消息
        │
        ▼
  队长先制定计划  ──►  策略（fire-once，按团队键控）
        │
        ▼
  计划被焊入每位成员的提示词
        │
        ▼
  成员在分配的协调任务下工作
```

成员不会脱离计划独立行动：每位成员针对一个分配的**协调任务**（Coord Task）工作，被焊入的计划使他们的工作与队长的战略保持一致。

## 协调任务与验证循环

每位成员在一个**协调任务**（Coord Task）下工作，其状态沿固定生命周期推进：

```
Pending → InProgress → WaitingReview → Completed
```

当成员提交交付物时，任务进入 `WaitingReview`，一次**合成式再分发**（synthetic re-dispatch）会确定性地将提交路由回**队长**进行审查。队长使用 `task_review` 工具接受或拒绝交付物：

```jsonc
// task_review 工具调用
{
  "task_id": "<coord-task-id>",
  "verdict": "approve",   // 或 "reject"
  "feedback": "..."        // 可选
}
```

`approve` 裁定将任务推进至 `Completed`；`reject` 裁定将其退回返工。该循环使交付物处于队长的掌控之下，而非在提交时被自动接受。

### 协调任务与群聊广播器

`CoordTaskStore` 被接入群聊广播器，使任务状态与团队工作线程保持同步。成员任务提交是上述确定性再分发的触发点：广播器将提交路由给队长，由队长通过 `task_review` 进行审查。持久的团队工作线程随后驱动[群聊](./groupchat)窗口中的归属气泡回放。

## JSON-RPC 接口

团队通过 JSON-RPC 驱动。已确认的 `teams.*` 方法：

| 分组 | 方法 |
|-------|---------|
| 生命周期 | `teams.create`（leader + members + 可选 `auto_name`）、`teams.get`、`teams.list`、`teams.rename`、`teams.disband`、`teams.delete` |
| 工作线程 | `teams.chat.send`、`teams.chat.thread`、`teams.chat.history` |
| 协调任务 | `teams.create_task`、`teams.update_task`、`teams.list_tasks`、`teams.add_task_comment`、`teams.list_task_comments`、`teams.list_task_events`、`teams.list_task_runs` |
| 任务控制 | `teams.task.pause`、`teams.task.resume`、`teams.task.retry`、`teams.task.skip`、`teams.task.trace`、`teams.task.journal.get`、`teams.task.journal.list` |
| 快照 | `teams.snapshot.create`、`teams.snapshot.get`、`teams.snapshot.list`、`teams.snapshot.restore`、`teams.snapshot.delete` |
| 工作流 | `teams.workflow.export_canvas`、`teams.workflow.import_canvas`、`teams.workflow.approve_step`、`teams.workflow.reject_step`、`teams.workflow.retry_step` |
| ACP 成员 | `teams.acp_member.add`、`teams.acp_member.list`、`teams.acp_member.remove` |
| 杂项 | `teams.list_templates`、`teams.usage` |

持久的每团队群聊线程通过 `teams.chat.send` / `teams.chat.thread` / `teams.chat.history` 暴露；该历史驱动[群聊](./groupchat)窗口中的归属气泡回放。`agents.teams` 方法返回的团队列表会附带一个精简的 `members_preview` 和 `last_message`。

> 队长的接受/拒绝决策通过 `task_review` 工具完成（而非 JSON-RPC 方法）。成员在通过 `teams.create_task` / `teams.update_task` 创建/更新的协调任务下工作。

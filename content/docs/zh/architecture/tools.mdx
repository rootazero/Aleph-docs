工具是 Aleph 与世界交互的方式。当代理循环的思考器决定采取行动时，它会调用一个工具——读取文件、运行 shell 命令、搜索网页或调用外部服务。本页涵盖 `ToolService` 外观、中间件栈、`AlephTool` trait、如何从 Rust 类型生成工具模式、注册管道、从验证到响应的执行流程，以及三类工具（内置、MCP、扩展）。

有关代理如何调用工具，请参阅[代理工具链](./harness)。有关记忆特定工具，请参阅[记忆系统](./memory)。

## ToolService 外观

所有工具消费者（代理工具链、MCP 服务器和扩展插件）都通过单个 `Arc<dyn ToolService>` 外观与工具交互。这是消费者看到的唯一接口——分发、权限检查和超时管理的复杂性都隐藏在其后。

```rust
#[async_trait]
pub trait ToolService: Send + Sync + 'static {
    /// 通过名称和 JSON 输入执行工具。
    async fn execute(&self, name: &str, input: Value) -> Result<ToolOutput, ToolError>;

    /// 列出所有当前可用工具。
    async fn list(&self) -> Vec<ToolDefinition>;

    /// 获取单个工具的定义。
    async fn describe(&self, name: &str) -> Option<ToolDefinition>;
}
```

**位置**: `src/tools/service.rs`

外观在启动时由 `src/tools/facade.rs` 中的 `build_tool_service()` 构建。它在 `CoreDispatch` 周围连接中间件层的装饰链，将调用路由到实际工具处理程序。返回的 `Arc<dyn ToolService>` 是代理循环持有的；共享的 `Arc<ToolRegistry>` 被线程化到 `McpClient` 和 `ExtensionManager` 中，以便动态工具注册扩展同一注册表。

## 中间件栈

每个工具调用都通过五层中间件栈，从外到内：

```
┌─────────────────────────────────────────────────────────┐
│  ExecAuditLayer                                         │
│  • 跟踪工具调用（开始 / 成功 / 错误）                    │
│  • 在 ToolOutput 元数据中记录 latency_ms                │
├─────────────────────────────────────────────────────────┤
│  PermissionLayer                                        │
│  • SmartFilter 分类：允许 / 确认 / 拒绝                  │
│  • 危险工具的 ApprovalGate 确认                          │
│  • 通过 set_smart_filter 在运行时可交换                  │
├─────────────────────────────────────────────────────────┤
│  ContextRuleLayer                                       │
│  • ContextRule 评估：允许 / 拒绝 / 重写                  │
│  • 默认空；通过 set_rules 接线规则                       │
├─────────────────────────────────────────────────────────┤
│  TimeoutLayer                                           │
│  • 默认超时 + 配置中的每个工具覆盖                        │
│  • 超时时发出 ToolError::Timeout                        │
├─────────────────────────────────────────────────────────┤
│  CoreDispatch → ToolRegistry                            │
│  • 用于无锁读取的 ArcSwap 支持快照                       │
│  • 路由到 BuiltinHandler / McpHandler / ExtHandler      │
└─────────────────────────────────────────────────────────┘
```

**位置**: `src/tools/middleware/`

每层都实现 `ToolService` 并包装下一层，因此 `execute()` 向内委托，`list()`/`describe()` 透明传递。堆栈在构建后不可变，除了 `PermissionLayer` 和 `ContextRuleLayer` 设置器，它们通过 `ArcSwap` 原子交换其策略。

### CoreDispatch

在堆栈底部，`CoreDispatch` 持有 `Arc<ToolRegistry>` 并通过注册表的 `ArcSwap` 快照路由 `execute` / `list` / `describe`。快照在 *await* 处理程序之前释放，以便并发的 `register`/`unregister` 调用永远不会阻塞正在进行的执行。

**位置**: `src/tools/dispatch.rs`

## AlephTool Trait

每个内置工具都实现 `AlephTool` trait，它提供编译时类型安全和自动 JSON Schema 生成：

```rust
#[async_trait]
pub trait AlephTool: Clone + Send + Sync + 'static {
    /// 工具名称（在 LLM tool_use 调用中使用）
    const NAME: &'static str;

    /// 显示给 LLM 的工具描述
    const DESCRIPTION: &'static str;

    /// 参数类型（通过 schemars 自动生成 JSON Schema）
    type Args: Serialize + DeserializeOwned + JsonSchema + Send;

    /// 返回类型
    type Output: Serialize + Send;

    /// 使用类型化参数执行工具
    async fn call(&self, args: Self::Args) -> Result<Self::Output>;

    /// JSON 接口（由 trait 自动实现）
    async fn call_json(&self, args: Value) -> Result<Value> {
        let typed_args: Self::Args = serde_json::from_value(args)?;
        let result = self.call(typed_args).await?;
        Ok(serde_json::to_value(result)?)
    }

    /// 获取工具定义（由 trait 自动实现）
    fn definition(&self) -> ToolDefinition {
        let schema = schema_for!(Self::Args);
        // ... 如果提供则包括示例，确认标志，严格模式
        ToolDefinition::new(Self::NAME, Self::DESCRIPTION, parameters, self.category())
            .with_confirmation(self.requires_confirmation())
            .with_strict(self.strict_schema())
    }
}
```

**位置**: `src/tools/traits.rs`

### 关键设计决策

- **`Args` 使用 `schemars::JsonSchema`**: `Args` 上的 trait 边界意味着 JSON Schema 在编译时从 Rust 类型定义生成。结构体字段上的文档注释成为模式的 `description`，LLM 使用它来理解每个参数。
- **`call_json` 是自动实现的**: trait 提供默认的 JSON-in/JSON-out 包装器，因此实现者只编写类型化的 `call` 方法。
- **`definition()` 是自动实现的**: 工具定义（名称、描述、模式、示例）是自动派生的——无需手动 JSON 维护。
- **`examples()` 用于少样本学习**: 覆盖以提供使用示例，这些示例被注入到工具定义的 `llm_context` 字段中。

### 动态分发

对于运行时多态性（注册表存储异构工具），动态分发 trait 包装静态 trait：

```rust
pub trait AlephToolDyn: Send + Sync {
    fn name(&self) -> &str;
    fn definition(&self) -> ToolDefinition;
    fn call(&self, args: Value) -> Pin<Box<dyn Future<Output = Result<Value>> + Send + '_>>;
}

// 空实现：任何 AlephTool 也是 AlephToolDyn
impl<T: AlephTool> AlephToolDyn for T { ... }
```

## 模式生成

当你定义工具的参数结构体时，JSON Schema 会自动从 Rust 类型生成：

```rust
#[derive(Debug, Clone, Serialize, Deserialize, JsonSchema)]
pub struct FileReadArgs {
    /// 要读取的文件路径
    pub path: String,

    /// 字符编码（默认：utf-8）
    #[serde(default)]
    pub encoding: Option<String>,
}
```

这会生成以下模式，发送给 LLM：

```json
{
  "type": "object",
  "properties": {
    "path": {
      "type": "string",
      "description": "File path to read"
    },
    "encoding": {
      "type": "string",
      "description": "Character encoding (default: utf-8)"
    }
  },
  "required": ["path"]
}
```

`#[serde(default)]` 属性使 `encoding` 在模式中是可选的（未列在 `required` 中），文档注释成为字段描述。这意味着 LLM 自然知道哪些参数是必需的以及每个参数的用途。

## 工具注册管道

工具在启动时通过 `build_tool_service()` 注册到共享的 `ToolRegistry` 中：

```
启动
    │
    ▼
┌──────────────────────────────────────────────┐
│ 1. 构建 CoreDispatch + ToolRegistry          │
│    → 带有 ArcSwap 快照的空注册表              │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────┐
│ 2. 注册内置工具                              │
│    register_builtins_into(&registry, server) │
│    → 将每个包装在 BuiltinHandler 中           │
│    → 插入到 ArcSwap 支持的注册表              │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────┐
│ 3. 接线中间件层                              │
│    TimeoutLayer → ContextRuleLayer           │
│    → PermissionLayer → ExecAuditLayer        │
│    → 返回 Arc<dyn ToolService>                │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────┐
│ 4. 连接 MCP 服务器                           │
│    → 将注册表 Arc 线程化到 McpClient         │
│    → 发现工具，包装在 McpHandler 中           │
│    → 注册到同一注册表                         │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────┐
│ 5. 加载扩展插件                              │
│    → 将注册表 Arc 线程化到 ExtManager        │
│    → 包装在 ExtensionHandler 中               │
│    → 注册到同一注册表                         │
└──────────────────────────────────────────────┘
```

### ToolRegistry

`ToolRegistry` 是持有所有工具的中央 `ArcSwap` 支持映射：

```rust
pub struct ToolRegistry {
    inner: Arc<ArcSwap<HashMap<String, Arc<dyn ToolHandler>>>>,
    change_tx: broadcast::Sender<RegistryChange>,
}

impl ToolRegistry {
    pub fn register(&self, name: String, handler: Arc<dyn ToolHandler>) -> Result<(), ToolError>;
    pub fn unregister(&self, name: &str) -> Option<Arc<dyn ToolHandler>>;
    pub fn snapshot(&self) -> Arc<HashMap<String, Arc<dyn ToolHandler>>>;
}
```

**位置**: `src/tools/registry.rs`

注册表在注册/注销时发出 `RegistryChange` 事件，MCP 和扩展管理器可以订阅这些事件以进行反应式更新。

## 执行管道

当思考器决定调用工具时，请求流经中间件栈：

```
Thinker 决策: tool_use("bash_exec", {"command": "ls -la"})
    │
    ▼
┌──────────────────────────────────────────────┐
│ 1. EXECAUDIT                                 │
│    • tracing::info!(target: "tool.call", ...) │
│    • 在 ToolOutput 元数据上标记 latency_ms   │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│ 2. PERMISSION                                │
│    • SmartFilter.classify(name, input)       │
│    • 允许 → 通过                              │
│    • 拒绝 → ToolError::PermissionDenied      │
│    • 确认 → ApprovalGate.ask()               │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│ 3. CONTEXTRULE                               │
│    • 评估 ContextRule 规则（如果有）          │
│    • 允许 → 通过                              │
│    • 拒绝 → PermissionDenied                 │
│    • 重写 → 使用新输入转发                    │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│ 4. TIMEOUT                                   │
│    • 应用默认或每个工具覆盖                   │
│    • tokio::time::timeout()                  │
│    • 超时 → ToolError::Timeout               │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│ 5. COREDISPATCH                              │
│    • 获取注册表的 ArcSwap 快照               │
│    • 按名称查找处理程序                       │
│    • 释放快照，等待 handler.invoke()         │
│    • 内置 → BuiltinHandler → AlephToolDyn    │
│    • MCP → McpHandler → McpClient RPC        │
│    • 扩展 → ExtensionHandler → 插件          │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│ 6. RESPOND                                   │
│    • 返回 ToolOutput（值 + 元数据）           │
│    • 反馈回代理工具链 (Feedback)              │
│    • 发出 stream.tool_end 事件               │
└──────────────────────────────────────────────┘
```

### ToolOutput

每个工具执行返回标准化输出：

```rust
pub struct ToolOutput {
    pub value: Value,              // 工具的 JSON 输出
    pub metadata: ToolOutputMetadata,
}

pub struct ToolOutputMetadata {
    pub latency_ms: u64,
    pub tokens_used: Option<u64>,
    pub source: ToolSource,
}
```

失败时，错误是 `ToolError` 变体之一（`NotFound`、`PermissionDenied`、`ValidationFailed`、`Execution`、`Timeout`、`Transport`），错误消息反馈给思考器，以便它可以决定如何恢复——使用不同的参数重试、尝试不同的工具，或向用户报告错误。

**位置**: `src/tools/service.rs`

## 开发自定义内置工具

### 步骤 1：定义参数

```rust
use schemars::JsonSchema;
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize, JsonSchema)]
pub struct TranslateArgs {
    /// 要翻译的文本
    pub text: String,

    /// 目标语言代码（如 "en"、"zh"、"ja"）
    pub target_language: String,

    /// 源语言（如果省略则自动检测）
    #[serde(default)]
    pub source_language: Option<String>,
}
```

### 步骤 2：实现工具

```rust
use crate::tools::AlephTool;

#[derive(Clone)]
pub struct TranslateTool {
    provider: Arc<dyn TranslationProvider>,
}

#[async_trait]
impl AlephTool for TranslateTool {
    const NAME: &'static str = "translate";
    const DESCRIPTION: &'static str =
        "Translate text between languages";

    type Args = TranslateArgs;
    type Output = String;

    async fn call(&self, args: Self::Args) -> Result<Self::Output> {
        self.provider
            .translate(&args.text, &args.target_language)
            .await
    }
}
```

### 步骤 3：注册

```rust
// 在 builtin_tools/mod.rs 中
pub fn register_builtins(server: &mut AlephToolServer) {
    server.register(TranslateTool::new(provider));
    // ... 其他工具
}
```

工具被包装在 `BuiltinHandler` 中，并在 `build_tool_service()` 期间插入到共享的 `ToolRegistry` 中。然后它就可用于具有完全类型模式、自动 JSON 序列化和与权限系统集成的 LLM。

## MCP 集成

[模型上下文协议 (MCP)](https://modelcontextprotocol.io/) 允许将外部工具服务器连接到 Aleph。MCP 工具通过对单独进程的 JSON-RPC 调用。

```rust
pub struct McpClient {
    transport: Transport,  // Stdio、WebSocket 或 HTTP
    tools: Vec<ToolDefinition>,
}

impl McpClient {
    pub async fn list_tools(&self) -> Result<Vec<ToolDefinition>>;
    pub async fn call_tool(&self, name: &str, args: Value) -> Result<Value>;
}
```

### 配置

```json
{
  "mcp": {
    "servers": [
      {
        "name": "filesystem",
        "command": "npx",
        "args": ["-y", "@anthropic/mcp-server-filesystem"],
        "env": { "HOME": "/Users/user" }
      }
    ]
  }
}
```

MCP 工具在启动时通过 `list_tools` RPC 调用发现，包装在 `McpHandler` 中，并注册到与内置工具相同的 `ToolRegistry` 中。注册表中的限定名称为 `{server_id}__{tool}` 以避免冲突。从思考器的角度来看，MCP 工具看起来与内置工具完全相同。

## 工具过滤

`PermissionLayer` 通过 `SmartFilter` trait 表面控制 LLM 看到哪些工具以及哪些需要用户确认：

```rust
pub trait SmartFilter: Send + Sync {
    /// 将工具调用分类为允许 / 确认 / 拒绝。
    fn classify(&self, name: &str, input: &Value) -> Classification;
}

pub enum Classification {
    Allow,
    Confirm { reason: String },
    Deny { reason: String },
}
```

生产实现（`src/tools/middleware/permission/resolver.rs` 中的 `LayeredPermissionResolver`）查询合并的全局 + 每个代理 `ToolPermissionsConfig`。过滤器和审批器可以通过 `PermissionLayer::set_smart_filter()` 和 `set_approver()` 在运行时交换。

### 遗留 ToolFilter（仍支持）

```rust
pub struct ToolFilter {
    pub allowed: Option<HashSet<String>>,  // 白名单（如果设置）
    pub blocked: HashSet<String>,           // 始终排除
    pub require_confirmation: HashSet<String>,
}
```

`allowed` 列表中支持 glob 模式（如 `memory_*` 匹配所有记忆工具）。

工具过滤与安全系统交互：即使工具通过过滤器，`PolicyEngine` 也会检查 `IdentityContext` 以确保会话的角色允许工具调用。访客会话只能调用其 `GuestScope.allowed_tools` 中列出的工具。

## 扩展工具

扩展工具在沙箱插件运行时中运行：

| 运行时 | 功能标志 | 执行模型 |
|---------|-------------|-----------------|
| WASM | `plugin-wasm` | 基于 Extism，沙箱内存 |
| Node.js | `plugin-nodejs` | 生成的进程，stdio 通信 |

扩展工具实现与内置工具相同的 `ToolDefinition` 接口，因此思考器可以无缝发现它们。`CoreDispatch` 透明地路由到正确的处理程序：

```rust
impl ToolService for CoreDispatch {
    async fn execute(&self, name: &str, input: Value) -> Result<ToolOutput, ToolError> {
        let snapshot = self.registry.snapshot();
        let handler = snapshot.get(name).cloned()
            .ok_or_else(|| ToolError::NotFound { name: name.to_string() })?;
        drop(snapshot);  // 在 await 之前释放
        handler.invoke(input).await
    }
}
```

扩展工具在注册表中限定为 `ext__{plugin_id}__{tool_name}` 以避免与 MCP 工具冲突。

## 存储架构

### 四特性后端设计

存储层使用**四个独立的特性**而不是单一的存储。每个调用者仅依赖其所需的能力：

```rust
// src/memory/store/mod.rs
pub trait NoteStore: Send + Sync { /* ... */ }
pub trait RawMemoryStore: Send + Sync { /* ... */ }
pub trait DreamStore: Send + Sync { /* ... */ }
pub trait CompressionStore: Send + Sync { /* ... */ }
```

所有四个都由 `SqliteMemoryBackend` 实现：

```rust
pub struct SqliteMemoryBackend {
    db: Arc<Mutex<Connection>>,
}

pub type MemoryBackend = Arc<SqliteMemoryBackend>;
```

### 模式 (SQLite + sqlite-vec)

#### 核心表

| 表 | 用途 |
|---|---|
| `raw_memories` | 会话记录、附件文本、`is_processed` 标志 |
| `notes_index` | 知识笔记元数据（标题、类别、标签、哈希） |
| `notes_links` | 双向 wikilink 图 |
| `notes_fts` | 笔记内容的 FTS5 全文索引 |
| `notes_vec_map` | 将 `(path, agent_id)` 链接到数字 rowid |
| `notes_vec_{dim}` | sqlite-vec 虚拟表（768/1024/1536 维度） |
| `memory_events` | 不可变事件日志（事件源突变） |
| `context_anchors` | 用于出处的会话到事实链接 |
| `dream_status` | 每日/每周 dream 运行跟踪 |
| `daily_insights` | Dream 守护进程输出缓存 |

#### sqlite-vec 虚拟表

```sql
-- 按嵌入维度创建
CREATE VIRTUAL TABLE notes_vec_768 USING vec0(
    embedding float[768]
);
CREATE VIRTUAL TABLE notes_vec_1024 USING vec0(
    embedding float[1024]
);
CREATE VIRTUAL TABLE notes_vec_1536 USING vec0(
    embedding float[1536]
);
```

`notes_vec_map` 表将人类可读的 `(path, agent_id)` 对桥接到 vec0 表所需的数字 rowid。

## 检索管道

### NoteFactRetrieval

知识笔记的主要检索接口：

```rust
// src/memory/note_retrieval/mod.rs
#[async_trait]
pub trait NoteFactRetrieval: Send + Sync {
    async fn retrieve(
        &self,
        query: &str,
        agent_id: &str,
        limit: usize,
    ) -> Result<Vec<ScoredFact<MemoryFact>>, AlephError>;
}
```

### 混合搜索（向量 + FTS）

```
查询
  ├── 向量搜索 (sqlite-vec) → 带距离分数的候选
  ├── FTS 搜索 (FTS5) → 带匹配分数的候选
  └── RRF 融合 → 统一排名列表
```

倒数排名融合 (RRF) 公式：
```
RRF_score = Σ 1 / (k + rank_i)
```
其中 `k = 60`（常数），`rank_i` = 来源 `i` 的排名。

### 评分管道（6 阶段）

| 阶段 | 说明 | 默认 |
|---|---|---|
| `cosine_rerank` | 将向量搜索分数与新鲜余弦相似度混合 | 启用 |
| `recency_boost` | 为最近创建的事实添加提升 | +0.1 |
| `length_normalization` | 惩罚非常长的内容 | 启用 |
| `time_decay` | 按年龄的指数衰减，下限 0.5 | 半衰期 30 天 |
| `hard_min_score` | 删除低于阈值的候选 | 0.35 |
| `mmr_diversity` | 最大边际相关性——将近似重复项推迟到尾部 | λ = 0.5 |

**删除的阶段：** `importance_weight`（在主权清理中删除）。

## 后台处理

### 压缩服务

在空闲时刻实时运行：

```
原始记忆 (is_processed = false)
  → SessionCompactor: 每会话压缩
  → NoteIndexer: 创建/更新知识笔记
  → 标记 is_processed = true
```

### Dream 守护进程

在系统空闲时运行，有两种节奏：

**每日管道（5 阶段）：**
```
[整合] → [漂移] → [清理] → [衰减] → [每日摘要]
```

**每周管道（6 阶段）：**
```
[整合] → [漂移] → [综合] → [清理] → [衰减] → [每日摘要]
```

```rust
// src/memory/dreaming/mod.rs
pub fn ensure_dream_daemon();
pub fn record_activity();  // 在用户活动时调用以重置空闲计时器
```

## 记忆中心（面板拓扑）

桌面端 **Memory** 顶层模式（`/memory`）是一个统一的宿主，将原先的两个记忆界面
合并到一个视图中：

| 视图 | 原为 | 渲染 |
|---|---|---|
| **图谱视图** | `/memory` 处的放射状"画布" | wikilink 图谱；节点为知识笔记 |
| **表格视图** | `/dashboard/memory` 处的分面"保险库" | 分面、分页的笔记列表，带抽屉 |

两个视图共享同一宿主和同一工具栏（一个 **图谱 ↔ 表格** 切换、一个统一搜索框，
以及一个共享智能体选择器）。该切换是在单一挂载宿主之上的 CSS 视图切换——而非
两个独立路由——因此状态（搜索查询、所选智能体、滚动/缩放）会在翻转后保持。

- **节点身份 == 笔记路径。** 指向同一笔记的图谱节点与表格行共享同一稳定身份，
  这正是跨视图导航得以确定性工作的原因。
- **双向导航：**
  - 表格行的 **在图谱中查看** 会在图谱中高亮对应的节点。
  - 图谱节点的 **在列表中查看** 会将表格跳转到该笔记的分面/页面并打开其抽屉。
- **路由：** 旧的 `/dashboard/memory` 路径现在重定向到 `/memory?view=table`，
  保留任何进行中的链接和书签。

## 治理界面（只读可观测性）

**Settings → Memory** 下的两个只读面板呈现记忆系统的自治理生命周期。
它们仅供观测——所有*写入*仍由 LLM/工具驱动；这些界面绝不会改动记忆。

### 梦境洞察

由 `dreaming.list_insights` 支撑，该面板读取 `DreamStore`（`daily_insights` 缓存
和 `dream_status` 运行跟踪），以展示由 [Dream 守护进程](#dream-守护进程) 生成的
近期**每日综合笔记**，以及**dream 运行历史**（哪种节奏运行过、何时运行、其状态）。
配套的 `dreaming.run_now` 可触发按需的 dream 运行。

```
dreaming.list_insights  →  近期每日综合笔记 + dream 运行历史
dreaming.run_now        →  触发按需的 dream 运行
```

### 纠正记录

由 `memory.list_corrections` 支撑，该面板呈现**纠正 → 蒸馏**生命周期：
对话过程中捕获的用户纠正，以及它们如何被蒸馏回知识笔记。它暴露系统*从纠正中学到了什么*，
而不授予任何直接编辑记忆的途径。

## 检索追踪

`memory.retrieve_with_trace` 接入了真实的 [NoteFactRetrieval](#notefactretrieval) 评分
管道，并返回各阶段的**遥测**，用于调试*为何某个事实出现或未出现*。
仪表数据通过 `TraceSink` 收集，它在每个阶段记录一条 `StageTrace`：

```
query
  → candidate pool
  → BM25 + vector scores
  → RRF fusion
  → cutoff
  → ranked
```

每条 `StageTrace` 携带进入/离开该阶段的候选项及其分数，因此调用方可以精确看到
某个事实在何处被提升、降权或丢弃。这是一个用于调试/检查的 RPC；
它不会改变检索行为——它观测的正是生产检索运行的同一管道。

## 工作记忆组装

### HybridAssembler（规范 2）

替换遗留的 `ContextComptroller::arbitrate` 路径。生成带有显式插槽的 `MemoryEnvelope`：

```rust
// src/memory/assembler/mod.rs
pub struct MemoryEnvelope {
    pub system_slots: Vec<MemorySlot>,      // 始终注入的事实
    pub user_slots: Vec<MemorySlot>,        // 用户查询相关事实
    pub scratchpad: Option<String>,         // 工作笔记
}

pub struct MemorySlot {
    pub content: String,
    pub source: String,
    pub score: f32,
}
```

```rust
#[async_trait]
pub trait WorkingMemoryAssembler: Send + Sync {
    async fn assemble(
        &self,
        query: &str,
        agent_id: &str,
        session_id: Option<&str>,
        budget: AssemblyBudget,
    ) -> Result<MemoryEnvelope, AlephError>;
}
```

### 组装预算

```rust
pub struct AssemblyBudget {
    pub max_tokens: usize,      // 组装记忆的目标令牌预算
    pub min_score: f32,         // 最小事实分数阈值
    pub max_facts: usize,       // 要包含的最大事实数
}
```

## 精选热记忆（规范 A）

手动策划的关键事实的冻结快照，绕过正常检索：

```rust
// src/memory/curated/mod.rs
pub struct CuratedHotMemory {
    pub facts: Vec<CuratedFact>,
    pub frozen_at: i64,
}

pub struct CuratedFact {
    pub content: String,
    pub priority: u8,        // 1-10，越高 = 越重要
    pub source: String,
}
```

- 通过 `remember` 内置工具创建
- 存储在 `~/.aleph/memory/curated/{agent_id}.json`
- 无条件注入每个提示（最多令牌预算）
- **冻结快照** — 更改需要显式重新策划

## 记忆扩展（规范 4）

可插拔的自定义记忆行为扩展：

```rust
// src/memory/extensions/mod.rs
#[async_trait]
pub trait MemoryExtension: Send + Sync {
    fn name(&self) -> &str;
    
    async fn on_memory_created(
        &self,
        event: &MemoryEvent,
        backend: &MemoryBackend,
    ) -> Result<(), AlephError>;
    
    async fn on_memory_retrieved(
        &self,
        query: &str,
        facts: &[MemoryFact],
        backend: &MemoryBackend,
    ) -> Result<Vec<MemoryFact>, AlephError>;
    
    async fn on_session_end(
        &self,
        session_id: &str,
        backend: &MemoryBackend,
    ) -> Result<(), AlephError>;
}

pub struct ExtensionRegistry {
    extensions: Vec<Box<dyn MemoryExtension>>,
}
```

**生命周期钩子：**
- `on_memory_created` — 笔记创建/更新后触发
- `on_memory_retrieved` — 可以修改/过滤检索到的事实
- `on_session_end` — 清理或总结

## 事件溯源

每个笔记突变都被捕获为不可变的 `MemoryEvent`：

```rust
// src/memory/events/mod.rs
pub struct MemoryEvent {
    pub id: i64,
    pub event_type: MemoryEventType,
    pub note_path: String,
    pub timestamp: i64,
    pub payload: String,  // JSON
}

pub enum MemoryEventType {
    NoteCreated,
    NoteUpdated,
    NoteDeleted,
    LinkAdded,
    LinkRemoved,
    TagChanged,
}
```

事件支持：
- 所有记忆更改的审计跟踪
- 时间旅行查询
- 冲突解决
- 扩展钩子触发

## 安全属性

| 关注点 | 缓解措施 |
|---|---|
| UTF-8 截断 | `chars().take(n)`（从不截断字符） |
| 锁中毒 | `unwrap_or_else(|e| e.into_inner())` |
| SQL 注入 | 通过 rusqlite 的参数化查询 |
| 向量边界 | 余弦钳制到 `[-1.0, 1.0]` |
| 令牌溢出 | AssemblyBudget 强制执行限制 |

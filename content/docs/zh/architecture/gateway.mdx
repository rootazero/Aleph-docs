网关是与 Aleph 通信的单一入口点。它在 `ws://127.0.0.1:18790/ws` 上运行 WebSocket 服务器，使用 JSON-RPC 2.0，并将来自任何连接客户端（无论是原生 macOS 应用、Telegram 机器人还是终端 CLI）的请求路由到适当的处理程序。本页涵盖协议层、消息管道、多频道路由和为实时流式传输提供支持的事件系统。

要了解请求到达后如何管理会话，请参阅[会话服务](./session-service)。要了解路由到达代理后会发生什么，请参阅[代理工具链](./harness)。

## 网关组件

```
┌─────────────────────────────────────────────────────────────────────┐
│                        网关服务器                                    │
│                  ws://127.0.0.1:18790/ws                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐        │
│  │   入站       │     │   处理程序   │     │   出站       │        │
│  │   路由器     │ ──▶ │   注册表     │ ──▶ │   发射器     │        │
│  │              │     │              │     │              │        │
│  │ • 解析请求   │     │ • 路由       │     │ • 流式传输   │        │
│  │ • 验证       │     │ • 执行       │     │ • 事件       │        │
│  └──────────────┘     └──────────────┘     └──────────────┘        │
│                                                                     │
│  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐        │
│  │   会话       │     │    事件      │     │  接口        │        │
│  │   管理器     │     │    总线      │     │  注册表      │        │
│  │              │     │              │     │              │        │
│  │ • SQLite     │     │ • 发布/订阅  │     │ • Telegram   │        │
│  │ • 压缩       │     │ • 主题       │     │ • Discord    │        │
│  │ • 历史       │     │ • 订阅       │     │ • iMessage   │        │
│  └──────────────┘     └──────────────┘     └──────────────┘        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

网关有六个核心组件：

- **入站路由器** — 解析传入的 JSON-RPC 帧，验证结构，并分派到正确的处理程序
- **处理程序注册表** — 将方法名（如 `agent.run`、`session.list`）映射到处理函数
- **出站发射器** — 将流式事件和最终响应发送回客户端
- **会话管理器** — 在 SQLite 中持久化对话历史，处理压缩
- **事件总线** — 支持 glob 模式订阅的实时事件分发发布/订阅系统
- **接口注册表** — 跟踪活动聊天集成（Telegram、Discord、iMessage）

## JSON-RPC 2.0 协议

所有通信都使用基于 WebSocket 帧的 [JSON-RPC 2.0](https://www.jsonrpc.org/specification)。有三种消息类型。

### 请求（客户端到网关）

```json
{
  "jsonrpc": "2.0",
  "id": "uuid-xxx",
  "method": "agent.run",
  "params": {
    "message": "Hello, what can you do?",
    "session_key": "agent:main:main"
  }
}
```

`id` 字段是客户端生成的 UUID。网关在响应中返回相同的 `id`，以便客户端可以将请求与响应匹配。

### 响应（网关到客户端）

```json
{
  "jsonrpc": "2.0",
  "id": "uuid-xxx",
  "result": {
    "run_id": "run-123",
    "status": "running"
  }
}
```

对于 `agent.run`，初始响应确认运行已启动。实际的 AI 输出通过流式事件到达。

### 事件（网关到客户端，无 id）

```json
{
  "jsonrpc": "2.0",
  "method": "event",
  "params": {
    "topic": "stream.chunk",
    "data": {
      "run_id": "run-123",
      "content": "Hello! I can help you with..."
    }
  }
}
```

事件是服务器发起的通知（无 `id` 字段）。客户端通过 `events.subscribe` 方法订阅事件主题。

## 消息管道

当 WebSocket 帧到达时，网关通过分层中间件管道处理它：

```
WebSocket 帧
    │
    ▼
┌────────────────────────────────┐
│ 1. 帧解析                      │
│    • 反序列化 JSON-RPC         │
│    • 验证结构                  │
│    • 提取方法 + 参数           │
└────────────────┬───────────────┘
                 │
                 ▼
┌────────────────────────────────┐
│ 2. 中间件管道                  │
│    • TraceLayer (跨度/请求-id) │
│    • MetricsLayer (计时)       │
│    • AuthLayer (认证检查)      │
│    • RateLimitLayer (配额)     │
│    • ValidateLayer (模式)      │
│    • HandlerService (分发)     │
└────────────────┬───────────────┘
                 │
                 ▼
┌────────────────────────────────┐
│ 3. 处理程序分发                │
│    • 在注册表中查找方法        │
│    • agent.run → ExecutionEngine│
│    • session.* → SessionManager│
│    • config.* → ConfigManager  │
└────────────────┬───────────────┘
                 │
                 ▼
┌────────────────────────────────┐
│ 4. 执行                        │
│    • 处理程序处理请求          │
│    • 可能生成 AgentLoop        │
│    • 通过 EventBus 流式传输事件│
└────────────────┬───────────────┘
                 │
                 ▼
┌────────────────────────────────┐
│ 5. 响应                        │
│    • JSON-RPC 结果帧           │
│    • 或 JSON-RPC 错误帧        │
└────────────────────────────────┘
```

中间件管道按此顺序为每个 JSON-RPC 请求运行：

1. **TraceLayer** — 附加请求跨度和跟踪 ID 以实现可观察性
2. **MetricsLayer** — 记录请求计时和方法级指标
3. **AuthLayer** — 验证认证（当 `auth.mode = "none"` 时跳过）
4. **RateLimitLayer** — 强制执行每个身份、每个范围的速率限制
5. **ValidateLayer** — 在到达处理程序之前验证请求模式
6. **HandlerService** — 分发到注册的处理程序进行执行

## 多频道路由

Aleph 同时支持多个聊天接口。每个接口（Telegram、Discord、iMessage 等）将其平台特定的消息转换为统一的 JSON-RPC 协议，并将其转发到网关。

### 接口 Trait

每个频道适配器实现 `Interface` trait：

```rust
#[async_trait]
pub trait Interface: Send + Sync {
    fn name(&self) -> &str;
    async fn start(&self) -> Result<()>;
    async fn stop(&self) -> Result<()>;
    async fn send_message(
        &self,
        target: &InterfaceTarget,
        message: &str,
    ) -> Result<()>;
    fn is_running(&self) -> bool;
}
```

### 可用接口

| 接口 | 功能标志 | 平台 |
|-----------|-------------|----------|
| CLI | `cli` | 所有平台 |
| Telegram | `telegram` | 所有平台 |
| Discord | `discord` | 所有平台 |
| iMessage | (内置) | 仅限 macOS |
| WebChat | `gateway` | 所有平台 |

### 会话键解析

每个频道生成一个唯一的会话键，决定对话是共享还是隔离：

| 会话键格式 | 用例 |
|-------------------|----------|
| `agent:main:main` | 跨频道共享会话 (主会话) |
| `agent:main:telegram:dm:user123` | 每个用户的 Telegram DM (直接消息) |
| `agent:main:discord:group:guild-id` | Discord 服务器频道 (群组) |
| `agent:main:cron:daily-summary` | Cron 作业 / webhook (任务) |
| `subagent:agent:main:translator` | 子代理委托 (子代理) |
| `agent:main:ephemeral:uuid` | 临时的，不持久化 (短暂) |

DM 范围可以按接口配置：

```rust
pub enum DmScope {
    Main,           // 所有 DM 共享主会话
    PerPeer,        // 每个用户隔离的会话（默认）
    PerChannelPeer, // 按频道 + 用户隔离
}
```

## 事件系统

EventBus 提供用于实时事件分发的发布/订阅机制。客户端使用 glob 模式订阅。

### 事件主题

| 模式 | 事件 |
|---------|--------|
| `stream.*` | 所有流式事件 |
| `stream.chunk` | 文本内容块 |
| `stream.agent_trace` | 结构化循环发起的执行跟踪 |
| `stream.tool_start` | 工具执行开始 |
| `stream.tool_end` | 工具执行完成 |
| `agent.*` | 代理生命周期事件 |
| `agent.started` | 运行开始 |
| `agent.completed` | 运行完成 |
| `agent.error` | 运行错误 |
| `session.*` | 会话事件 |
| `config.*` | 配置更改事件 |
| `system.tick` | 定期心跳（正常运行时间、连接、状态版本） |
| `system.shutdown` | 服务器关闭通知 |

### 订阅事件

```json
{
  "jsonrpc": "2.0",
  "id": "sub-1",
  "method": "events.subscribe",
  "params": { "pattern": "stream.*" }
}
```

订阅后，客户端接收所有匹配的事件作为服务器发起的通知，直到取消订阅。

## 连接生命周期

### 认证流程

```
客户端 WebSocket 连接
    │
    ▼
┌─────────────────────────────────┐
│ require_auth 启用？              │
│   是 → 第一帧必须是             │
│         "connect" 方法          │
│   否  → 允许直接访问            │
└─────────────────────────────────┘
    │
    ▼ (如果需要认证)
┌─────────────────────────────────┐
│ 验证凭据                        │
│   • Bearer 令牌                 │
│   • 设备指纹                    │
│   • 公钥签名                    │
└─────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────┐
│ 授予会话令牌                    │
│ 设置客户端角色 (operator/node)  │
└─────────────────────────────────┘
```

认证模式默认为 `Token`。在配置中设置 `auth.mode = "none"` 以完全禁用认证。遗留的 `require_auth` 字段仍然接受以保持向后兼容。

### Connect 请求

启用认证时，客户端必须将 `connect` 消息作为第一帧发送：

```json
{
  "method": "connect",
  "params": {
    "minProtocol": 1,
    "maxProtocol": 1,
    "client": {
      "id": "macos-app",
      "version": "1.0.0",
      "platform": "macos"
    },
    "role": "operator",
    "auth": {
      "token": "bearer_token"
    }
  }
}
```

## OpenAI 兼容路由

除了 WebSocket JSON-RPC API 外，网关还在同一端口 (`18790`) 上暴露 OpenAI 兼容的 HTTP API。这允许使用 OpenAI API 格式的现有客户端和工具无需修改即可连接到 Aleph。

| 路由 | 方法 | 说明 |
|-------|--------|-------------|
| `/v1/models` | GET | 列出可用模型 |
| `/v1/models/{model_id}` | GET | 获取模型详情 |
| `/v1/chat/completions` | POST | 聊天完成（流式 + 非流式） |
| `/v1/embeddings` | POST | 文本嵌入 |
| `/v1/responses` | POST | OpenAI 响应 API |
| `/v1/health` | GET | API 健康检查 |

这些路由支持 bearer 令牌认证（当 `auth.mode = "token"` 时与 WebSocket 认证使用相同的令牌）。在开发模式 (`auth.mode = "none"`) 下，端点是开放的。

## 热重载

网关监视 `~/.aleph/config.toml` 的更改并应用它们而不重启：

```
~/.aleph/config.toml 修改
    │
    ▼
┌─────────────────────────────────┐
│ 去抖 (500ms)                    │
└─────────────────┬───────────────┘
                  │
                  ▼
┌─────────────────────────────────┐
│ 解析 + 验证新配置               │
└─────────────────┬───────────────┘
                  │
                  ▼
┌─────────────────────────────────┐
│ 应用更改：                      │
│ • 重启受影响的接口              │
│ • 更新路由规则                  │
│ • 发出 config.changed 事件      │
└─────────────────────────────────┘
```

`~/.aleph/protocols/` 中的协议适配器定义也通过类似的机制热重载（有关协议适配器系统的详细信息，请参阅[架构概览](./overview)）。

## HTTP 端点

除了 WebSocket 服务器外，网关还在同一端口上提供 HTTP 端点：

| 端点 | 用途 |
|----------|---------|
| `/ws` | WebSocket 升级端点 |
| `/health` | 健康检查（返回 200 OK） |
| `/metrics` | Prometheus 兼容指标 |
| `/` | 内置 WebChat UI 的静态文件 (ControlPlane) |
| `/login` | 会话 cookie 登录页面 |
| `/auth/login` | 登录表单提交 |
| `/auth/logout` | 注销（清除会话 cookie） |

启用 A2A（代理到代理）时，额外的 A2A 路由会挂载在同一 HTTP 服务器上。

## 接口配置示例

```json
{
  "interfaces": {
    "telegram": {
      "token": "BOT_TOKEN",
      "allowFrom": ["+1234567890"],
      "groups": {
        "*": { "requireMention": true }
      }
    },
    "discord": {
      "token": "BOT_TOKEN",
      "guilds": ["guild-id-1"]
    }
  }
}
```

每个接口都有自己的认证和范围配置。Telegram 可以限制允许哪些电话号码；Discord 可以限制机器人加入哪些公会。

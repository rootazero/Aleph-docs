Aleph 组织为五个水平层，形成一个清晰的处理管道：**接口层**、**网关层**、**编排器层**、**工具链层**和**存储层**。每个用户消息从上到下遍历这些层；每个响应从下到上返回。本页映射整个系统，展示模块如何相互依赖，并跟踪请求的完整生命周期。

要深入了解各个子系统，请参阅[网关架构](./gateway)、[工具链](./harness)、[会话服务](./session-service)、[工具架构](./tools)和[记忆系统](./memory)。

## 设计理念

> **一核多端，壳核分离；网关互通，权限控制。**

Aleph 的架构建立在四个原则之上，这些原则指导每一个设计决策：

1. **一核多端** —— 一个 `aleph-server` 进程驱动所有界面。无论你通过桌面应用、CLI、Telegram 还是 Discord 聊天，同一个核心处理所有推理、工具执行和状态管理。

2. **壳核分离** —— 桌面应用是一个*薄原生壳*（Tauri v2）。它拥有窗口、托盘、通知和 OS 集成——但**零业务逻辑和零业务 UI**。所有产品 UI 在由核心提供的 Leptos Panel 中。所有推理、工具和状态在 `aleph-server` 中。壳可以连接本地核心或远程核心（通过 Tailnet/VPN）。

3. **网关互通** —— 每个终端使用相同的认证 WebSocket 协议（`JSON-RPC 2.0` over `ws://`）。CLI 工具、Telegram 机器人和桌面壳都在同一个网关上调用相同的方法。

4. **权限控制** —— 每个连接都经过认证。桌面壳从核心的安全数据库读取共享令牌。远程连接使用相同的令牌机制。访客会话携带受限范围。详细信息请参阅[身份与安全](#身份与安全)。

> **从「多端」到「多体」。** 一核多端让一个大脑拥有多个 I/O 通道。[集群联邦](./cluster) 把同一原则推进到多个*执行体*：一个中心核编排多台机器（节点）协同工作 —— 而集群始终只有一个心智。

### 第 1 层：接口

最顶层包含所有客户端应用程序。每个客户端——无论是桌面壳、终端 CLI 还是 Telegram/Discord 机器人——都通过单个经过认证的 WebSocket 端点 `ws://127.0.0.1:18790/ws` 连接到同一个网关。通信使用 JSON-RPC 2.0，意味着每个客户端无论平台如何都使用相同的协议。

**桌面壳（Tauri v2）。** 原生桌面应用是"最后一公里原生壳"。它提供：
- 托管 Leptos Panel（`http://127.0.0.1:18790`）的原生窗口 + webview
- 带有"退出"和"退出并停止 Aleph"选项的系统托盘
- 通过 WebSocket 事件桥接的 OS 通知
- 全局召唤热键（`CmdOrCtrl+Shift+A`）
- 用于深度链接的 `aleph://` URL 方案
- 后台自动更新检查器
- 开机自启
- 守护进程生命周期监督（核心崩溃时重新启动）

关键的是，壳包含**零业务逻辑和零业务 UI**。所有产品 UI（聊天、记忆、设置）在 Panel 中（由核心提供的 WASM）。所有推理、工具和状态在 `aleph-server` 中。这种分离意味着：
- 壳可以独立于核心更新
- 核心可以在服务器上无头运行，而壳远程连接
- 同一个核心同时服务桌面壳、CLI 和消息机器人

**远程核心支持。** 桌面壳可以通过 Tailnet 或 VPN 连接到远程 `aleph-server`。Panel 的目录选择器通过 JSON-RPC（`fs.*`）浏览*服务器*的文件系统，确保远程部署的正确语义。认证使用壳在启动时注入的共享令牌（通过核心的 `bootstrap-token` 子命令从 `~/.aleph/data/security.db` 读取）。

### 第 2 层：网关

网关是控制平面。它接受 WebSocket 连接，解析 JSON-RPC 消息，将它们路由到正确的处理程序，并管理会话、事件和认证。它还托管接口注册表（跟踪活动聊天集成）、配置热重载监视器和后台任务的 cron 调度器。详细信息请参阅[网关架构](./gateway)。

### 第 3 层：编排器

编排器为每个传入请求解析 `AgentDef` 和 `FlowSpec`，组装 `HarnessDeps`（注入 `SessionService`、`ToolService`、`Sandbox` 和 `AiProvider`），并分发到 `HarnessRunner::run`。这一层将路由关注点与执行逻辑分离，使网关保持为薄控制平面。请参阅[工具链](./harness)了解编排器-工具链交互。

### 第 4 层：工具链

工具链层运行核心智能循环。当消息到达时，`AgentHarness` 执行 **Think→Act** 循环，包括上下文预算执行、停止钩子评估和可选压缩：它在预算约束下收集上下文，调用 LLM 决定做什么，如果需要则执行工具，评估停止钩子和守卫，并可选地压缩历史记录。`ToolService` 处理工具执行，而守卫强制执行迭代上限和令牌预算等安全限制。请参阅[工具链](./harness)了解完整循环。

### 第 5 层：存储

底层提供持久存储和状态管理。**记忆**使用 SQLite 和 `sqlite-vec` 扩展进行向量搜索，加上 markdown 笔记和 wikilink 图。**弹性恢复**在 SQLite 中存储任务恢复和状态。**配置**通过 TOML 文件和密钥管理持久化设置。详细信息请参阅[记忆系统](./memory)。

## 数据流：请求生命周期

当用户发送消息时，会发生以下序列：

```
客户端请求 (WebSocket 上的 JSON-RPC)
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 网关：InboundRouter                                  │
│   • 解析 JSON-RPC 消息                              │
│   • 路由到适当的处理程序                             │
│   • 认证检查（如果启用）                             │
│   • 构建 FlowRequest { agent_id, input, ... }      │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 编排器                                              │
│   • 解析 AgentDef + FlowSpec                        │
│   • 构建 HarnessDeps (SessionService, ToolService, │
│     Sandbox, AiProvider)                            │
│   • 分发到 HarnessRunner::run                       │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ AgentHarness: Think → Act 循环                      │
│   1. 上下文：应用预算、预检准备                      │
│   2. 思考：调用 Thinker (LLM) 做决定                 │
│   3. 行动：通过 ToolService 执行工具                 │
│   4. 停止钩子：评估完成/守卫                         │
│   5. 压缩：如果溢出，压缩历史                        │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ FlowOutcome → 网关渲染响应                           │
│   • 通过 TraceSink / EventBus 流式传输事件           │
│   • 最终响应作为 JSON-RPC 结果                       │
│   • 会话历史持久化到 SQLite                          │
└─────────────────────────────────────────────────────┘
```

### 工具执行子流程

当 Thinker 决定调用工具时，执行通过分发器和引擎分支：

```
Thinker 决策 (tool_use)
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 分发器                                              │
│   • 分析工具请求                                    │
│   • 检查权限 (ToolFilter)                           │
│   • 风险评估                                        │
│   • 确认流程（如果需要）                             │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 引擎                                                │
│   • 查找工具 (Builtin / MCP / Extension)            │
│   • 反序列化参数                                    │
│   • 带超时执行                                      │
│   • 捕获输出                                        │
└─────────────────────────────────────────────────────┘
    │
    ├─── 内置工具 (AlephTool trait)
    │       • 直接 Rust 执行
    │
    ├─── MCP 工具 (Model Context Protocol)
    │       • 对外部进程的 JSON-RPC
    │
    └─── 扩展工具 (WASM / Node.js)
            • 插件运行时执行
```

Exec 类工具 (`code_exec`, `bash_exec`) 在工具和进程执行之间通过额外的**沙箱层** (`src/sandbox/`)。沙箱拥有每会话工作空间配置 (`~/.aleph/workspaces/{session_id}/`)、能力执行和通过 `OsSandboxDriver` 的 OS 级 seatbelt 隔离。

## 设计模式

Aleph 在代码库中采用了几种 Rust 惯用的设计模式。

### 上下文模式

将相关函数参数分组到专用结构体中，减少参数膨胀并启用构建器模式：

```rust
// 之前：7 个参数直接传递
legacy_agent_loop.run(request, context, tools, identity, callback, abort_signal, initial_history).await

// 之后：结构化 deps + FlowRequest
let deps = HarnessDeps { session, tool_service, sandbox, provider, trace_sink };
let flow = FlowRequest::new(agent_id, input, identity)
    .with_abort_signal(abort_rx);
HarnessRunner::run(flow, deps).await
```

**好处：**
- 可扩展性：添加参数而不会破坏更改
- 可读性：清晰的参数分组
- 类型安全：编译时验证
- 人体工程学：可选参数的构建器模式

### 新类型模式

将原始类型包装在 distinct 结构体中以实现编译时安全。你不能意外地将 `SessionId` 传递给需要 `ExperimentId` 的地方：

```rust
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub struct ExperimentId(String);

impl ExperimentId {
    pub fn new(id: impl Into<String>) -> Self { Self(id.into()) }
    pub fn as_str(&self) -> &str { &self.0 }
}

impl Deref for ExperimentId {
    type Target = str;
    fn deref(&self) -> &Self::Target { &self.0 }
}
```

**新类型目录：**
- **ID**: `ExperimentId`, `VariantId`, `ContextId`, `TaskId`, `SubscriptionId`
- **集合**: `Ruleset` (权限规则)
- **值**: `Answer` (问题响应)

### FromStr Trait 模式

在整个代码库中提供一致的解析接口：

```rust
impl FromStr for TaskStatus {
    type Err = String;
    fn from_str(s: &str) -> Result<Self, Self::Err> {
        match s.to_lowercase().as_str() {
            "pending" => Ok(Self::Pending),
            "running" => Ok(Self::Running),
            _ => Err(format!("Invalid TaskStatus: {}", s)),
        }
    }
}

// 用法
let status: TaskStatus = "pending".parse()?;
```

**已实现于：** `FactType`, `HookKind`, `DeviceType`, `TaskStatus`, `RuntimeKind` 和 10+ 其他类型。

## 功能标志

所有生产功能始终编译；不需要功能标志。仅保留测试功能：

```toml
[features]
default = []
loom = ["dep:loom"]       # 并发测试
test-helpers = []          # 集成测试辅助
```

频道通过 `aleph.toml` 配置在运行时启用或禁用。

## 协议适配器系统

Aleph 通过分层协议适配器架构支持多个 AI 提供商：

| 层 | 类型 | 示例 |
|-------|------|---------|
| **内置** | 编译 Rust | `OpenAiProtocol`, `AnthropicProtocol`, `GeminiProtocol`, `OllamaProvider` |
| **可配置** | YAML 热重载 | `~/.aleph/protocols/` 中的自定义协议，扩展现有或完全自定义 |
| **扩展** | 插件 | WASM/Node.js 协议（未来） |

`ProtocolRegistry` 通过首先检查动态 (YAML) 协议，然后检查内置协议来解析协议名称。YAML 协议定义在文件更改后 600ms 内使用 `notify-debouncer-full` 热重载。

## 身份和安全

不可变的 `IdentityContext` 从会话创建到工具执行贯穿整个执行链。这支持细粒度的、基于角色的访问控制：

- **所有者**会话对所有工具有完全访问权限
- **访客**会话携带限制哪些工具可用以及会话何时过期的 `GuestScope`
- **匿名**会话默认被拒绝

`PolicyEngine` 在执行器级别执行无状态权限检查，确保每个工具调用在执行前都经过授权。

### 安全属性

1. **不变性**: `IdentityContext` 创建后无法修改
2. **冻结权限**: 访客范围在会话创建时冻结
3. **无状态检查**: `PolicyEngine` 没有可变状态
4. **授权证书**: 身份构建一次并向下传递
5. **故障安全**: 缺失或无效的元数据默认为所有者（向后兼容）

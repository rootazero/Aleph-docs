## Spec 1：记忆捕获钩子

**状态：** 已发布  
**重点：** 记忆生命周期事件的可观察性和扩展点。

引入了三个钩子：
- `PreCompress` —— 在会话压缩之前触发；允许扩展检查/修改原始记忆
- `Delegation` —— 当记忆操作委托给子代理时触发
- `SessionEnd` —— 当会话关闭时触发；触发最终压缩

这些钩子成为后续 `MemoryExtension` trait（Spec 4）的基础。

## Spec 2：MemoryReflector

**状态：** 已发布  
**重点：** 原始检索之上的综合层。

`MemoryReflector` 不返回事实的排名列表，而是：
1. 通过 `HybridAssembler` 检索候选事实
2. 要求 LLM 综合一个连贯的答案
3. 返回带有引用来源的答案

```rust
pub struct MemoryReflector {
    assembler: Arc<dyn WorkingMemoryAssembler>,
    llm: Arc<dyn LlmBackend>,
}
```

通过 `memory_reflect` 内置工具暴露。

**关键洞察：** 检索提供候选；综合产生答案。

## Spec 3：上下文围栏 + 记忆模式

**状态：** 已发布  
**重点：** 控制记忆如何注入提示词。

三种记忆模式：
- `Context` —— 记忆直接注入系统提示词
- `Tools` —— 通过 `memory_search` / `memory_reflect` 按需检索记忆
- `Hybrid` —— 两者兼有：关键事实在上下文中，详细查询通过工具

上下文围栏通过使用 `AssemblyBudget` 限制确保记忆永远不会超过 token 预算。

## Spec 4：可插拔记忆扩展

**状态：** 已发布  
**重点：** 第三方自定义记忆行为的扩展。

```rust
#[async_trait]
pub trait MemoryExtension: Send + Sync {
    fn name(&self) -> &str;
    async fn on_memory_created(&self, event: &MemoryEvent, backend: &MemoryBackend
    ) -> Result<(), AlephError>;
    async fn on_memory_retrieved(
        &self, query: &str, facts: &[MemoryFact], backend: &MemoryBackend
    ) -> Result<Vec<MemoryFact>, AlephError>;
    async fn on_session_end(
        &self, session_id: &str, backend: &MemoryBackend
    ) -> Result<(), AlephError>;
}
```

扩展在启动时注册，并接收所有记忆事件的回调。

> `MemoryExtension` trait 此后已收敛到五个钩子点：`on_capture`、`on_retrieve`、`on_delegation`、`on_pre_compress` 和 `on_session_switch`。

## Spec A：策划的热记忆

**状态：** 已发布  
**重点：** 人类策划的、冻结的关键事实快照。

问题：自动检索有时会遗漏对人类来说显而易见的事实。

解决方案：允许用户通过 `remember` 工具固定关键事实：

```
<builtin_tool>remember</builtin_tool>
<parameter name="content">用户对贝类过敏</parameter>
<parameter name="priority">10</parameter>
```

- 存储在 `~/.aleph/memory/curated/{agent_id}.json`
- 冻结快照 —— 更改需要显式重新策划
- 无条件注入每个提示词（在 token 预算内）
- 完全绕过正常检索和评分

## Spec B：session_search 摘要

**状态：** 已发布  
**重点：** 摘要匹配查询的多个会话。

`session_search` 工具现在返回：
1. 带有相关性评分的匹配会话
2. 跨会话综合发现的 AI 生成摘要
3. 来自转录的关键引用

```
<builtin_tool>session_search</builtin_tool>
<parameter name="query">我们关于数据库架构的决定是什么？</parameter>
<parameter name="limit">5</parameter>
```

## Spec C：跨进程安全

**状态：** 已发布  
**重点：** 从多个进程安全并发访问 SQLite。

问题：Aleph 可以运行多个进程（CLI + GUI，或多个代理）。

解决方案：
- SQLite 的 **WAL 模式**（预写日志）
- **忙超时**（5 秒）带指数退避
- **锁恢复**：`unwrap_or_else(|e| e.into_inner())`
- **文件锁定**用于策划记忆写入
- **原子操作**用于笔记索引更新

## 记忆中心统一

**状态：** 已发布（2026-06-20）  
**重点：** 将两个原有的记忆界面折叠到一个宿主中。

放射状的图谱画布（原 `/memory`）和分面式的表格库（原 `/dashboard/memory`）现在合并为单一的**记忆中心**，配备共享工具栏 —— 图谱视图 ↔ 表格视图切换、统一搜索和共享的智能体选择器。

- 节点身份即笔记路径；同一篇笔记在两个视图中是同一个实体。
- 双向导航：表格行的**在图谱中查看**会高亮对应节点，图谱节点的**在列表中查看**会将表格跳转到该笔记的分面/页面并打开其抽屉。
- 旧的 `/dashboard/memory` 路由现在重定向到 `/memory?view=table`。

## 治理呈现

**状态：** 已发布（2026-06-20）  
**重点：** 记忆治理生命周期的只读可观察性。

写入仍由 LLM/工具驱动；这些界面仅作观察。两者都位于 **设置 → 记忆** 下。

- **梦境洞察**面板 —— 来自梦境守护进程的近期每日综合笔记以及 dream-run 历史，由 `dreaming.list_insights` 支撑（并通过 `dreaming.run_now` 按需触发一次综合）。
- **纠正记录**面板 —— 呈现纠正 → 蒸馏的生命周期，由 `memory.list_corrections` 支撑。
- **检索追踪** —— `memory.retrieve_with_trace` 接入真实的 `NoteFactRetrieval` 评分流水线，返回各阶段的遥测数据（查询 → 候选池 → BM25 + 向量评分 → RRF 融合 → 截断 → 排名），用于调试某个事实为何浮现或未浮现。

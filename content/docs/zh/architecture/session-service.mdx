会话服务是 Aleph 中所有对话状态的中心枢纽。它不仅处理活动会话的生命周期，还负责持久化会话数据、协调参与者、以及管理会话的 CRDT 合并。本页涵盖会话服务外观、Actor 模型实现、CRDT 持久化、命令查询分离、通道生命周期，以及会话与认证系统的交互。

---

## SessionService 外观

```rust
#[async_trait]
pub trait SessionService: Send + Sync + 'static {
    /// 创建新会话
    async fn create(
        &self,
        owner_id: String,
        model_id: String,
        config: SessionConfig,
    ) -> Result<Session, SessionError>;

    /// 获取会话（包括活跃和已归档）
    async fn get(&self, id: &str) -> Result<Session, SessionError>;

    /// 列出用户可访问的会话
    async fn list(&self, user_id: &str) -> Result<Vec<Session>, SessionError>;

    /// 向会话发送消息（命令入口）
    async fn send_message(
        &self,
        session_id: &str,
        message: Message,
    ) -> Result<MessageResponse, SessionError>;

    /// 订阅会话事件流
    async fn subscribe(
        &self,
        session_id: &str,
    ) -> Result<mpsc::Receiver<SessionEvent>, SessionError>;

    /// 归档会话（软删除）
    async fn archive(&self, id: &str) -> Result<(), SessionError>;

    /// 删除会话（硬删除）
    async fn delete(&self, id: &str) -> Result<(), SessionError>;
}
```

**位置**: `src/sessions/service.rs`

外观隐藏了 Actor 模型、CRDT 合并和持久化的复杂性。所有操作通过命令发送到对应的会话 Actor，Actor 处理状态变更并响应。

## Actor 模型实现

每个活跃会话由一个独立的 `SessionActor` 管理：

```rust
pub struct SessionActor {
    session_id: String,
    state: SessionState,           // 当前会话状态
    crdt: SessionCRDT,             // CRDT 文档
    participants: Vec<Participant>, // 参与者列表
    event_tx: broadcast::Sender<SessionEvent>,
    command_rx: mpsc::Receiver<SessionCommand>,
}

impl SessionActor {
    async fn run(mut self) {
        while let Some(cmd) = self.command_rx.recv().await {
            match cmd {
                SessionCommand::SendMessage { message, respond_to } => {
                    let result = self.handle_send_message(message).await;
                    let _ = respond_to.send(result);
                }
                SessionCommand::AddParticipant { participant, respond_to } => {
                    let result = self.handle_add_participant(participant).await;
                    let _ = respond_to.send(result);
                }
                SessionCommand::MergeCRDT { delta, respond_to } => {
                    let result = self.handle_merge_crdt(delta).await;
                    let _ = respond_to.send(result);
                }
                // ... 其他命令
            }
        }
    }
}
```

**位置**: `src/sessions/actor.rs`

### 为什么使用 Actor 模型

- **顺序处理**: 每个 Actor 单线程处理命令，无需在会话状态上加锁
- **容错性**: Actor 崩溃只影响单个会话，监督者可以重启它
- **背压**: `mpsc` 通道在 Actor 过载时自然产生背压
- **事件广播**: `broadcast` 通道向所有订阅者发送会话事件

### 监督树

```
SessionSupervisor
    ├── SessionActor (session-001)
    ├── SessionActor (session-002)
    ├── SessionActor (session-003)
    └── ...
```

`SessionSupervisor` 监控所有会话 Actor。如果 Actor 崩溃，监督者根据策略决定：重启 Actor（从持久化状态恢复）、停止 Actor（终止会话）或升级错误（影响整个节点）。

## CRDT 持久化

会话状态使用无冲突复制数据类型（CRDT）建模，支持多设备实时协作：

```rust
pub struct SessionCRDT {
    messages: ListCRDT<Message>,       // 消息的有序列表
    metadata: MapCRDT<String, Value>,  // 键值元数据
    participants: SetCRDT<Participant>, // 参与者集合
}
```

### 持久化流程

```
消息到达
    │
    ▼
┌──────────────────────────────────────────────┐
│ 1. 应用到本地 CRDT                           │
│    → 生成本地 delta                          │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────┐
│ 2. 广播 delta 给所有参与者                    │
│    → 通过 event_tx.broadcast()               │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────┐
│ 3. 异步持久化到存储                           │
│    → SQLite / PostgreSQL / S3                │
│    → 每 N 秒或 M 条消息批量写入               │
└──────────────────────────────────────────────┘
```

### 合并策略

当离线编辑的设备重新连接时：

```rust
impl SessionCRDT {
    pub fn merge(&mut self, other: &SessionCRDT) {
        self.messages.merge(&other.messages);
        self.metadata.merge(&other.metadata);
        self.participants.merge(&other.participants);
    }
}
```

CRDT 保证：无论合并顺序如何，所有设备最终都会收敛到相同状态，无需中央协调器。消息列表使用 LWW（最后写入胜出）元素集，元数据使用 LWW 寄存器，参与者集合使用 OR-Set。

## 命令与查询分离

会话服务遵循 CQRS 模式：

```
┌─────────────────────────────────────────────────────────┐
│                     Command Side                        │
│  send_message() → SessionCommand::SendMessage          │
│  add_participant() → SessionCommand::AddParticipant    │
│  merge_crdt() → SessionCommand::MergeCRDT              │
│                                                         │
│  → 发送到 SessionActor → 状态变更 → 事件广播            │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│                     Query Side                          │
│  get() → 从持久化存储读取最新快照                       │
│  list() → 从索引查询用户会话列表                        │
│  subscribe() → 连接到 event_rx 接收实时事件             │
│                                                         │
│  → 不修改状态，可水平扩展                               │
└─────────────────────────────────────────────────────────┘
```

### 为什么使用 CQRS

- **读优化**: 查询可以走单独的读取副本或缓存，不干扰写入
- **事件溯源**: 所有状态变更都通过事件表示，支持审计和回放
- **扩展性**: 读侧可以独立扩展（更多查询节点），写侧保证一致性
- **实时同步**: 事件流天然支持实时协作

## 通道生命周期

会话通过通道（Channel）连接到外部客户端：

```
客户端连接
    │
    ▼
┌──────────────────────────────────────────────┐
│ 1. WebSocket / SSE / HTTP 连接建立            │
│    → 创建 Channel                            │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────┐
│ 2. 认证                                        │
│    → validate_token()                          │
│    → 获取 IdentityContext                      │
│    → 检查会话访问权限                          │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────┐
│ 3. 加入会话                                    │
│    → SessionCommand::AddParticipant            │
│    → 广播 participant_joined 事件              │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────┐
│ 4. 消息循环                                    │
│    → 接收客户端消息 → send_message()           │
│    → 订阅事件 → 转发给客户端                   │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────┐
│ 5. 断开连接                                    │
│    → SessionCommand::RemoveParticipant         │
│    → 广播 participant_left 事件                │
│    → 如果没有参与者，启动超时归档              │
└──────────────────────────────────────────────┘
```

### 空闲超时

会话在没有参与者时不会立即销毁，而是进入"空闲"状态：

```rust
const IDLE_TIMEOUT: Duration = Duration::from_secs(300); // 5 分钟

// 当最后一个参与者离开时
if self.participants.is_empty() {
    self.idle_since = Some(Instant::now());
    self.schedule_idle_check();
}

// 空闲检查
if self.idle_since.map(|t| t.elapsed() > IDLE_TIMEOUT).unwrap_or(false) {
    self.archive().await;
}
```

这允许用户短暂断开（网络波动、页面刷新）后重新连接而不会丢失会话。

## 会话与认证

会话服务与认证系统紧密集成：

### 创建会话

```rust
pub async fn create(
    &self,
    token: &str,           // JWT 或 API key
    model_id: String,
    config: SessionConfig,
) -> Result<Session, SessionError> {
    // 1. 验证令牌
    let identity = self.auth.verify(token).await
        .map_err(|_| SessionError::Unauthorized)?;

    // 2. 检查模型访问权限
    if !identity.can_use_model(&model_id) {
        return Err(SessionError::ModelNotAllowed);
    }

    // 3. 检查会话数量限制
    let count = self.count_user_sessions(&identity.user_id).await?;
    if count >= identity.max_sessions {
        return Err(SessionError::SessionLimitExceeded);
    }

    // 4. 创建会话
    let session = Session::new(identity.user_id.clone(), model_id, config);

    // 5. 启动 Actor
    self.supervisor.spawn_actor(session.id.clone(), session.clone()).await?;

    // 6. 持久化
    self.store.save(&session).await?;

    Ok(session)
}
```

### 访问控制

每个会话操作都检查调用者的权限：

```rust
pub enum SessionPermission {
    Owner,      // 所有者：完全控制
    Member,     // 成员：发送消息、查看历史
    Viewer,     // 观察者：只读访问
}

impl Session {
    pub fn check_permission(&self, user_id: &str) -> Result<SessionPermission, SessionError> {
        if self.owner_id == user_id {
            Ok(SessionPermission::Owner)
        } else if self.participants.iter().any(|p| p.user_id == user_id) {
            Ok(SessionPermission::Member)
        } else if self.is_public {
            Ok(SessionPermission::Viewer)
        } else {
            Err(SessionError::AccessDenied)
        }
    }
}
```

## 会话状态机

```
┌─────────┐    create()    ┌─────────┐
│  None   │ ─────────────→ │ Created │
└─────────┘                └────┬────┘
                                │ join()
                                ▼
                          ┌───────────┐
                          │  Active   │◄────────┐
                          └─────┬─────┘         │
                                │               │
                    ┌───────────┼───────────┐   │
                    │           │           │   │
                    ▼           ▼           ▼   │
               ┌────────┐  ┌────────┐  ┌────────┐
               │Paused  │  │Error   │  │Merging │
               └───┬────┘  └───┬────┘  └───┬────┘
                   │           │           │
                   │  resume() │  retry()  │  complete()
                   └───────────┴───────────┘
                                   │
                                   ▼
                             ┌───────────┐
                             │ Archived  │
                             └───────────┘
```

| 状态 | 说明 | 允许的操作 |
|------|------|----------|
| Created | 刚创建，等待第一个参与者 | 加入、删除 |
| Active | 正常运行中 | 发送消息、添加参与者、归档 |
| Paused | 所有者暂停 | 恢复、归档 |
| Error | 处理错误 | 重试、归档、删除 |
| Merging | 正在合并 CRDT | 等待完成 |
| Archived | 已归档（软删除） | 查看、恢复、硬删除 |

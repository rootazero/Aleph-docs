## 概览

设备配对是 Aleph 的信任建立协议。新设备（macOS 应用、iOS 客户端、Web 浏览器）或新聊天发送者（Telegram 用户、Discord 成员）在与 Aleph 交互之前，必须完成验证连接是否由服务器所有者授权的配对流程。

该协议使用短命的 **8 字符 Base32 代码** 显示给所有者，所有者确认或拒绝配对请求。这在概念上类似于蓝牙配对或 TV 应用激活——对非技术用户足够简单，同时在密码学上健全。

**源代码位置：**
- 配对管理器：`src/gateway/security/pairing.rs`
- 设备存储：`src/gateway/device_store.rs`
- 加密工具：`src/gateway/security/crypto.rs`
- 安全存储：`src/gateway/security/store.rs`
- CLI 命令：`src/bin/aleph_server/commands/pairing.rs`
- 设备命令：`src/bin/aleph_server/commands/devices.rs`

## 为什么配对？

Aleph 是一个自托管的 AI 助手，可以访问你的文件、shell 和在线账户。允许不受限制的连接将是一个重大的安全风险。配对协议解决了三个问题：

1. **身份验证** — 确保连接的设备或用户由服务器所有者授权。
2. **密钥交换** — 设备在配对期间提交其公钥，启用未来连接的加密认证。
3. **范围限制** — 频道发送者可以在配对时获得有限的权限。

## 配对类型

系统通过统一的 `PairingRequest` 枚举支持两种类型的配对：

### 设备配对

用于桌面应用、移动应用和 CLI 客户端：

```rust
PairingRequest::Device {
    request_id: String,
    code: String,              // 8 字符 Base32 代码
    device_name: String,       // "Alice's MacBook Pro"
    device_type: Option<DeviceType>,  // iOS、macOS、Android、CLI、Web
    public_key: Vec<u8>,       // Ed25519 公钥（32 字节）
    fingerprint: DeviceFingerprint,   // SHA256(public_key)[..16]
    remote_addr: Option<String>,
    created_at: i64,
    expires_at: i64,
}
```

### 频道配对

用于验证聊天平台上的发送者：

```rust
PairingRequest::Channel {
    request_id: String,
    code: String,              // 8 字符 Base32 代码
    channel: String,           // "telegram"、"discord"、"imessage"
    sender_id: String,         // 平台特定的用户 ID
    metadata: Option<Value>,   // 额外上下文
    created_at: i64,
    expires_at: i64,
}
```

## 配对代码

### 生成

配对代码是 8 字符字符串，从受限的 Base32 字母表中抽取，排除视觉上容易混淆的字符：

```rust
pub const PAIRING_CODE_LENGTH: usize = 8;
pub const PAIRING_CODE_CHARSET: &[u8] = b"ABCDEFGHJKLMNPQRSTUVWXYZ23456789";
```

排除的字符：`0`（零）、`1`（一）、`I`（大写 I）、`O`（大写 O）。这避免了在代码被大声朗读或在模糊字体中显示时产生混淆。

代码空间是 `28^8 = ~3770 亿` 个可能的代码，在 10 请求容量窗口内碰撞实际上是不可能的。

### 过期

代码在 **5 分钟**后默认过期：

```rust
const DEFAULT_PAIRING_EXPIRY_MS: i64 = 5 * 60 * 1000;
```

过期代码自动被拒绝。清理例程定期从数据库中清除过期的请求。

### 容量限制

为防止滥用，系统限制待处理配对请求：

```rust
const MAX_PENDING_REQUESTS: usize = 10;
```

如果达到限制，新的配对请求将被拒绝并显示 `TooManyPending` 错误，直到现有请求过期或被解决。

## 加密基础

### Ed25519 密钥对

设备在安装时生成 Ed25519 密钥对。公钥在配对期间提交：

```rust
pub fn generate_keypair() -> ([u8; 32], [u8; 32]) {
    let signing_key = SigningKey::generate(&mut OsRng);
    let verifying_key = signing_key.verifying_key();
    (signing_key.to_bytes(), verifying_key.to_bytes())
}
```

### 设备指纹

每个设备由其公钥派生的指纹标识 —— `SHA256(public_key)` 的前 16 个十六进制字符：

```rust
pub struct DeviceFingerprint(pub String);

impl DeviceFingerprint {
    pub fn from_public_key(public_key: &[u8]) -> Self {
        let hash = Sha256::digest(public_key);
        let hex = hex::encode(hash);
        Self(hex[..16].to_string())
    }
}
```

示例指纹：`a1b2c3d4e5f6a7b8`

### 令牌签名

配对确认后，Gateway 使用 HMAC-SHA256 签发签名令牌：

```rust
pub fn hmac_sign(secret: &[u8], token: &str) -> String {
    let mut mac = HmacSha256::new_from_slice(secret)
        .expect("HMAC accepts any key length");
    mac.update(token.as_bytes());
    hex::encode(mac.finalize().into_bytes())
}
```

签发的令牌格式为 `{token}:{signature}`。验证使用恒定时间比较来防止定时攻击：

```rust
pub fn hmac_verify(secret: &[u8], token: &str, signature: &str) -> Result<(), CryptoError> {
    let expected = hmac_sign(secret, token);
    if subtle::ConstantTimeEq::ct_eq(expected.as_bytes(), signature.as_bytes()).into() {
        Ok(())
    } else {
        Err(CryptoError::HmacVerificationFailed)
    }
}
```

## 设备管理

### 已批准设备存储

已批准设备存储在 `~/.aleph/devices.db` 的 SQLite 数据库中：

```rust
pub struct ApprovedDevice {
    pub device_id: String,
    pub device_name: String,
    pub device_type: Option<String>,  // macos、ios、android、cli、web
    pub approved_at: String,          // ISO 8601
    pub last_seen_at: Option<String>,
    pub permissions: Vec<String>,     // ["*"] = 完全访问
}
```

### CLI 命令

通过 CLI 管理设备和配对请求：

```bash
# 列出待处理配对请求
aleph pairing list

# 批准配对请求
aleph pairing approve XKRF4B7N

# 拒绝配对请求
aleph pairing reject XKRF4B7N

# 列出已批准设备
aleph devices list

# 撤销设备
aleph devices revoke <device-id>
```

`aleph devices list` 的示例输出：

```
Approved devices:
DEVICE ID                            NAME                 TYPE       APPROVED AT
------------------------------------------------------------------------------------------
a1b2c3d4-e5f6-...                    Alice's MacBook      macos      2026-01-28T12:00:00
b2c3d4e5-f6a7-...                    Alice's iPhone       ios        2026-02-01T09:30:00
```

### 撤销访问

当设备被撤销时，它将从已批准设备数据库中删除。该设备的任何现有令牌都将失效：

```rust
pub fn revoke_device(&self, device_id: &str) -> SqliteResult<bool> {
    let rows = conn.execute(
        "DELETE FROM approved_devices WHERE device_id = ?1",
        params![device_id],
    )?;
    Ok(rows > 0)
}
```

### 更新权限

设备权限可以在初始配对后缩小：

```rust
pub fn update_permissions(
    &self,
    device_id: &str,
    permissions: &[String],
) -> SqliteResult<bool>;
```

<Callout type="info">
默认情况下，新批准的设备获得完全访问权限（`["*"]`）。你可以在配对后将设备限制为特定工具类别（例如 `["translate", "search"]`）。
</Callout>

## 安全保证

1. **短命代码** — 配对代码在 5 分钟后过期，限制攻击窗口。
2. **容量限制** — 最多 10 个待处理请求防止资源耗尽。
3. **唯一代码** — 系统最多重试 10 次以生成不与现有待处理请求冲突的代码。
4. **仅所有者批准** — 只有服务器所有者（通过 CLI 或认证接口）可以批准配对请求。
5. **一次性使用** — 确认的配对代码会立即从数据库中删除。
6. **恒定时间验证** — 令牌验证使用恒定时间比较来防止定时侧信道。
7. **加密身份** — 设备由 Ed25519 公钥标识，启用未来的挑战-响应认证。

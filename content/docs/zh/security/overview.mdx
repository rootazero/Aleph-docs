## 安全架构

Aleph 的安全模型建立在四个防御层之上。每一层独立运行——一层的失败不会导致其他层崩溃。

```
Layer 1: 认证
├─ 设备配对（8 字符代码，Ed25519 密钥）
├─ Bearer 令牌验证
└─ 会话建立

Layer 2: 授权
├─ IdentityContext（不可变的身份快照）
├─ PolicyEngine（无状态权限检查器）
├─ 基于角色的访问（Owner / Guest / Anonymous）
└─ GuestScope（工具级限制）

Layer 3: 沙箱
├─ WorkspaceSandbox（文件系统隔离）
├─ OsSandboxDriver（平台原生强制执行）
├─ 基于能力的配置文件
└─ 信任阶段（Draft → Trial → Verified）

Layer 4: 审计
├─ capability_ledger（结构化审计日志）
├─ SandboxAuditLog（执行跟踪）
└─ ExecAuditEntry（审批决策）
```

## 身份和访问控制

到达 Aleph 的每个请求都携带一个 [IdentityContext](/docs/zh/concepts/identity-context) —— 一个不可变的快照，说明调用者是谁以及他们被允许做什么。

身份系统支持三种角色：

| 角色 | 行为 |
|------|----------|
| **Owner** | 完全访问；绕过所有权限检查 |
| **Guest** | 范围受限；限制为允许的工具，可选过期时间 |
| **Anonymous** | 除非禁用认证，否则被拒绝 |

访客访问通过带有细粒度 `GuestScope` 的 [邀请](/docs/zh/concepts/identity-context) 授予：

```rust
pub struct GuestScope {
    pub allowed_tools: Vec<String>,
    pub expires_at: Option<i64>,
    pub display_name: Option<String>,
}
```

## PolicyEngine

PolicyEngine 是一个**无状态**的权限检查器，没有可变状态。它根据调用者的 IdentityContext 评估每个工具调用：

- **Owner** — 始终允许
- **Guest** — 检查范围存在、过期时间和工具成员资格
- **Anonymous** — 始终拒绝（在生产环境中）

因为它是有状态的，PolicyEngine 可以在执行链中的任何地方实例化，而无需考虑生命周期或同步问题。

## 沙箱

Aleph 在操作系统级别强制执行基于能力的沙箱。每个工具声明它需要的权限（文件系统路径、网络访问、内存限制），系统生成一个平台原生的沙箱配置文件，强制执行这些边界。

关键特性：
- **能力预设** — `file_processor`、`web_scraper`、`code_analyzer`、`data_transformer`
- **信任阶段** — Draft → Trial → Verified，逐步审批
- **升级检测** — 敏感目录访问触发重新审批
- **平台强制执行** — macOS `sandbox-exec` 带 Seatbelt 配置文件

参见 [沙箱](/docs/zh/security/sandboxing) 获取完整参考。

## 执行审批

在任何 shell 命令运行之前，它会通过人机回环审批流程：

1. **风险分析** — 命令被评为 Low / Medium / High / Critical
2. **白名单检查** — 已知安全的命令自动批准
3. **人工提示** — 未知命令通过 CLI、macOS 应用或聊天路由给所有者
4. **决策持久化** — "始终允许" 将命令添加到永久白名单

参见 [执行审批](/docs/zh/security/exec-approval) 获取完整参考。

## 设备配对

新设备和聊天发送者必须完成配对流程才能与 Aleph 交互：

1. 设备请求配对并收到 8 字符 Base32 代码
2. 所有者收到带有代码的通知
3. 所有者通过 `aleph pairing approve <code>` 批准
4. 设备收到签名令牌用于未来认证

配对使用 Ed25519 密钥交换、HMAC-SHA256 令牌签名和 5 分钟代码过期。

参见 [设备配对](/docs/zh/security/pairing) 获取完整参考。

## IPC 安全

Gateway 通过 Unix 域套接字与本地桌面客户端（例如 macOS 应用）通信，位于 `~/.aleph/exec-approvals.sock`。IPC 协议提供：

- 每次连接上的 **HMAC-SHA256 挑战-响应** 认证
- 套接字上的 **0600 文件权限**（仅所有者访问）
- **独立连接处理** — 每个客户端在自己的任务中运行
- Gateway 重启时的 **陈旧套接字清理**

参见 [IPC 协议](/docs/zh/security/ipc) 获取完整参考。

## 审计日志

每个安全相关事件都记录在结构化审计日志中：

### capability_ledger

沙箱执行事件的跟踪目标：

```rust
pub struct SandboxAuditLog {
    pub skill_id: String,
    pub capabilities: Capabilities,
    pub status: ExecutionStatus,
    pub platform: String,
    pub timestamp: DateTime<Utc>,
}
```

### 执行审计条目

记录 shell 命令的审批决策：

```rust
pub struct AuditEntry {
    pub timestamp: DateTime<Utc>,
    pub command: String,
    pub risk_level: RiskLevel,
    pub decision: ApprovalDecision,
    pub executor: String,
    pub session_key: String,
    pub duration_ms: u64,
    pub exit_code: Option<i32>,
}
```

这些日志提供了谁执行了什么、何时执行以及具有什么权限的完整跟踪。

title: "Resilience System"

The resilience system makes Aleph fault-tolerant. It ensures that interrupted tasks can be recovered, running agents can checkpoint gracefully, and the system degrades predictably under failure. The system is built around a SQLite-based `StateDatabase` and three core mechanisms: **Shadow Replay**, **Graceful Shutdown**, and **Risk-Aware Recovery**.

## StateDatabase

The `StateDatabase` is the persistence backbone of the resilience system. It uses SQLite (via rusqlite) to store four categories of data:

| Table | Purpose | Key Fields |
|-------|---------|------------|
| **events** | Tiered event persistence (Skeleton + Pulse) | `task_id`, `seq`, `event_type`, `is_structural` |
| **tasks** | Task state and recovery metadata | `id`, `status`, `risk_level`, `lane`, `checkpoint_snapshot_path` |
| **traces** | Execution traces for Shadow Replay | `task_id`, `step_index`, `role`, `content_json` |
| **sessions** | Subagent session lifecycle | `id`, `agent_type`, `status`, `context_path` |

### Event Tiers: Skeleton and Pulse

Events follow a two-tier persistence model:

```rust
impl AgentEvent {
    // Structural events (Skeleton) -- persisted immediately
    pub const TYPE_TASK_STARTED: &str = "task_started";
    pub const TYPE_TOOL_CALL_STARTED: &str = "tool_call_started";
    pub const TYPE_TOOL_CALL_COMPLETED: &str = "tool_call_completed";
    pub const TYPE_ARTIFACT_CREATED: &str = "artifact_created";
    pub const TYPE_TASK_COMPLETED: &str = "task_completed";
    pub const TYPE_TASK_FAILED: &str = "task_failed";

    // Streaming events (Pulse) -- batched before persistence
    pub const TYPE_AI_STREAMING: &str = "ai_streaming";
}
```

**Skeleton events** represent state transitions that must survive crashes. They are written synchronously and used for recovery. **Pulse events** are high-frequency streaming data (token-by-token output) that is batched and can be dropped without affecting correctness.

### Lane

The lane system provides resource isolation between user interactions and background work:

```rust
pub enum Lane {
    Main,      // User interactions, abort commands (high priority)
    Subagent,  // Background work (normal priority)
}
```

The Main lane is never starved by subagent work. User-facing requests (conversation, abort commands) always get priority scheduling.

## Shadow Replay

Shadow Replay is the mechanism for deterministic task recovery without consuming LLM tokens. When a task is interrupted, its execution traces (the sequence of assistant messages and tool results) are stored in the StateDatabase. On recovery, these traces are replayed to reconstruct the exact conversation state.

### How It Works

```
1. During Execution: Record every step as a TaskTrace
   в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ     в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ     в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
   в”Ӯ Assistant:  в”Ӯв”Җв”Җв”Җв”Җв–¶в”Ӯ Tool:      в”Ӯв”Җв”Җв”Җв”Җв–¶в”Ӯ Assistant:  в”Ӯ
```

### Trace Recording

Each step in execution is recorded as a `TaskTrace`:

```rust
pub struct TaskTrace {
    pub id: i64,
    pub task_id: String,
    pub step_index: u32,          // 0-based step counter
    pub role: TraceRole,          // Assistant | Tool
    pub content_json: String,     // Serialized message content
    pub timestamp: i64,
}

pub enum TraceRole {
    Assistant,  // LLM output (including tool calls)
    Tool,       // Tool execution result
}
```

### Divergence Detection

After replay, the system performs a "Handover Inference" check. It asks the LLM what it would do next, and compares that to the recorded next step:

```rust
pub enum DivergenceStatus {
    Aligned,    // LLM agrees with recorded next step
    Diverged {  // LLM wants a different action
        expected_tool: String,
        actual_tool: String,
    },
    ReachedEnd, // No more recorded steps -- start live execution
}
```

If the LLM diverges, the system can either follow the LLM's new direction (adapting to changed state) or continue the recorded path (preserving determinism).

## Graceful Shutdown

When Aleph receives SIGTERM or SIGINT, the `GracefulShutdown` handler checkpoints all running tasks:

```rust
pub struct GracefulShutdown {
    db: Arc<StateDatabase>,
    shutdown_tx: broadcast::Sender<ShutdownSignal>,
}

pub enum ShutdownSignal {
    Term,       // SIGTERM
    Interrupt,  // SIGINT (Ctrl+C)
    Requested,  // Programmatic shutdown
}
```

The checkpoint process:

1. Signal handler receives SIGTERM/SIGINT
2. Broadcast shutdown signal to all subsystems
3. Mark all `Running` tasks as `Interrupted` in the StateDatabase
4. Each task's trace history is already persisted (traces are written during execution)
5. Process exits cleanly

```rust
impl GracefulShutdown {
    pub async fn checkpoint(&self) -> Result<u64> {
        // Mark all running tasks as interrupted
        let count = self.db.mark_running_as_interrupted().await?;
        info!(interrupted_tasks = count,
              "Marked running tasks as interrupted for recovery");
        Ok(count)
    }
}
```

On restart, the Recovery Manager finds these interrupted tasks and decides how to handle them.

## Recovery Manager

The `RecoveryManager` scans for recoverable tasks on startup and makes risk-aware decisions:

```rust
pub enum RecoveryDecision {
    AutoResume {
        task: AgentTask,
        replay_engine: Arc<ShadowReplayEngine>,
    },
    PendingConfirmation {
        task: AgentTask,
    },
    Skip {
        task_id: String,
        reason: String,
    },
}
```

### Decision Logic

```
For each task where status = Running | Interrupted:
    в”Ӯ
    в”ңв”Җв”Җ No traces available?
    в”Ӯ   в””в”Җв”Җ Skip("No execution traces available")
    в”Ӯ
    в”ңв”Җв”Җ RiskLevel::Low?
    в”Ӯ   в””в”Җв”Җ AutoResume (Shadow Replay + continue)
    в”Ӯ
    в””в”Җв”Җ RiskLevel::High?
        в””в”Җв”Җ PendingConfirmation (wait for user approval)
```

The rationale is simple: read-only operations (file search, code analysis) are safe to retry automatically. Write operations (file modification, shell commands) need human confirmation because the system state may have changed since the interrupt.

### Recovery Summary

```rust
pub struct RecoverySummary {
    pub auto_resume_count: usize,
    pub pending_confirmation_count: usize,
    pub skipped_count: usize,
    pub total_count: usize,
}
```

On startup, Aleph logs a recovery summary and begins auto-resuming low-risk tasks while queuing high-risk tasks for user confirmation.

## Subagent Sessions (Session-as-a-Service)

Subagent sessions allow specialized agents (explorer, coder, researcher) to persist across multiple task invocations:

```rust
pub struct SubagentSession {
    pub id: String,
    pub agent_type: String,        // "explorer", "coder", etc.
    pub status: SessionStatus,     // Active | Idle | Swapped
    pub context_path: Option<String>, // Disk path for swapped context
    pub parent_session_id: String,
    pub created_at: i64,
    pub last_active_at: i64,
    pub total_tokens_used: u64,
    pub total_tool_calls: u64,
}

pub enum SessionStatus {
    Active,   // Executing right now
    Idle,     // In memory, waiting for reuse
    Swapped,  // Context serialized to disk
}
```

### Context Swapping

When memory pressure is high, idle sessions can be swapped to disk:

```
Active в”Җв”Җ(task completes)в”Җв”Җв–¶ Idle в”Җв”Җ(memory pressure)в”Җв”Җв–¶ Swapped
                              в”Ӯ                              в”Ӯ
                              в”Ӯв—Җв”Җв”Җ(new task for agent)в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                              в”Ӯ   (deserialize from disk)
```

Swapped sessions are stored as serialized context files. When a new task arrives for a swapped agent type, the system deserializes the context and the session resumes with its full conversation history intact.

### Handle Reuse

Session handles enable efficient reuse:

```
Task 1: "Search for auth code" вҶ’ spawn explorer session (sess-001)
Task 2: "Find the test files"  вҶ’ reuse explorer session (sess-001)
                                  (session retains context from Task 1)
```

This avoids the cold-start cost of building context from scratch for related tasks.
